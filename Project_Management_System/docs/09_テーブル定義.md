# 1. 本書の目的

本書は`08_ER図.md`の論理設計を、MySQL 8上で実行可能な物理テーブル定義に落とし込む。本書の型・制約定義がそのままLaravel Migrationの実装根拠となる。

# 2. テーブル設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| TBL-POL-001 | 全テーブルにタイムスタンプ | `created_at`（+更新がありうるテーブルは`updated_at`）を必須とする。例外3件: `task_comments`（編集機能なしで不変、BR-CMT-003）と`task_files`（アップロード後不変）は`created_at`のみ。`notifications`は状態変化（既読化）を`read_at`が記録するため`updated_at`を持たない。 |
| TBL-POL-002 | 外部キーは物理制約で強制 | Laravel側のリレーション定義だけでなく、DBのFOREIGN KEY制約でも整合性を強制する（Project 01/02と同一方針）。 |
| TBL-POL-003 | 無効化と物理削除の使い分け | `users`（inactive化）・`projects`（archived化）は物理削除しない。タスク系（`tasks`/`task_comments`/`task_files`）はタスク削除時に連動物理削除する（BR-TSK-007。Project 02の「会計証憑は永続保持」との対比を意図した設計）。 |
| TBL-POL-004 | 外部キー命名の例外 | 原則`[参照先単数形]_id`だが、役割名を優先する3例外（`assignee_id` / `created_by` / `uploaded_by`、いずれも`users`参照）を許容する。詳細は10章。 |
| TBL-POL-005 | DB制約とアプリ層の役割分担の明示 | UNIQUE・CHECK・FKで守れるものはDBで守る。DBで表現できない2件（positionの一意性、task_due_soonの重複防止）は理由と代替保証をセットで11章に記録する。 |

# 5. 外部キー制約

| 子テーブル | 子カラム | 親テーブル | 親カラム | ON DELETE |
| --- | --- | --- | --- | --- |
| project_members | project_id | projects | id | CASCADE |
| project_members | user_id | users | id | CASCADE |
| tasks | project_id | projects | id | CASCADE |
| tasks | assignee_id | users | id | SET NULL |
| tasks | created_by | users | id | RESTRICT |
| task_comments | task_id | tasks | id | CASCADE |
| task_comments | user_id | users | id | RESTRICT |
| task_files | task_id | tasks | id | CASCADE |
| task_files | uploaded_by | users | id | RESTRICT |
| notifications | user_id | users | id | CASCADE |
| notifications | task_id | tasks | id | **SET NULL** |

**方針**: `users`/`projects`の物理削除はスコープ外のため、実際に機能するCASCADEは「タスク削除→コメント/ファイルの連動削除」（BR-TSK-007）である。`notifications.task_id`のSET NULLだけが意味的に特殊で、「参照先が消えても通知は残す」（UC-015 3-a）をDB層で表現する。RESTRICT系（created_by等）は履歴作成者の保護（`08_ER図.md` 7章）。

# 8. CHECK制約

| テーブル | 制約 | 目的 |
| --- | --- | --- |
| task_files | `size_bytes <= 10485760` | 10MB制限（BR-FIL-001）の最終防衛線。アプリ検証をすり抜けた場合もDBが拒否する（Project 02の在庫CHECKと同じ「二重防御」哲学） |
| notifications | `(is_read = FALSE AND read_at IS NULL) OR (is_read = TRUE AND read_at IS NOT NULL)` | 「既読なのに日時がない/未読なのに日時がある」矛盾状態の物理的防止 |

ステータス系の許容値はENUM型自体が制約として機能するため、Project 02のような別途CHECKは設けない（`orders.status`はBR上の遷移制限があったためCHECKを重ねたが、本システムのステータスは全遷移自由でありENUMで十分）。

# 10. 命名規則

| 対象 | 規則 | 例 |
| --- | --- | --- |
| テーブル名 | snake_case・複数形 | project_members |
| 主キー | id（BIGINT UNSIGNED） | id |
| 外部キー | [参照先単数形]_id | project_id, user_id, task_id |
| タイムスタンプ | created_at, updated_at | - |
| フラグ | is_ prefix | is_read |
| enum | 小文字snake_case | in_progress, task_due_soon |

**命名規則の例外（意図的な逸脱、TBL-POL-004）**:

| カラム | 本来の規則どおりなら | 実際の名前 | 理由 |
| --- | --- | --- | --- |
| `tasks.assignee_id` | `user_id` | `assignee_id` | 同一テーブルに`users`参照FKが2本あり（担当者と作成者）、役割名でなければ区別できない。 |
| `tasks.created_by` | `user_id` | `created_by` | 同上。監査系カラムの慣用名（Project 02 `inventory_logs.created_by`と同じパターン）。 |
| `task_files.uploaded_by` | `user_id` | `uploaded_by` | 削除権限判定（BR-FIL-003）に使う「アップロード者」であることを名前で明示。 |
| `task_files.size_bytes` | `size` | `size_bytes` | 単位（バイト）を名前に含め、KB/MB混同による制限値バグを防ぐ。 |

# 11. 設計判断（設計判断メモ）

`00_開発計画書.md` 9章の設計論点への最終回答を記録する。

1. **positionの採番方式 — 隙間法（gap method）を採用し、UNIQUEは張らない**:
   - **採番**: 新規タスクは列末尾に`最大position + 1024`で追加。D&Dによる挿入は前後タスクのpositionの中間値（`floor((prev + next) / 2)`）を採る。中間値が作れなくなった（前後差が1になった）時点で、その列（project_id × status）全体を1024刻みで振り直す（リナンバリング）。
   - **UNIQUE(project_id, status, position)を張らない理由**: リナンバリングや隣接タスクの入れ替えでは更新途中に一時的な重複が発生し得るが、MySQLのUNIQUE制約は文の実行ごとに即時検査されるため（遅延制約非対応）、正当な並び替え操作自体が失敗してしまう。
   - **代替保証**: 並び替えはService層で列単位に`SELECT ... FOR UPDATE`（対象列のタスク行をロック）した上で実行し、同時ドラッグを直列化する（Project 02の在庫ロックで習得した悲観的ロックの再適用）。万一positionが重複しても、表示は`ORDER BY position, id`で決定的であり業務影響は「見た目の順序が1回ズレる」に留まる — 在庫のような金銭的損害はないため、この保証レベルで十分と判断した。
2. **task_due_soonの重複防止 — アプリ層の存在チェックで保証**: 「1タスク1回」（BR-NTF-003）を`UNIQUE(task_id, type)`で表現すると、`task_assigned`/`task_commented`（同一タスクに複数回発生し得る）まで巻き込まれる。type条件付きの部分UNIQUEはMySQLでは表現できない（Project 02の「1会員1activeカート」と同じ制約事情）。期限バッチは毎時単一実行（`withoutOverlapping`）で並行性がないため、INSERT前の存在チェック（`INDEX(task_id, type)`使用）で十分と判断した。
3. **通知の生存戦略（物理層）**: `task_id` SET NULL + `task_title`スナップショット（`08_ER図.md` REL-011/10章の物理実装）。遷移時にtask_idがNULLなら、フロントはE007相当の「タスクが見つかりません」を表示する（G-07）。
4. **task_files.size_bytesのCHECK制約**: FormRequest（アプリ層）の検証が正だが、直接SQLやバグをすり抜けた場合の最終防衛線としてDB側でも拒否する。10485760 = 10 × 1024 × 1024。
5. **projectsに集計列・作成者列を持たない**: `08_ER図.md` 10章の正規化判断を物理層でも維持する。未完了タスク数（SCR-003）はCOUNT導出、Ownerは`project_members`から導出。

# 13. まとめ

7テーブルの物理設計を確定した。本書の核心は11章の2つの設計判断 — ①position隙間法（UNIQUEを張らない理由と代替保証のセット）、②task_due_soon重複防止のアプリ層委譲 — であり、いずれも「DB制約で守れないものは、守れない理由と代替保証を明文化する」（TBL-POL-005）の実践である。実装時は`project_members`のUNIQUE複合キーと`notifications`のCHECK制約を必ずMigrationに含め、`15_単体試験仕様書.md`でDB制約レベルの検証（直接INSERT試験）を行うこと。
