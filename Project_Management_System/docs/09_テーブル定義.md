# 1. 本書の目的

本書は08_ER図の論理設計を、MySQL 8上で実行可能な物理テーブル定義に落とし込む。本書の型・制約定義がそのままLaravel Migrationの実装根拠となる。

# 2. テーブル設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| TBL-POL-001 | 全テーブルにタイムスタンプ | `created_at`（+更新がありうるテーブルは`updated_at`）を必須とする。例外3件: `task_comments`（編集機能なしで不変、BR-CMT-003）と`task_files`（アップロード後不変）は`created_at`のみ。`notifications`は状態変化（既読化）を`read_at`が記録するため`updated_at`を持たない。 |
| TBL-POL-002 | 外部キーは物理制約で強制 | Laravel側のリレーション定義だけでなく、DBのFOREIGN KEY制約でも整合性を強制する（Project 01/02と同一方針）。 |
| TBL-POL-003 | 無効化と物理削除の使い分け | `users`（inactive化）・`projects`（archived化）は物理削除しない。タスク系（`tasks`/`task_comments`/`task_files`）はタスク削除時に連動物理削除する（BR-TSK-007。Project 02の「会計証憑は永続保持」との対比を意図した設計）。 |
| TBL-POL-004 | 外部キー命名の例外 | 原則`[参照先単数形]_id`だが、役割名を優先する3例外（`assignee_id` / `created_by` / `uploaded_by`、いずれも`users`参照）を許容する。詳細は10章。 |
| TBL-POL-005 | DB制約とアプリ層の役割分担の明示 | UNIQUE・CHECK・FKで守れるものはDBで守る。DBで表現できない3件（positionの一意性、task_due_soonの重複防止、`tasks.assignee_id`が当該プロジェクトのメンバーであることの継続的保証）は理由と代替保証をセットで11章に記録する。3件目は`assignee_id`が`users`への単純FKで`project_members`を経由しないため（複合FK `(project_id, assignee_id) → project_members(project_id, user_id)` で表現する案も検討したが、既存のロック戦略（11章-6）との一貫性を優先し見送った）、アプリ層のプロジェクト単位ロックで担保する。 |

# 5. 外部キー制約

| 子テーブル | 子カラム | 親テーブル | 親カラム | ON DELETE |
| --- | --- | --- | --- | --- |
| project_members | project_id | projects | id | CASCADE |
| project_members | user_id | users | id | CASCADE |
| tasks | project_id | projects | id | CASCADE |
| tasks | assignee_id | users | id | SET NULL |
| tasks | created_by | users | id | RESTRICT |
| task_comments | task_id | tasks | id | CASCADE |
| task_comments | user_id | users | id | RESTRICT |
| task_files | task_id | tasks | id | CASCADE |
| task_files | uploaded_by | users | id | RESTRICT |
| notifications | user_id | users | id | CASCADE |
| notifications | task_id | tasks | id | **SET NULL** |

**方針**: `users`/`projects`の物理削除はスコープ外のため、実際に機能するCASCADEは「タスク削除→コメント/ファイルの連動削除」（BR-TSK-007）である。`notifications.task_id`のSET NULLだけが意味的に特殊で、「参照先が消えても通知は残す」（UC-015 3-a）をDB層で表現する。RESTRICT系（created_by等）は履歴作成者の保護（08_ER図 7章）。

# 11. 設計判断（設計判断メモ）

00_開発計画書 9章の設計論点への最終回答を記録する。

1. **positionの採番方式 — 隙間法（gap method）を採用し、UNIQUEは張らない**:
   - **採番**: 新規タスクは列末尾に`最大position + 1024`で追加。D&Dによる挿入は前後タスクのpositionの中間値（`floor((prev + next) / 2)`）を採る。中間値が作れなくなった（前後差が1になった）時点で、その列（project_id × status）全体を1024刻みで振り直す（リナンバリング）。
   - **UNIQUE(project_id, status, position)を張らない理由**: リナンバリングや隣接タスクの入れ替えでは更新途中に一時的な重複が発生し得るが、MySQLのUNIQUE制約は文の実行ごとに即時検査されるため（遅延制約非対応）、正当な並び替え操作自体が失敗してしまう。
   - **代替保証**: 並び替えはService層で列単位に`SELECT ... FOR UPDATE`（対象列のタスク行をロック）した上で実行し、同時ドラッグを直列化する（Project 02の在庫ロックで習得した悲観的ロックの再適用）。万一positionが重複しても、表示は`ORDER BY position, id`で決定的であり業務影響は「見た目の順序が1回ズレる」に留まる — 在庫のような金銭的損害はないため、この保証レベルで十分と判断した。
   - **⚠️ 設計監査で発見した欠陥と対策（v1.1）**: 上記の列単位`FOR UPDATE`だけでは、**移動先の列が空の場合にInnoDBのデッドロックが発生し得る**。`(project_id, status, position)`は非UNIQUEの複合INDEXであり、0件ヒットのSELECT ... FOR UPDATEでもREPEATABLE READ下ではギャップロックが取得される。2つのトランザクションが同時に同じ空列へ別々のタスクを移動すると、両者とも同じギャップにギャップロックを取得（ギャップロック同士は互換）した後、両者ともそのギャップへの書き込み（INSERT相当のインデックスエントリ挿入）に必要な挿入意図ロックを要求し、互いのギャップロックと衝突して相互に待ち合い、デッドロックする。対策は次項6のプロジェクト単位ロックで、この操作自体を直列化して解消する（12_詳細設計書 5.6節）。
2. **task_due_soonの重複防止 — `tasks.is_due_soon_notified`フラグで保証（v1.1で存在チェック方式から変更）**:
   - **旧方式の欠陥**: 「1タスク1回」（BR-NTF-003）を`INDEX(task_id, type)`使用のnotifications存在チェックで実装すると、`due_date`が変更され新しい期限が到来しても「既に1回送信済み」の判定に巻き込まれ、新しい期限への通知が永久にスキップされる。BR-NTF-003が明示的に許容している「担当者変更後は再発火しない」（新担当者はtask_assignedで気づけるため代替手段がある）を、代替手段の存在しない`due_date`変更にも誤って適用してしまっていた。
   - **新方式**: `tasks.is_due_soon_notified`（BOOLEAN、デフォルトfalse）を通知済みフラグとして持つ。バッチの抽出条件に`is_due_soon_notified=false`を含め、通知送信後にtrueへ更新する（12_詳細設計書 5.13節）。`TaskService::update`で`due_date`が変更された場合はfalseへリセットする（5.5節）。
   - **`UNIQUE(task_id, type)`を張らない理由（変更なし）**: `task_assigned`/`task_commented`（同一タスクに複数回発生し得る）まで巻き込まれるため、type条件付きの部分UNIQUEはMySQLでは表現できない（Project 02の「1会員1activeカート」と同じ制約事情）。`INDEX(task_id, type)`自体は通知一覧の一般的な絞り込みに引き続き使うため残す。
3. **通知の生存戦略（物理層）**: `task_id` SET NULL + `task_title`スナップショット（08_ER図 REL-011/10章の物理実装）。遷移時にtask_idがNULLなら、フロントはE007相当の「タスクが見つかりません」を表示する（G-07）。
4. **task_files.size_bytesのCHECK制約**: FormRequest（アプリ層）の検証が正だが、直接SQLやバグをすり抜けた場合の最終防衛線としてDB側でも拒否する。10485760 = 10 × 1024 × 1024。
5. **projectsに集計列・作成者列を持たない**: 08_ER図 10章の正規化判断を物理層でも維持する。未完了タスク数（SCR-003）はCOUNT導出、Ownerは`project_members`から導出。
6. **プロジェクト単位ロック（アグリゲートルートロック）— v1.1で追加**: 設計監査で、`tasks.assignee_id`への書き込みとメンバー除名（BR-PRJ-005、`project_members`削除＋`assignee_id`のNULL化）が競合すると、非メンバーへのタスク割当が成立し得る競合を発見した（担当者判定がトランザクション開始前の無ロック読み取りだったため）。`assignee_id`は`users`への直接FKで`project_members`を経由しないため、DB制約だけではこの整合性を守れない（TBL-POL-005）。対策として、`projects`行に対する`SELECT ... FROM projects WHERE id = ? FOR UPDATE`を、①タスクの担当割当・変更（assignee_id指定時のみ）、②メンバー除名・自主脱退、③カンバン移動（1のデッドロック対策）の3箇所で共通の直列化ポイントとして使う。1つのロック対象に統一することで、複数の局所的ロック戦略を個別に設計・検証するより挙動を追いやすく、学習上も一貫性がある（12_詳細設計書 5.3〜5.6節）。
