# 1. 本書の目的

`17_システム試験仕様書.md`が技術的な正しさ（E2E・IDOR実地確認・障害回復）を確認するのに対し、本書は実際の**チーム運用の言葉・操作感**でシステムを通し、`02_要件定義書.md`の受入条件（19章）と`01_企画書.md`のKPI（11章）・成功条件（12章）を最終的に満たすことを確認する。

# 2. UAT方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UAT-POL-001 | Business Language | 手順・結果はプロジェクト運用者が読める言葉で記述する（「E007を確認」ではなく「他チームの内容が見えないこと」）。 |
| UAT-POL-002 | 実チーム体験 | 実際に2アカウント（Owner/Member）を行き来し、チームでの利用感を確認する。 |
| UAT-POL-003 | セルフレビュー | 個人開発プロジェクトのため、開発者自身が「初めてチームに招待された新人」の視点を意識して実施する。 |
| UAT-POL-004 | Admin=運用者の視点固定 | Admin編は「業務内容を見ない運用者」という立場を崩さずに評価する（見えてしまうこと自体が不具合）。 |

# 6. UATシナリオ（Owner編）

| シナリオID | 内容 | 確認の観点（業務言葉） |
| --- | --- | --- |
| UAT-OWN-001 | 新しいプロジェクトを立ち上げる | プロジェクト名を決めてすぐ作れるか、自分がオーナーになったと分かるか |
| UAT-OWN-002 | チームメンバーを招待する | メールアドレスを知っているだけで招待でき、承諾を待たずにすぐ一緒に作業を始められるか |
| UAT-OWN-003 | タスクをメンバーに割り振る | 誰が何を担当しているか一覧でひと目で分かるか |
| UAT-OWN-004 | しばらく使わないプロジェクトをしまう | アーカイブ後、間違って編集してしまう心配がないか（表示で分かるか）。必要になったらすぐ元に戻せるか |
| UAT-OWN-005 | 辞めるメンバーを外す | 外した後、そのメンバーの担当タスクが「担当者未定」に自然に戻り、チームが困らないか |

# 7. UATシナリオ（Member編）

| シナリオID | 内容 | 確認の観点（業務言葉） |
| --- | --- | --- |
| UAT-MEM-001 | 招待されたプロジェクトに参加する | 招待から参加までの手間が少ないか（パスワード設定以外の余計な手順がないか） |
| UAT-MEM-002 | 自分のタスクをカンバンで進める | ドラッグ&ドロップで直感的に「着手」「完了」を動かせるか、動かした結果がすぐ画面に反映されるか |
| UAT-MEM-003 | 相手にコメントで確認事項を伝える | コメントを書いたら相手に気づいてもらえる仕組み（通知）があると感じられるか |
| UAT-MEM-004 | 資料ファイルを共有する | ファイルを添付してから、相手が迷わずダウンロードできるか |
| UAT-MEM-005 | 期限が近いタスクに気づく | リマインドが来て「あ、忘れてた」を防げるか |
| UAT-MEM-006 | 自分から抜けたいプロジェクトを離れる | 気まずさなく自分の意思で脱退でき、オーナー権限がない自分でも問題なく操作できるか |

---

# 8. UATシナリオ（Admin編）

| シナリオID | 内容 | 確認の観点（業務言葉） |
| --- | --- | --- |
| UAT-ADM-001 | 全体のプロジェクト数・利用状況を把握する | プロジェクトの中身（タスク内容等）を見なくても運用判断（活発/停滞）ができる情報量になっているか |
| UAT-ADM-002 | 退職者のアカウントを止める | ログインだけを止め、その人が過去に担当していた履歴・コメントが消えずに残るか（チームの記録が壊れないか） |
| UAT-ADM-003 | 長期間放置されたプロジェクトを整理する | 強制的にしまう操作が簡単で、かつ元に戻す権限がオーナー側にきちんと残っていることが分かるか |

---

# 9. 横断シナリオ

| シナリオID | 内容 |
| --- | --- |
| UAT-CROSS-001 | Ownerがメンバーを招待→Memberが参加してタスクを進める→Ownerがカンバンでリアルタイムに近い形で進捗を把握できる |
| UAT-CROSS-002 | Memberがコメント・ファイルを追加→Ownerが確認→双方が同じ最新状態を見ていると感じられる（食い違いがない） |
| UAT-CROSS-003 | 無関係な第三者（他プロジェクトのユーザー）がURLを推測してアクセスを試みても、何も見えず「存在しないページ」としか感じられない（プロジェクトの存在すら気づけない） |

---

# 10. 不具合判定基準

| 重要度 | 基準 |
| --- | --- |
| Critical | 権限の壁が破れる（他プロジェクトの内容が見える、Memberがオーナー操作をできる等）、カンバンの並び順やタスクが消失・重複する |
| High | 業務は続行できるが誤解を招く表示・通知の欠落や誤発火 |
| Medium | UI改善要望レベル（ボタン配置・文言の分かりにくさ） |
| Low | 表記ゆれ等 |

# 11. 受け入れ判定基準（サインオフ条件）

- UATシナリオ（6/7/8/9章）すべてがPassすること
- Critical不具合が0件であること
- `02_要件定義書.md` 19章の受入条件をすべて満たすこと（6章REQ受入条件・8章権限マトリクス全セルPass・9章業務ルール違反状態がDBに存在しない・非メンバー全経路E007）
- `01_企画書.md` 11章KPI（権限マトリクス網羅・IDOR防御・通知正確性・フロントエンド品質・チーム開発プラクティス・デプロイ）をすべて満たすこと
- `01_企画書.md` 12章の成功条件（Checklist Bắt Buộc全項目・7テーブルDDLのクリーン実行・Owner/Member並行操作デモの一連動作）を満たすこと

# 13. まとめ

本書は`17_システム試験仕様書.md`が技術的に確認した内容を、開発者自身が「初めてチームに招待された新人」「業務内容を見ない運用者」という2つの視点に立ち直して再確認するものである。特にUAT-CROSS-003（無関係な第三者が存在にすら気づけないこと）は、`14_セキュリティ設計.md`のIDOR対策とBR-PRM-006（存在の秘匿）が技術試験だけでなく体感としても成立していることの最終確認であり、これをもって`01_企画書.md`のKPIと成功条件を最終的にサインオフする。
