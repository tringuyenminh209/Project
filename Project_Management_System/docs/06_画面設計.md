# 1. 本書の目的

Project Management Systemの各画面について、表示項目、入力項目、操作、バリデーション、エラー、状態バリエーションを定義する。`02_要件定義書.md` `03_ユースケース.md` `05_画面遷移図.md`と整合し、`10_API設計.md`（作成後に8章へAPI対応を追記）および試験仕様書の基準とする。

# 2. 画面設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UI-001 | OOUI（オブジェクト指向UI） | 「プロジェクト」→「タスク」という対象物を選んでから操作する設計とする。 |
| UI-002 | PC First | 業務ツールのため1024px以上を最適化対象とする。タブレット・モバイルは閲覧・既読操作を保証し、カンバンD&Dは対象外としてよい（NFR-020。Project 02のMobile Firstとは意図的に方針が異なる）。 |
| UI-003 | 楽観的UI更新 | カンバンD&D等の高頻度操作は先に画面へ反映し、API失敗時に巻き戻す（NFR-001: 反映1秒以内）。 |
| UI-004 | 権限による表示制御はUIの配慮、防御はAPI | Owner専用ボタンはMemberに非表示にするが、権限の実体は常にAPI側のPolicy判定（BR-PRM）。UIの出し分けをセキュリティと混同しない。 |
| UI-005 | Traceability | 各画面はSCR・UC・REQ・エラーIDと対応させる。 |

## 2.1 デザイントークン

全12画面で使い回す値をここで確定する。個別画面で新しい色・余白を発明しない。**基本セットはEC Site（`EC_Site/docs/06_画面設計.md` 2.1節）と同一のTailwindデフォルトスケールを踏襲**し（ポートフォリオ内の一貫性・二重管理回避）、本システム固有のカンバン系トークンのみ追加する。

### 基本トークン（EC Siteと共通）

| 区分 | トークン | Tailwindクラス例 | 用途 |
| --- | --- | --- | --- |
| カラー | Primary | `blue-600` / hover `blue-700` | 主要CTA（プロジェクト作成、タスク作成、招待） |
| カラー | Secondary | `gray-600` | 補助ボタン（キャンセル、閉じる） |
| カラー | Background | `gray-50` | ページ背景 |
| カラー | Surface | `white` | カード・モーダル・パネル背景 |
| カラー | Text-primary | `gray-900` | 見出し・本文 |
| カラー | Text-secondary | `gray-500` | キャプション・補足・Empty文言 |
| カラー | Border | `gray-200` | カード枠・区切り線・スケルトン枠 |
| カラー | Success | `green-600` | 完了メッセージ |
| カラー | Warning | `amber-500` | 注意（期限接近等、下記Due-soonと共用） |

| タイポ | H1 / H2 / H3 / Body / Caption | `text-2xl font-bold` / `text-xl font-semibold` / `text-lg font-semibold` / `text-base` / `text-sm text-gray-500` | EC Siteと同一の5段階 |
| スペーシング | 基準単位 | Tailwind標準スケール（4px刻み） | `gap-2`/`p-4`/`p-6`/`gap-8`を基本 |

# 5. 画面詳細

## 5.0 状態バリエーション 共通ルール

EC Site `06_画面設計.md` 5.0節と同一の書式を初版から全画面へ適用する。

| 共通ルール | 内容 |
| --- | --- |
| Loading（取得系） | スケルトン（`Border`色の枠 + `Background`色のバー）で件数分プレースホルダー表示 |
| Loading（送信系） | 送信ボタンはButtonトークンのloading状態（二重送信防止）。応答までフォーム非活性 |
| Empty | 0件は正常応答。「なぜ0件か + 次に取るべき行動」を`Text-secondary`で案内 |
| Error（通信失敗） | 「取得に失敗しました。」+ 再試行ボタン |
| Error（業務エラー） | E001〜E011は7章のメッセージを`Error`色で表示。E003は該当Inputのerror状態 + 直下赤字 |

## 5.1 SCR-001 ログイン画面

| Empty | 対象外 |
| Error | E001/E003をフォーム上部に表示。E001はどちらの項目が誤りかを明示しない |

## 5.4 SCR-004 カンバンボード

本システムの中核画面（UC-005/011/012）。

### 表示項目
- Status-todo / Status-in_progress / Status-done の3列。各列にタスク件数
- タスクカード: タイトル、担当者アバター（未割当はプレースホルダー）、期限（Due-overdue/Due-soonトークンで強調）、優先度バッジ（Priority-*トークン）
- 検索・絞込バー: キーワード（タイトル部分一致）、担当者、ステータス（REQ-013）
- 「タスク作成」ボタン（→SCR-006）

### 操作
- カードのD&D: 列間移動=ステータス変更、列内移動=並び替え（UC-012）。楽観的更新、失敗時は巻き戻し（UI-003）
- カード選択 → SCR-005（URLが変わる: `/projects/:id/tasks/:taskId`）
- lg未満（UI-002）: 列を縦積み表示、D&D無効。ステータス変更はSCR-005のプルダウンで代替

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 3列の骨格 + カードスケルトンを表示 |
| Empty | 列単位で「タスクなし」を`Text-secondary`表示。全列0件なら「最初のタスクを作成しましょう」+ 作成ボタン。検索絞込0件は「条件に一致するタスクがありません」+ 条件リセット |
| Error | 取得失敗は再試行。D&DのE006（archived）はArchived-bannerへ誘導、E007（他者による削除）はカードを消して再取得 |
| （bonus） | Reverb稼働時、他メンバーの変更が自動反映される。切断時はポーリングに縮退（BR-NTF-006）し、UI上の区別は設けない |

---

## 5.5 SCR-005 タスク詳細（パネル）

### 表示項目
- タイトル、説明、担当者、期限、優先度、ステータス、作成者・作成日時
- コメント一覧（投稿者名・日時、自分のコメントに削除ボタン、Ownerは全コメントに削除ボタン）
- 添付ファイル一覧（元ファイル名、サイズ、アップロード者。ダウンロード/削除ボタン）

### 入力項目
- 各フィールドのインライン編集（タイトル Required/Max 200、説明 Max 5000、期限 日付、優先度・ステータス プルダウン、担当者 メンバーのみのセレクト — BR-TSK-003）
- コメント入力（Required / Max 2000）、ファイル選択（10MB以下・許可種別・タスクあたり20件以下 — BR-FIL-001）

### 操作
- 編集は項目単位で即時保存（UC-009）。担当者変更時は新担当者へ通知（BR-NTF-001）
- 「タスクを削除」ボタン: Ownerのみ表示（UI-004）。確認ダイアログ後、連動削除（BR-TSK-007）してSCR-004へ
- パネルを閉じる → SCR-004（URLもボードに戻る）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | パネル内をブロック単位でスケルトン表示。ファイルアップロード中は進捗表示 + Button loading |
| Empty | コメント0件「まだコメントがありません」、ファイル0件「添付ファイルなし」（いずれも`Text-secondary`） |
| Error | E007（削除済みタスク、通知からの遷移で発生 — G-07）→「タスクが見つかりません」+ ボードへ戻る。E003（ファイル制限）は選択直後にクライアント側でも検証して即表示 |

## 5.6 SCR-006 タスク作成（モーダル）

### 入力項目
- タイトル（Required / Max 200）、説明（Max 5000）、担当者（任意・メンバーのみ）、期限（任意・過去日許容 BR-TSK-004）、優先度（デフォルトmiddle）

### 操作
- 「作成する」→ status=todo・列末尾で作成（UC-008）。担当者指定時は通知発火（本人除外）。完了後モーダルを閉じボードに反映

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 「作成する」押下後はButton loading状態 |
| Empty | 対象外 |
| Error | E003は該当Input直下。E006（archived）はモーダルを閉じてバナー表示 |

## 5.7 SCR-007 メンバー管理

### 表示項目
- メンバー一覧（氏名、メール、ロールバッジ、参加日時）。Memberには閲覧のみ（操作列非表示、UI-004）

### 入力項目（Ownerのみ）
- 招待: メールアドレス（Required / Email形式。登録済みユーザーであること — Service判定）
- ロール変更: owner⇔memberセレクト。除名: 確認ダイアログ付きボタン

### 操作
- 招待 → 即時参加（BR-PRJ-004）。除名 → 担当タスクの担当解除を伴う（BR-PRJ-005、確認ダイアログにその旨を明記する）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 一覧行スケルトン。招待・除名実行中はButton loading |
| Empty | 対象外（自分自身が必ず1行いる） |
| Error | E011（既参加）「このユーザーは既にメンバーです」。E007（未登録メール）は標準文言のまま（登録有無を丁寧に区別しない、BR-PRJ-004）。E006（最後のOwner保護）「最後のOwnerは変更できません。先に他のメンバーをOwnerにしてください。」（7章 補足の文脈メッセージ） |

## 5.9 SCR-009 通知一覧（ドロップダウン + ページ）

### 表示項目
- ドロップダウン: 最新10件（種別アイコン、本文要約、経過時間、未読ドット）+ 「すべて表示」
- ページ: 全通知（新しい順、ページネーション）、未読/全件フィルタ
- 通知本文の書式: `task_assigned`「[タスク名]の担当になりました」/ `task_commented`「[タスク名]にコメントがあります」/ `task_due_soon`「[タスク名]の期限が近づいています（残り24時間以内）」

### 操作
- 通知選択 → 該当タスク（SCR-005）へ遷移 + 当該通知を既読化
- 「すべて既読にする」→ 一括既読（BR-NTF-004）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 行スケルトン |
| Empty | 「通知はありません」（`Text-secondary`。業務が平常であることを示す正常状態） |
| Error | 取得失敗は再試行。task_idがNULL（タスク削除済み）の通知はクリック無効とし「タスクは削除されました」をインライン表示（遷移しない — 03 UC-015 3-a。G-07は直接URLアクセス用） |

## 5.12 SCR-012 プロジェクト管理（Admin）

### 表示項目
- 全プロジェクト一覧（名称、Owner名、メンバー数、状態、作成日時）。**タスク・コメント・ファイルの内容は表示しない**（BR-PRM-004）

### 操作
- 強制アーカイブ（確認ダイアログ付き）。復元ボタンは置かない（復元はOwnerのみ — UC-018）

| Error | 取得失敗は再試行。強制アーカイブ失敗はエラートースト + 再取得 |

# 9. まとめ

全12画面をPC First（UI-002）+ OOUI（UI-001）で設計した。デザイントークンはEC Siteの基本セットを踏襲しつつ、本システム固有のカンバン系トークン（優先度・ステータス・期限・未読）を追加で確定した — 以降の実装はこのトークン以外の色・余白を使わない。状態バリエーションはEC Site v1.2→v1.3の「後追い展開」の教訓を活かし、初版から全画面に適用済み。UIにおける権限の出し分け（UI-004）はあくまで配慮であり、防御の実体がAPI側のPolicyにあることを全画面で徹底する。
