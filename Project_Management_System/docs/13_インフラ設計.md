# 1. 本書の目的

本書はProject Management Systemのインフラ構成を定義する。Project 01/02のDocker Compose / AWS基本構成を踏襲しつつ、本プロジェクト特有の3要素 — ①SPA配信（ビルド成果物 + historyモードフォールバック）、②WebSocket（Reverb、bonus）、③Privateバケット限定のS3 — を追加定義する。

---

# 2. インフラ設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| INF-POL-001 | Docker First | ローカル・本番ともにDocker Composeベース（Project 01/02と統一）。 |
| INF-POL-002 | 段階的AWS移行 | 開発はローカルDocker、本番はAWS EC2 + RDS + S3。 |
| INF-POL-003 | Secret外部化 | DB Password・AWSキー・Reverbシークレットは環境変数で管理し、コードにもGitにも含めない。 |
| INF-POL-004 | 最小権限 | S3はLaravel用IAMロールのみ読み書き可。Publicアクセスは全ブロック（Project 02と異なり読み取りも公開しない — BR-FIL-002）。 |
| INF-POL-005 | FE/BEビルド分離 | SPAのビルド（Node）とAPIの実行（PHP）を分離し、本番EC2にNodeランタイムを置かない（ビルドはCIで実施）。 |

# 7. Docker設計

## 7.1 Container構成

| Container | 稼働環境 | 役割 |
| --- | --- | --- |
| nginx | 共通 | SPA静的配信 + `/api`リバースプロキシ + `/app` WebSocketプロキシ |
| app（PHP-FPM/Laravel） | 共通 | REST API本体 |
| db（MySQL 8） | ローカルのみ | 開発用DB（本番はRDS） |
| scheduler | 共通 | `php artisan schedule:work`（期限接近バッチ、Project 02構成を流用） |
| vite | **ローカルのみ** | Vite dev server（HMR）。本番ビルドはCIで実施 |
| reverb（bonus） | 共通 | `php artisan reverb:start`（appと同一イメージ・コマンド差替え） |

## 7.2 Container起動方針

Project 02の`docker-compose.yml`をベースに`vite` / `reverb`サービスを追加する。Restart Policyは`unless-stopped`（viteを除く）。reverbはappと同じイメージを使いコマンドのみ変える（schedulerと同じパターン）。

# 8. Nginx設計

SPA + API + WebSocketの3系統ルーティング。**Project 01/02（Blade）との最大の差分はSPAフォールバック**である。

| location | 内容 |
| --- | --- |
| `/api/*` | PHP-FPM（Laravel）へプロキシ |
| `/app/*`（bonus） | Reverbへプロキシ。`Upgrade` / `Connection`ヘッダを付与（WebSocketアップグレード） |
| `/broadcasting/auth` | Laravelへ（チャンネル認可、`12_詳細設計書.md` 8章） |
| その他すべて | `try_files $uri /index.html` — **SPA historyモードのフォールバック**。`/projects/1/tasks/5`のようなディープリンク（SCR-005のURL直リンク、`05_画面遷移図.md`）を直接開いてもindex.htmlを返し、ルーティングはVue Routerが行う |

隠しファイル拒否・セキュリティヘッダ（`14_セキュリティ設計.md` 5章）はProject 02のパターンを踏襲する。

# 14. 環境変数・Secret管理

## 14.1 バックエンド（実行時に参照）

| 変数 | 内容 |
| --- | --- |
| DB_HOST / DB_DATABASE / DB_USERNAME / DB_PASSWORD | RDS接続情報 |
| AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_BUCKET | S3接続情報（本番はIAMロール推奨、キー直書きは開発時のみ） |
| REVERB_APP_ID / REVERB_APP_KEY / REVERB_APP_SECRET | Reverb認証（bonus。SECRETは絶対にコミットしない） |
| SANCTUM関連 / CORS_ALLOWED_ORIGINS | SPA配信オリジンのみ許可（NFR-013、`14_セキュリティ設計.md` 5章） |

## 14.2 フロントエンド（**ビルド時に焼き込み** — SPA特有の注意）

| 変数 | 内容 |
| --- | --- |
| VITE_API_BASE_URL | APIベースURL |
| VITE_REVERB_APP_KEY / VITE_REVERB_HOST | Echo接続情報（bonus） |

**注意**: `VITE_`変数はビルド時にJSへ焼き込まれ、**配布物として全ユーザーに見える**。したがって①シークレットは絶対に`VITE_`に入れない（REVERB_APP_KEYは公開前提のキー、SECRETはバックエンド側のみ）、②環境ごとに再ビルドが必要（ビルド成果物の環境間流用は不可）。この2点はProject 01/02（サーバーサイドのみ）には存在しなかったSPA特有の制約である。

# 16. CI/CD設計

GitHub Actions。**FE/BEの2ジョブ構成**（Project 01/02から拡張）:

| ジョブ | 内容 |
| --- | --- |
| backend | PHPUnit実行（権限マトリクス試験を含む — `01_企画書.md` KPI） |
| frontend | `tsc --noEmit`（TypeScript strict、NFR-019）+ `vite build` |
| deploy（main） | FEビルド成果物 + BEコードをEC2へ配置、`php artisan migrate`、コンテナ再起動 |

mainブランチ直接push禁止・PR必須（`00_開発計画書.md` 7章のチーム開発プラクティス）はGitHubブランチ保護で強制する。

# 21. まとめ

Project 01/02のインフラ資産（Docker Compose・VPC・Scheduler）を最大限流用しつつ、SPA特有の3点 — ①Nginxの`try_files`フォールバック（ディープリンク対応）、②FE/BEビルド分離（本番にNodeを置かない）、③`VITE_`変数のビルド時焼き込みとシークレット禁止 — を本書で確定した。WebSocket（Reverb）はネットワーク的にもbonusとして隔離されており（8080非公開・Nginx経由のみ）、失敗してもコア構成に影響しない。
