# 1. 本書の目的

本書はProject Management Systemの単体試験（PHPUnit / Feature + Unit）を定義する。本プロジェクトのKPI（`01_企画書.md` 11章）である**「権限マトリクス網羅率100%」**を8章で担保する — Project 02の重点が「金額・在庫の整合」だったのに対し、本プロジェクトの重点は**「認可の網羅」と「並び順・通知の整合」**である。

# 2. 試験方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UT-POL-001 | Requirement Based | REQ / FUNC / API / BRに対応した試験を作成する。 |
| UT-POL-002 | Normal / Abnormal / Boundary | 正常系・異常系に加え、文字数・サイズ・件数の境界値（最小未満/最小/最大/最大超）を確認する。 |
| UT-POL-003 | **Matrix Exhaustive** | `02_要件定義書.md` 8章の権限マトリクス20操作×4ロール=**80セル全部**を試験する。〇セル（成功）だけでなく×/秘セル（E002/E007が返ること）を必ず含める — 認可バグは拒否側でしか見つからない（`14_セキュリティ設計.md` 7.2節）。 |
| UT-POL-004 | Database Assertion | ロール・position・通知・スナップショットはDB保存結果を直接確認する（レスポンスだけで判定しない）。 |
| UT-POL-005 | Concurrency | カンバン移動（列ロック）とOwner相互降格（BR-PRJ-002）の同時実行試験を必須とする。 |
| UT-POL-006 | Snapshot Integrity | タスク改題・削除後も通知の`task_title`が発火時点の値のまま変わらないことを確認する（TBL-007、REL-011）。 |
| UT-POL-007 | Bonus隔離 | Reverb配信はEvent fakeで「発火の有無・タイミング（afterCommit）」のみ検証する。実ソケット接続の試験は結合試験（`16_結合試験仕様書.md`）へ委ねる。 |

# 6. 判定基準

| 判定 | 条件 |
| --- | --- |
| Pass | HTTPステータス、エラーコード、レスポンス内容、**DB状態（position・ロール・通知行）**が仕様通り |
| Fail | 期待と異なる。特に: 認可の取り違え（E002⇔E007）、positionの重複・逆転、ロールバック漏れ、通知の過剰/欠落 |
| Blocked | 依存機能の未実装等で実行不可 |

# 8. 権限マトリクス網羅試験（重点・KPI）

`02_要件定義書.md` 8章（v1.2）の20操作×4ロール=**80セルを1セル=1試験**で網羅する。Test IDは`UT-PRM-<操作番号>-<A/O/M/N>`（A=Admin、O=Owner、M=Member、N=非メンバー）。

**期待値の規約（02 v1.2の凡例補足と一致）**: Adminの×は`/projects/{projectId}/...`配下では**E007**（スコープ付きバインディングで遮断、Policyまで到達しない — `12_詳細設計書.md` 8章）。E002が返るのは「バインディングを通過した後にPolicyで拒否される」ケース（Memberの越権、API-007のAdmin、`/admin/*`の非Admin）のみ。

# 17. まとめ

本書の重心は8章 — 権限マトリクス80セルの**全数試験**であり、〇セルの成功確認だけでなく×/秘セルでE002/E007が正しく（かつ文言まで同一に）返ることを検証する。これがKPI「権限マトリクス網羅率100%」の実体である。次点の重点は11.2章のカンバン移動（隙間法・リナンバリング・同時実行・古いクライアント状態のフォールバック）と13章の通知fan-out（本人除外・スナップショット・afterCommit）。いずれもProject 02で習得したDB整合の考え方を「認可と並び順」へ応用したものである。
