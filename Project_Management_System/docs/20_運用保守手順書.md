# 1. 本書の目的

本書はProject Management Systemの日常運用・障害対応の手順を定義する。Project 02（EC）が決済・在庫を重点としたのに対し、本プロジェクトは**権限（IDOR）・カンバンの整合・S3孤児オブジェクト・Reverb（bonus）**を重点的に定義する。設計フェーズを通じて20_運用保守手順書へ委ねられていた項目（11章14.2節のS3孤児棚卸、13章のバッチ監視、14章のAudit調査、19章のロールバック後始末）はすべて本書に集約する。

# 3. 日次運用タスク

```
□ /api/health 死活確認
□ auditログでログイン失敗（E001）の当日件数を確認（総当たり攻撃の兆候がないか、14 4.6節のレートリミットが機能しているか）
□ notifications の task_due_soon が当日分作成されているか確認（SendDueSoonNotificationsの稼働確認）
□ Reverbコンテナの稼働状態確認（bonus、停止していてもコアは継続するが早期発見のため）
```


# 6. 権限・認可関連の障害対応（重点）

## 6.1 IDOR疑い（他プロジェクトのデータが見えた/操作できたとの報告）

```
1. 報告された画面・APIエンドポイントを特定する
2. 当該APIがスコープ付きルートバインディング（12_詳細設計書 8章）配下にあるか確認する
   → 配下にない新規APIの場合、バインディング未実装がバグの直接原因である可能性が高い
3. auditログで実際に発行されたリクエストのuser_id・対象project_idを確認する
4. 15_単体試験仕様書 8章の該当セル（UT-PRM-xx）を再実行し、Fail箇所を特定する
5. 修正後、8章の権限マトリクス全セル（80セル）を再実行してから緊急リリースする（部分修正が
   他のセルを壊していないことの確認 — 権限まわりの修正は必ず全数再試験、部分試験で済ませない）
```

## 6.2 最後のOwner消失・Owner不在プロジェクトの発生

```
1. project_membersでrole=ownerが0件のプロジェクトをクエリで特定する
   （BR-PRJ-002のロック（12_詳細設計書 11章）が機能していれば理論上発生しない。発生していれば
    MemberService::changeRole/removeOrLeaveのロック漏れをバグとして最優先で調査）
2. 発生していた場合、当該プロジェクトの最古のメンバー（project_membersのcreated_at最小）を
   一時的にAdminがDB操作でOwnerへ復帰させる（API経由の復旧手段がないための例外対応。実施内容は
   auditログに手動で追記する）
3. 原因コードを緊急修正し、15_単体試験仕様書 UT-MBR-009（Concurrency試験）を再実行してから
   リリースする
```

## 6.3 Admin無効化後もログインできてしまう（14 v1.1 4.4節の実装漏れ疑い）

```
1. 該当ユーザーのpersonal_access_tokensが無効化操作時に削除されているか確認する
2. 削除されていない場合、UserService（Admin無効化処理）がトークン削除を欠落させている実装バグとして扱う
   （14_セキュリティ設計.md 4.4節で確定済みの仕様 — 「無効化=即時失効」からの逸脱）
3. 応急対応として該当ユーザーのトークンを手動でDBから削除する
4. 16_結合試験仕様書 IT-SCN-005を再実行してから修正をリリースする
```

# 7. カンバン・通知関連の障害対応（重点）

## 7.1 カンバンのposition重複・順序崩れ

```
1. 対象プロジェクト×ステータス列のtasksをposition昇順で取得し、重複・逆転を確認する
   （12_詳細設計書 5.6のFOR UPDATE直列化が機能していれば理論上発生しない）
2. 発生していた場合、該当列を1024刻みで手動リナンバリングする（09_テーブル定義 11章-1と同じロジック:
   ORDER BY position, id で暫定順序を確定し、上から1024,2048,...を振り直す）
3. 原因が同時アクセスであれば、15_単体試験仕様書 UT-MOV-006を再実行して直列化を再確認する
```

## 7.2 通知が来ない・二重に来る

```
1. 「来ない」場合:
   a. notify()の3条件（recipient非null・本人以外・NotificationService呼び出し自体の到達）を
      アプリログで確認する（12_詳細設計書 5.12）
   b. Reverb経由の場合はbonus障害の可能性が高い — 9章の手順でReverbを確認し、コア（ポーリング）
      側は正常か切り分ける
2. 「二重に来る」場合:
   a. task_due_soonであればSendDueSoonNotificationsの重複防止チェック（09_テーブル定義 11章-2）の
      INDEX(task_id, type)が外れていないか確認する
   b. task_assigned/task_commentedであれば、afterCommitの二重登録（12_詳細設計書 5.12-4）が
      疑われるため、該当Serviceのトランザクション境界を確認する
```

# 8. ファイル・S3関連の運用（孤児オブジェクト棚卸）

11_基本設計書 14.2節・13_インフラ設計 12章・16_結合試験仕様書 IT-S3-007が本節へ委ねた運用手順。DBが正・S3孤児許容の設計上、**孤児オブジェクトの発生自体は異常ではない**（アップロード失敗時のロールバック、削除時のS3側エラー等で発生しうる正常な副産物）。

```
1. 週次でS3バケット（pms-task-files-prod）のオブジェクト一覧を取得する
2. task_filesテーブルのs3_key全件と突き合わせ、DBに存在しないS3キーを抽出する
3. 抽出されたオブジェクトが直近7日以内に作成されたものは対象外とする
   （アップロード処理の完了待ち・削除処理のタイムラグの可能性があるため）
4. 7日以上前のものを孤児オブジェクトと確定し、削除する（または調査用に一時退避してから削除）
5. 孤児化件数が急激に増加している場合、11_基本設計書14.2節のDB↔S3順序処理でエラーが
   頻発している兆候のため、アプリログのS3操作失敗ログを調査する
```

# 9. Reverb障害対応（bonus）

19_リリース手順書 8章が本節へ委ねた復旧手順。OPS-POL-003に従い、コア機能への影響がないことを前提に対応する。

```
1. docker compose ps でreverbコンテナの状態を確認する
2. 停止していた場合、docker compose up -d reverb で再起動する
3. 再起動後、Nginx経由のwss接続が確立するか確認する（16_結合試験仕様書 IT-WS-007相当の手動確認）
4. SPA側は自動的にポーリング（30秒間隔）へ縮退しているため、復旧まではユーザー影響を「通知の
   反映が最大30秒遅れる」程度に抑えられていることを確認する（BR-NTF-006 — 復旧を急ぐ必要はない）
5. 頻発する場合、EC2のメモリ・CPU余力を確認する（t3.micro上での複数コンテナ同居によるリソース
   逼迫が疑われる — 13_インフラ設計 19章のコスト制約とのトレードオフ、将来的なインスタンス
   タイプ見直しの判断材料とする）
```

# 11. セキュリティ運用

14_セキュリティ設計 11章（監査ログ）・12章（脆弱性管理）を参照。加えて以下を運用ルールとする。

| 事前通知 | 商用ローンチ後を想定した方針: SPAログイン画面へのバナー表示 + 実施予定日時をリリースノートに記載。現状（学習用途・利用者なし）は本手順を省略してよい |

| 実施記録 | 実施日時・作業内容・所要時間を本節末尾または別途運用ログに追記する（現状は未実施のため記録なし） |

# 13. 変更管理・エスカレーション

個人開発プロジェクトのため、変更管理・承認フローは簡略化する。ただし**権限・認可に関わる変更**（Policy・スコープ付きバインディング・Middleware）は、必ず15_単体試験仕様書 8章の権限マトリクス80セル**全数**を再実行してからリリースする（6.1節と同一方針 — 部分試験での見切りリリースを禁止する）。

# 15. まとめ

本書はProject 02（決済・在庫中心）と対照的に、**権限（IDOR）・カンバン整合・S3孤児・bonus隔離**の4点を運用の重点とした。6章の障害対応はいずれも「発生した場合は設計上のロック・スコープ・トランザクション境界のどこかが破れている」ことを意味するため、応急対応で終わらせず必ず該当する単体/結合試験を再実行してからリリースする方針を徹底する。これにより21文書構成（PMS-000〜020）の設計フェーズを完了する。
