# 1. 開発目的

本プロジェクトは、Portfolio Roadmap（SE/BrSE 2026-2027）における Project 03 として位置づけられる、チーム向けプロジェクト・タスク管理システム（Kanban方式）である。

Project 01（HR & Attendance System）でCRUD・認証・グローバルRBACを、Project 02（EC Site）でトランザクション・決済・在庫の整合性を習得したのに対し、本プロジェクトは以下の新しい難易度軸を目的とする。

- **SPA分離アーキテクチャ**: 初のBlade脱却。フロントエンド（Vue 3 SPA）とLaravel APIを完全分離し、SPA向けトークン認証・CORS・ビルド/デプロイ分離を習得する
- **プロジェクト単位の認可（2層ロールモデル）**: Project 01のグローバルロールと異なり、「システム全体のロール」×「プロジェクトごとのロール」の2層で権限を判定する（Laravel Policyの本格活用）。IDORとの戦いが全APIに及ぶ
- **通知基盤**: 「誰の・どの操作が・誰への通知を生むか」というfan-out設計と、既読管理。bonusとしてWebSocket（Laravel Reverb）によるリアルタイム配信
- **ファイルアップロード**: S3への添付ファイル保存と、サイズ・種別検証、権限に基づく取得制御
- **チーム開発の作法**: 個人開発ながらfeatureブランチ + Pull Request + セルフレビューのフローを全期間で実践する（Roadmapの学習目標「Team Development」に対応）

# 2. 開発範囲

## 2.1 開発対象（In Scope）

- 会員登録・ログイン・ログアウト（Sanctum、SPA向けトークン認証）
- プロジェクト管理（作成・編集・アーカイブ）
- メンバー管理（メールアドレス指定の招待、ロール変更、除名）
- タスク管理（CRUD、担当者、期限、優先度、ステータス）
- カンバンボード（ドラッグ&ドロップによるステータス変更・列内並び替え）
- コメント（タスクへの投稿・削除）
- ファイル添付（タスクへのアップロード・ダウンロード・削除、S3保存）
- アプリ内通知（担当割当・自分の担当タスクへのコメント・期限接近の3種、既読管理）
- 期限接近通知の定期バッチ（Project 02のScheduler資産を流用）
- （bonus）Laravel Reverbによる通知・カンバンのリアルタイム配信

## 2.2 開発対象外（Out Scope）

- ガントチャート・工数管理・レポート集計
- 複数ワークスペース／組織（テナント）機能
- メール通知・Slack等の外部サービス連携
- コメントの@メンション解析（通知種別としては将来拡張、13章）
- 全文検索（タスク検索はタイトル部分一致のみ）
- ゲスト共有リンク（未ログイン閲覧）
- モバイルアプリ・多言語UI（日本語のみ。内部文書はベトナム語併記）

# 3. 対象ユーザー

| ユーザー種別 | 層 | 説明 |
|---|---|---|
| Admin | グローバル | システム管理者。全ユーザー・全プロジェクトの管理（運用目的のみ、業務参加はしない） |
| Owner | プロジェクト単位 | プロジェクト作成者または昇格されたメンバー。メンバー招待・除名・プロジェクト設定・タスク削除が可能 |
| Member | プロジェクト単位 | 招待されたメンバー。タスクCRUD・カンバン操作・コメント・ファイル添付が可能 |

**2層ロールモデル（本プロジェクトの核心）**: `users.role`（`admin` / `user`）はシステム運用の権限、`project_members.role`（`owner` / `member`）はプロジェクト内の権限を表す。一般ユーザー（`user`）はプロジェクトAではOwner、プロジェクトBではMemberになり得る。Project 01の「Manager承認フロー」やProject 02の「注文状態遷移」と異なり、**すべてのAPI呼び出しで「対象リソースが属するプロジェクトのメンバーか？ロールは？」の判定**が必要になる。この判定表（権限マトリクス）は `02_要件定義書.md` で確定する。

# 4. システム概要

## システム構成

```
[Browser]
   │ HTTPS
   ▼
[Nginx (Reverse Proxy)]
   ├── /            → Vue 3 SPA（ビルド済み静的ファイル配信）
   ├── /api         → [Laravel API (PHP-FPM)] ──► [AWS S3 (添付ファイル)]
   │                        │
   │                        ▼
   │                  [MySQL 8 (RDS)]
   └── /app (WebSocket) → [Laravel Reverb Container]（bonus）
                              ▲
                              └── Laravel API（通知発火時にbroadcast）
```

フロントエンドはVue 3 SPAとし、Nginxがビルド済み静的ファイルを配信、`/api`をLaravelへリバースプロキシする。ローカル開発ではVite dev serverコンテナ、通知のリアルタイム配信（bonus）はReverb専用コンテナを追加する。Project 01/02のDocker Compose構成（nginx/app/db/scheduler）を土台に拡張する。

# 5. 主な機能

## 5.1 認証機能
登録・ログイン・ログアウト。Sanctumトークン認証をProject 02から踏襲するが、消費者向けではないため**トークン期限は8時間**（Project 01と同じ業務システム基準）に戻す。

## 5.2 プロジェクト管理機能
作成（作成者が自動的にOwner）・名称/説明編集・アーカイブ（論理無効化、物理削除しない）。

## 5.3 メンバー管理機能
登録済みユーザーをメールアドレス指定で招待（即時参加方式、招待承諾フローは対象外）。Ownerによるロール変更・除名。最後のOwnerの降格・脱退は禁止（孤児プロジェクト防止）。

## 5.4 タスク管理機能
タイトル・説明・担当者（プロジェクトメンバーのみ）・期限・優先度（low/middle/high）・ステータス（todo/in_progress/done）のCRUD。

## 5.5 カンバン機能
ステータス3列のボード表示。ドラッグ&ドロップで列間移動（ステータス変更）と列内並び替え（表示順変更）。表示順は`tasks.position`で管理し、同時操作の整合は`08_ER図.md`/`12_詳細設計書.md`で設計する。

## 5.6 コメント機能
タスク詳細でのコメント投稿・自分のコメントの削除。担当者への通知を発火する。

## 5.7 ファイル添付機能
タスクへのファイルアップロード（S3保存、サイズ・拡張子検証）・一覧・ダウンロード・削除。非メンバーからの取得を遮断する（権限判定を通したダウンロードURL発行。方式は`12_詳細設計書.md`/`14_セキュリティ設計.md`で確定）。
**→ 確定済み（2026/07/19）**: `14_セキュリティ設計.md` 6.2節で**案B（有効期限60秒のpresigned URLへの302リダイレクト）**に確定。権限判定はLaravel側に残る。

## 5.8 通知機能
通知種別は初期リリースで3種に限定する（全文書でこのenumを統一する）:

| 種別 | 発火条件 | 通知先 |
|---|---|---|
| `task_assigned` | タスクの担当者に設定された | 新担当者 |
| `task_commented` | 自分が担当するタスクにコメントが付いた | 担当者（コメント投稿者本人を除く） |
| `task_due_soon` | 期限24時間前（定期バッチが判定） | 担当者 |

アプリ内通知一覧・未読件数・既読化を提供。bonusでReverbによるリアルタイムpush（未対応環境・失敗時はポーリングにフォールバック）。

# 6. 使用技術

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | Vue 3 + TypeScript + Vite + Pinia + Vue Router | 下記「フロントエンド選定理由」参照 |
| CSS | Tailwind CSS | Project 02と同一。デザイントークン（06_画面設計）をユーティリティクラスへ直結できる |
| HTTPクライアント | fetchベースの自作APIクライアント | 依存を増やさない（axiosは採用しない — 個人ポートフォリオ全体の規約） |
| バックエンド | Laravel 12 / PHP 8.4 | Project 01/02と統一。Controller→Service→Model構成を踏襲し、新規学習をPolicy・通知・broadcastに集中する |
| 認可 | Laravel Policy | 2層ロールモデル（3章）の判定をPolicyに集約し、Controller散在を防ぐ |
| データベース | MySQL 8 | Project 01/02と統一 |
| リアルタイム（bonus） | Laravel Reverb | Laravel公式のWebSocketサーバー。Echo（クライアント）との組合せが標準構成で学習効率が高い |
| ストレージ | AWS S3 | 添付ファイル保存。Project 02の商品画像で習得したFilesystem(s3)を拡張 |
| インフラ | Docker Compose → AWS EC2 / RDS / S3 | Project 02構成 + vite(開発時)/reverbコンテナ |
| CI/CD | GitHub Actions | push時にバックエンドテスト + フロントエンドビルド/型チェックを実行 |

**フロントエンド選定理由（Vue 3 — ADR-001相当、詳細は`docs/adr/`に別途記録する）**:

- **候補**: React 18 / Vue 3（Roadmapでは「Vue.js hoặc React」と両論併記）
- **決定**: Vue 3（Composition API + TypeScript）
- **理由**: ①Roadmap全体ではProject 04/07がVue、Project 05/06/08がReactであり、本プロジェクトでVueを選ぶとProject 04と連続2案件でVueを深掘りでき、学習の文脈切替が減る。②SFC（テンプレート構文）はBlade経験からの移行コストが低く、本プロジェクトの主目的（SPA分離・認可・通知）に集中できる。③Composition APIの習得はProject 05以降のReact Hooks理解にも転移する
- **トレードオフ**: 求人市場はReact優位だが、ポートフォリオ全体でReact 3案件を確保済みのため問題ない。ReactはProject 05（E-Learning）で導入する

# 9. データベース概要

## テーブル一覧（7テーブル + フレームワーク管理分）

`users`, `projects`, `project_members`, `tasks`, `task_comments`, `task_files`, `notifications`

Roadmap記載の6テーブルに`users`（Project 01/02のパターンを踏襲）を加えた構成。Sanctumの`personal_access_tokens`等フレームワーク管理テーブルは数に含めない。最終的なカラム・制約は`08_ER図.md`/`09_テーブル定義.md`で確定する。

## ER概要

中心エンティティは`projects`。`project_members`が`users`×`projects`の中間テーブルとして**プロジェクト内ロールを保持**する（2層ロールモデルのDB表現。`UNIQUE(project_id, user_id)`）。`tasks`は`projects`に属し、担当者として`users`を参照する。`notifications`は通知種別enum + 参照先（タスク等）を持つ。カンバン表示順は`tasks.position`で管理する。

設計上の主要論点（08/09で判断を記録する）:

- `tasks.position`の型と採番方式 → `09_テーブル定義.md` 11章-1で「隙間法（1024刻み・中間値挿入・枯渇時リナンバリング）+ UNIQUEを張らず列単位の悲観的ロックで直列化」に確定済み
- `notifications`の参照先表現 → `08_ER図.md` REL-011で「明示FK（task_id, ON DELETE SET NULL）+ task_titleスナップショット」に確定済み（通知種別が3種ともタスク参照でありポリモーフィックの汎用性が不要なため）
- 除名されたメンバーが担当していたタスクの扱い → `02_要件定義書.md` BR-PRJ-005で「担当解除（assignee NULL化）」に確定済み

# 10. テスト計画

## 単体試験
Policy（権限マトリクス）の全組合せ（Admin/Owner/Member/非メンバー × 主要操作）、タスク状態・position更新、通知発火条件（3種それぞれの正例・負例）、ファイル検証（サイズ超過・不正拡張子）。

## 結合試験
API ↔ DB、API ↔ S3（アップロード/取得制御）、通知発火 ↔ Reverb配信（bonus）、期限バッチ ↔ 通知作成。

## システム試験
登録→プロジェクト作成→メンバー招待→タスク作成/割当（通知確認）→カンバン移動→コメント（通知確認）→ファイル添付→期限バッチ→既読化、の一連シナリオをOwner/Member両視点で実施。

## テスト項目（概要）
- 非メンバーによるプロジェクト資源（タスク・コメント・ファイル）へのアクセス拒否（IDOR）
- Owner専用操作（招待・除名・プロジェクト編集）のMember実行拒否
- 最後のOwnerの降格・脱退の拒否
- カンバン同時並び替えでpositionが破綻しないこと
- コメント投稿者本人に`task_commented`通知が飛ばないこと
- 通知の既読が他ユーザーの通知に影響しないこと

詳細は`15〜17`の各試験仕様書で定義する。
