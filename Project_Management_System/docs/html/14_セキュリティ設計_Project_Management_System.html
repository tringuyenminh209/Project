■ 2. 脅威モデルと設計方針
2.1 攻撃面の特徴

11_基本設計書.md 10章のとおり、本システムにはProject 02（EC）に存在した外部インバウンド連携（Stripe Webhook等）がない。また未認証ユーザーに公開する画面はログイン・登録のみである。したがって攻撃面は「認証済みユーザーによる越権（横方向・縦方向）」に集中する。

■ 4. 認証設計
4.1 認証方式

Laravel SanctumのPersonal Access Token（Bearerトークン）方式を採用する（NFR-010）。SPAはAuthorization: Bearer {token}ヘッダを全APIリクエストに付与する。

Sanctumにはもう1つ「SPA Cookieモード（httpOnly Cookie + CSRFトークン）」があるが、採用しない。理由:

観点Bearerトークン（採用）SPA Cookieモード（不採用）
トークンライフサイクル発行・8時間期限・ログアウト失効を明示的に実装 — 本プロジェクトの学習目標そのもの（00_開発計画書.md 1章「SPA向けトークン認証」）セッションに隠蔽され学習効果が薄い
WebSocket認可（bonus）Echo認可リクエストに同じBearerを載せるだけCookie依存で構成が複雑化
CSRFCookieを認証に使わないためCSRF攻撃自体が成立しない（8章 8.4）CSRFトークン往復の実装が必要
XSS時のトークン窃取弱い（→4.2節の緩和策で補う）強い（httpOnlyで読めない）

XSS耐性ではCookieモードが勝るが、その差分は4.2節の緩和策で受け止め、学習目標との整合を優先する。

4.2 トークン保管方式の確定 ★論点①

localStorageに保管する（12_詳細設計書.md 9.2節 authストアの前提を正式確定）。

リスクの直視: localStorageはJavaScriptから読めるため、XSSが1箇所でも成立するとトークンが窃取される。これを受け入れる代わりに、以下の緩和策をセットで必須とする。

#緩和策内容
1XSSの根本遮断Vueのテキスト補間のみ使用。v-htmlの使用を原則禁止（使用する場合はレビュー必須 + サニタイズ、8章 8.3）
2CSPscript-src 'self'を基本とするContent-Security-Policyヘッダ（5.2節）。インラインスクリプト・外部オリジンのJSを遮断
3有効期限8時間窃取されても当日限りで失効（NFR-010）
4ログアウトで即失効ログアウトAPIがサーバー側のトークンレコードを削除 — クライアント側の破棄だけに頼らない
5HTTPS全面化経路上の盗聴を遮断（NFR-016、5.1節）

タブを閉じても再ログイン不要というUX（05_画面遷移図.md G-01の前提）はlocalStorageだから成立する。sessionStorage（タブごと消失）はカンバンの複数タブ利用と相性が悪く不採用。

4.3 ログイン失敗時の応答設計

06_画面設計.md 5章（SCR-001）の「E001はどちらの項目が誤りかを明示しない」の設計根拠を定義する。

ルール内容
メッセージ統一メール不存在・パスワード不一致・アカウント無効の3ケースすべてで同一のE001「メールアドレスまたはパスワードが正しくありません。」を返す（API挙動は10_API設計.md 6章 API-002「不一致・inactive共通」、文言は06_画面設計.md 7章）
理由ケースを区別すると「このメールは登録済みか」を外部から判定できてしまう（アカウント列挙攻撃）。無効アカウントの区別も「退職者がいる」等の情報漏洩になる
レスポンス時間メール不存在時もパスワードハッシュ照合相当の処理時間を確保する（Hash::checkをダミー実行）— タイミング差による列挙を防ぐ
会員登録との整合登録API（重複メール時E003）は列挙可能だが、これは登録フローのUX上許容する（一般的なトレードオフ）。ただし4.6節のレートリミットで総当たりを抑制する
4.4 トークンライフサイクル
イベント挙動
ログイン成功既存トークンは失効させず新規発行（複数デバイス許容）。有効期限8時間
期限切れ / 失効後のAPIコールE010 (401) → SPAはauthストアを破棄しSCR-001へ（G-01、12_詳細設計書.md 9.3節のAPI Client共通処理）
ログアウト使用中トークンのみサーバー側で削除（他デバイスのトークンは維持）
パスワード変更（API-005 / REQ-025）現在のパスワード確認のうえ変更（不一致はE003）。成功時は使用中トークン以外をすべて失効 — 窃取済みトークンを持つ攻撃者を締め出すための標準動作（変更操作をした本人のセッションだけ生かす）
Admin無効化（API-033、REQ-026 / UC-017）既発行トークンを全件即時失効させる（v1.1で確定 — 16_結合試験仕様書.md IT-SCN-005が持ち越した論点）。無効化はログインを止めるだけでなく「今すぐアクセスを止める」操作であるべきで、有効期限内のトークンが生き残ると無効化の意図と矛盾するため。実装はusers.is_active更新と同一トランザクションで対象ユーザーのpersonal_access_tokensを削除する（12_詳細設計書.md UserService相当）。再有効化しても旧トークンは復活せず、再ログインが必要
4.5 パスワードポリシー
項目内容
保存bcryptハッシュ（Laravelデフォルト）。平文・可逆暗号は禁止
文字数8〜20文字（02_要件定義書.md 12章 データバリデーションルール users.password）
ログへの非出力パスワードはいかなるログにも出力しない（11章）

対象制限実装

■ 5. 通信セキュリティ
5.1 HTTPS

全通信をHTTPSとする（NFR-016）。HTTP(80)はHTTPSへリダイレクトのみ。WebSocket（bonus）もwss://（443のNginx経由、13_インフラ設計.md 8章）。

5.2 セキュリティヘッダ

Nginxで付与する（13_インフラ設計.md 8章。Project 02のセットにCSPを追加）。

ヘッダ値目的
Content-Security-Policydefault-src 'self'; script-src 'self'; connect-src 'self' wss://{host}; img-src 'self' data:; style-src 'self' 'unsafe-inline'XSS緩和の中核（4.2節の前提）。Viteビルドは外部CDNに依存しない構成のため'self'で成立する
X-Content-Type-OptionsnosniffMIMEスニッフィング防止（6.1節と連動）
X-Frame-OptionsDENYクリックジャッキング防止
Strict-Transport-Securitymax-age=31536000HTTPS強制
Referrer-Policystrict-origin-when-cross-originURL情報の漏洩抑制

style-src 'unsafe-inline'はVueコンポーネントの動的style（カンバンのドラッグ位置等）のために許容する — script-srcの'self'厳守が本丸であり、styleの緩和はトークン窃取に直結しない。

5.3 CORS

本構成ではNginxがSPAとAPIを同一オリジンで配信するため（13_インフラ設計.md 8章）、本番ではクロスオリジンリクエストは本来発生しない。それでもconfig/cors.phpで許可オリジンをSPA配信オリジンのみに明示設定する（NFR-013）— ローカル開発（Vite dev server :5173 → API）ではクロスオリジンになるため、環境変数CORS_ALLOWED_ORIGINSで環境ごとに切り替える（13_インフラ設計.md 14.1節）。*（全許可）は全環境で禁止。

■ 6. ファイルセキュリティ
6.1 アップロード検証（BR-FIL-001）
#検証内容
1サイズ10MB以下（FormRequest max:10240）
2拡張子許可リスト方式: png/jpg/jpeg/gif/webp/pdf/docx/xlsx/pptx/txt/csv/md/zip。拒否リスト方式は禁止（抜け道が無限にある）
3MIMEタイプLaravelのmimes:検証（ファイル内容ベースの判定）。拡張子とMIMEの両方が許可リストに一致して初めて受理（BR-FIL-001）
4件数1タスク20ファイルまで（Service層でカウント、BR-FIL-001）
5保存名S3キーは{uuid}.{ext}（BR-FIL-004）。元ファイル名をキーに使わない — パストラバーサル（../入りファイル名）と日本語ファイル名問題を同時に潰す。元名はDB列に保持し表示専用
6実行遮断アップロードされたファイルはS3（静的ストレージ）にのみ置かれ、EC2のWebルートには保存しない — サーバーで実行される経路が構成上存在しない
6.2 ダウンロード方式の確定 ★論点②

12_詳細設計書.md 5.10の2案から案B: 有効期限60秒のpresigned URLへの302リダイレクトを採用する（API-026。10_API設計.md 3.4節の302はこれ）。

観点案A: Laravelストリーム案B: presigned 302（採用）
権限判定Laravelで判定同じ（presigned発行前にINC-002を通す — 判定はLaravelが持ち続ける）
EC2負荷10MBファイルがPHP-FPMワーカーを占有し、t3.microの帯域を往復で消費転送はS3が直接担い、EC2はリダイレクト応答のみ
URL漏洩時漏洩リスクなし60秒で失効するため、共有された頃には無効。恒久URLは応答に含めない（BR-FIL-002）
実装シンプルStorage::temporaryUrl() + ResponseContentDispositionで元ファイル名を指定（日本語名はRFC 5987エンコード）

presignedは「権限判定通過者への短命な受け渡し」にすぎず、判定自体はLaravel側に残るため、BR-FIL-002「権限判定を通した経路のみ」を満たす。

6.3 削除とS3整合

BR-FIL-003（DB行とS3オブジェクトの両方削除）は11_基本設計書.md 14.2節の順序方針に従う: DBコミット成功後にS3削除。S3削除失敗は孤児オブジェクトとして許容（エラーログのみ）し、棚卸は20_運用保守手順書.mdに委ねる。逆順（S3先行削除）はDBロールバック時に「DB上は存在するのに実体がない」最悪ケースを生むため禁止。

■ 8. 入力検証・インジェクション対策
#脅威対策
8.1SQLインジェクション全DBアクセスをEloquent/クエリビルダ経由（プレースホルダ自動適用）。生SQLの文字列連結は禁止。ソート列等に外部入力を使う場合は許可リストで検証
8.2バリデーション境界全APIの入力はFormRequestで検証（12_詳細設計書.md 7章）。Service層は検証済み入力のみを受け取る
8.3XSS（格納型）タスク説明・コメント等の自由記述はDBに原文保存し、表示時にVueのテキスト補間でエスケープ。v-htmlは原則禁止（4.2節 緩和策#1）。将来Markdownレンダリングを入れる場合はサニタイザ必須
8.4CSRFCookieを認証に使わないため攻撃自体が成立しない（4.1節）。/broadcasting/authもBearerで認可
8.5Mass Assignment全モデルに$fillableを明示。$request->all()をそのままcreate()/update()へ渡さない（usersのrole列が書き換えられると縦方向越権に直結）
8.6情報列挙IDはBIGINT連番だが、E007の存在秘匿（7章）により総当たりしても情報が得られない。招待APIの未登録E007/既参加E011の区別は認証済み・同一プロジェクトOwnerにのみ返る情報であり許容（BR-PRJ-004）
■ 9. WebSocketセキュリティ（bonus）
#項目内容
1チャンネル認可private channelのみ使用（public channel禁止）。購読時に/broadcasting/authでSanctum認証 + チャンネルごとの認可（本人確認 / メンバーシップ確認 — 12_詳細設計書.md 8章）
2除名との連動チャンネル認可は購読時のみのため、除名後も既存接続の購読が残りうる。対策: ①REST側はE007で完全遮断済み（コア保護）、②SPAはE007受信時にWebSocketを再接続（再認可が走り購読失敗）。残存購読の強制切断まではbonusスコープ外として許容し、制約として明記する
3シークレットREVERB_APP_SECRETはバックエンドのみ。VITE_REVERB_APP_KEY（公開前提のキー）と混同しない（13_インフラ設計.md 14.2節）
4ネットワーク8080は外部非公開、Nginx経由のwss://（443）のみ（13_インフラ設計.md 5.1節）

■ 14. まとめ

本書で持ち越し論点2件を確定した。①トークンはlocalStorage保管とし、その弱点（XSS窃取）は「CSP + v-html禁止 + 8時間期限 + サーバー側失効」のセット運用で受け止める — どれか1つでも欠くと成立しない緩和策である点に注意。②ファイルダウンロードはpresigned URL 302とし、権限判定はLaravel側に残す。攻撃面が「認証済みユーザーの越権」に集中する本システムでは、7章のIDOR対策と権限マトリクスの×セル側の試験（15章へ引き渡し）が防御の本丸である。
