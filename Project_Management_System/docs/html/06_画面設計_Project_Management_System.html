■ 1. 本書の目的

Project Management Systemの各画面について、表示項目、入力項目、操作、バリデーション、エラー、状態バリエーションを定義する。02_要件定義書.md 03_ユースケース.md 05_画面遷移図.mdと整合し、10_API設計.md（作成後に8章へAPI対応を追記）および試験仕様書の基準とする。

■ 2. 画面設計方針
方針ID方針内容
UI-001OOUI（オブジェクト指向UI）「プロジェクト」→「タスク」という対象物を選んでから操作する設計とする。
UI-002PC First業務ツールのため1024px以上を最適化対象とする。タブレット・モバイルは閲覧・既読操作を保証し、カンバンD&Dは対象外としてよい（NFR-020。Project 02のMobile Firstとは意図的に方針が異なる）。
UI-003楽観的UI更新カンバンD&D等の高頻度操作は先に画面へ反映し、API失敗時に巻き戻す（NFR-001: 反映1秒以内）。
UI-004権限による表示制御はUIの配慮、防御はAPIOwner専用ボタンはMemberに非表示にするが、権限の実体は常にAPI側のPolicy判定（BR-PRM）。UIの出し分けをセキュリティと混同しない。
UI-005Traceability各画面はSCR・UC・REQ・エラーIDと対応させる。

5.0 状態バリエーション 共通ルール

EC Site 06_画面設計.md 5.0節と同一の書式を初版から全画面へ適用する。

共通ルール内容
Loading（取得系）スケルトン（Border色の枠 + Background色のバー）で件数分プレースホルダー表示
Loading（送信系）送信ボタンはButtonトークンのloading状態（二重送信防止）。応答までフォーム非活性
Empty0件は正常応答。「なぜ0件か + 次に取るべき行動」をText-secondaryで案内
Error（通信失敗）「取得に失敗しました。」+ 再試行ボタン
Error（業務エラー）E001〜E011は7章のメッセージをError色で表示。E003は該当Inputのerror状態 + 直下赤字

5.4 SCR-004 カンバンボード

本システムの中核画面（UC-005/011/012）。

表示項目

  • Status-todo / Status-in_progress / Status-done の3列。各列にタスク件数
  • タスクカード: タイトル、担当者アバター（未割当はプレースホルダー）、期限（Due-overdue/Due-soonトークンで強調）、優先度バッジ（Priority-*トークン）
  • 検索・絞込バー: キーワード（タイトル部分一致）、担当者、ステータス（REQ-013）
  • 「タスク作成」ボタン（→SCR-006）

操作

  • カードのD&D: 列間移動=ステータス変更、列内移動=並び替え（UC-012）。楽観的更新、失敗時は巻き戻し（UI-003）
  • カード選択 → SCR-005（URLが変わる: /projects/:id/tasks/:taskId）
  • lg未満（UI-002）: 列を縦積み表示、D&D無効。ステータス変更はSCR-005のプルダウンで代替

状態バリエーション

状態表示
Loading3列の骨格 + カードスケルトンを表示
Empty列単位で「タスクなし」をText-secondary表示。全列0件なら「最初のタスクを作成しましょう」+ 作成ボタン。検索絞込0件は「条件に一致するタスクがありません」+ 条件リセット
Error取得失敗は再試行。D&DのE006（archived）はArchived-bannerへ誘導、E007（他者による削除）はカードを消して再取得
（bonus）Reverb稼働時、他メンバーの変更が自動反映される。切断時はポーリングに縮退（BR-NTF-006）し、UI上の区別は設けない
5.5 SCR-005 タスク詳細（パネル）

表示項目

  • タイトル、説明、担当者、期限、優先度、ステータス、作成者・作成日時
  • コメント一覧（投稿者名・日時、自分のコメントに削除ボタン、Ownerは全コメントに削除ボタン）
  • 添付ファイル一覧（元ファイル名、サイズ、アップロード者。ダウンロード/削除ボタン）

入力項目

  • 各フィールドのインライン編集（タイトル Required/Max 200、説明 Max 5000、期限 日付、優先度・ステータス プルダウン、担当者 メンバーのみのセレクト — BR-TSK-003）
  • コメント入力（Required / Max 2000）、ファイル選択（10MB以下・許可種別・タスクあたり20件以下 — BR-FIL-001）

操作

  • 編集は項目単位で即時保存（UC-009）。担当者変更時は新担当者へ通知（BR-NTF-001）
  • 「タスクを削除」ボタン: Ownerのみ表示（UI-004）。確認ダイアログ後、連動削除（BR-TSK-007）してSCR-004へ
  • パネルを閉じる → SCR-004（URLもボードに戻る）

状態バリエーション

状態表示
Loadingパネル内をブロック単位でスケルトン表示。ファイルアップロード中は進捗表示 + Button loading
Emptyコメント0件「まだコメントがありません」、ファイル0件「添付ファイルなし」（いずれもText-secondary）
ErrorE007（削除済みタスク、通知からの遷移で発生 — G-07）→「タスクが見つかりません」+ ボードへ戻る。E003（ファイル制限）は選択直後にクライアント側でも検証して即表示

表示項目

■ 7. エラーメッセージ
エラーコード画面表示メッセージ
E001メールアドレスまたはパスワードが正しくありません。
E002この操作を実行する権限がありません。
E003入力内容に誤りがあります。
E006この操作は現在の状態では実行できません。
E007対象のデータが見つかりません。
E010ログインが必要です。
E011この操作は既に行われています。

補足1: エラーコードの意味はProject 01/02と共通の語彙（02_要件定義書.md 16章）。E004/E005/E008/E009は本システムでは使用しない。

補足2: 画面固有の文脈に合わせ、上表の標準文言を言い換えて表示してよい（例: SCR-007の最後のOwner保護はE006を「最後のOwnerは変更できません。先に他のメンバーをOwnerにしてください。」と表示）。ただしエラーコード自体は変更しない（試験仕様書はコードで判定する）。

■ 9. まとめ

全12画面をPC First（UI-002）+ OOUI（UI-001）で設計した。デザイントークンはEC Siteの基本セットを踏襲しつつ、本システム固有のカンバン系トークン（優先度・ステータス・期限・未読）を追加で確定した — 以降の実装はこのトークン以外の色・余白を使わない。状態バリエーションはEC Site v1.2→v1.3の「後追い展開」の教訓を活かし、初版から全画面に適用済み。UIにおける権限の出し分け（UI-004）はあくまで配慮であり、防御の実体がAPI側のPolicyにあることを全画面で徹底する。
