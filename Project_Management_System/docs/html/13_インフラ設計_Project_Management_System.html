■ 1. 本書の目的

本書はProject Management Systemのインフラ構成を定義する。Project 01/02のDocker Compose / AWS基本構成を踏襲しつつ、本プロジェクト特有の3要素 — ①SPA配信（ビルド成果物 + historyモードフォールバック）、②WebSocket（Reverb、bonus）、③Privateバケット限定のS3 — を追加定義する。

■ 2. インフラ設計方針
方針ID方針内容
INF-POL-001Docker Firstローカル・本番ともにDocker Composeベース（Project 01/02と統一）。
INF-POL-002段階的AWS移行開発はローカルDocker、本番はAWS EC2 + RDS + S3。
INF-POL-003Secret外部化DB Password・AWSキー・Reverbシークレットは環境変数で管理し、コードにもGitにも含めない。
INF-POL-004最小権限S3はLaravel用IAMロールのみ読み書き可。Publicアクセスは全ブロック（Project 02と異なり読み取りも公開しない — BR-FIL-002）。
INF-POL-005FE/BEビルド分離SPAのビルド（Node）とAPIの実行（PHP）を分離し、本番EC2にNodeランタイムを置かない（ビルドはCIで実施）。

■ 7. Docker設計
7.1 Container構成

nginx共通SPA静的配信 + /apiリバースプロキシ + /app WebSocketプロキシ

7.2 Container起動方針

11_基本設計書.md 3.3節の表を正とする（Web SG: 80/443公開、DB SG: EC2からの3306のみ、S3はIAMポリシー制御）。Reverbの8080はSGに公開ルールを作らない（Nginx経由のみ）。

■ 14. 環境変数・Secret管理
14.1 バックエンド（実行時に参照）
変数内容
DB_HOST / DB_DATABASE / DB_USERNAME / DB_PASSWORDRDS接続情報
AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_BUCKETS3接続情報（本番はIAMロール推奨、キー直書きは開発時のみ）
REVERB_APP_ID / REVERB_APP_KEY / REVERB_APP_SECRETReverb認証（bonus。SECRETは絶対にコミットしない）
SANCTUM関連 / CORS_ALLOWED_ORIGINSSPA配信オリジンのみ許可（NFR-013、14_セキュリティ設計.md 5章）
14.2 フロントエンド（ビルド時に焼き込み — SPA特有の注意）
変数内容
VITE_API_BASE_URLAPIベースURL
VITE_REVERB_APP_KEY / VITE_REVERB_HOSTEcho接続情報（bonus）

注意: VITE_変数はビルド時にJSへ焼き込まれ、配布物として全ユーザーに見える。したがって①シークレットは絶対にVITE_に入れない（REVERB_APP_KEYは公開前提のキー、SECRETはバックエンド側のみ）、②環境ごとに再ビルドが必要（ビルド成果物の環境間流用は不可）。この2点はProject 01/02（サーバーサイドのみ）には存在しなかったSPA特有の制約である。

■ 18. 可用性・復旧設計

学習用途のためSLA保証なし・単一AZ（NFR-004〜006）。RTO 24時間・RPO 1日（RDS日次バックアップ）。Reverb障害時はポーリング縮退で業務継続（NFR-007）— 復旧手順は20_運用保守手順書.md。

■ 21. まとめ

Project 01/02のインフラ資産（Docker Compose・VPC・Scheduler）を最大限流用しつつ、SPA特有の3点 — ①Nginxのtry_filesフォールバック（ディープリンク対応）、②FE/BEビルド分離（本番にNodeを置かない）、③VITE_変数のビルド時焼き込みとシークレット禁止 — を本書で確定した。WebSocket（Reverb）はネットワーク的にもbonusとして隔離されており（8080非公開・Nginx経由のみ）、失敗してもコア構成に影響しない。
