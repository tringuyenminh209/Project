■ 1. 本書の目的

実際のブラウザ操作に近い形で、会員登録〜プロジェクト運用〜タスク完了までの一連の業務フローが、SPA・API・DB・S3・（bonus）Reverbを含めたシステム全体として期待通り動作することを確認する。15/16章で分解確認した要素を、実際のチーム運用の時系列に沿って結合する最終段階である。

■ 2. 試験方針
方針ID方針内容
ST-POL-001E2E実際の画面操作の連鎖で確認する（APIを直接叩かず、SPA経由）。
ST-POL-002Role Based未認証 / Member / Owner / Adminそれぞれの視点でシナリオを作成する。
ST-POL-003RecoveryReverb障害・ファイルアップロード中断等の異常系からの復旧動線を確認する。
ST-POL-004Multi-Actorチーム運用の本質（複数人の同時参加）を反映し、可能な限り2ブラウザ（Owner/Member）で並行実施する。
■ 3. 試験環境

Docker Composeのフルスタック環境（Local）またはAWS Development環境。13_インフラ設計.mdの構成（nginx/app/db/scheduler/vite build/reverb）を使用し、16_結合試験仕様書.mdと同一の環境を流用する。

■ 4. E2Eシナリオ試験
ST-SCN-001 チーム立ち上げ〜タスク完了までの一気通貫
1. 未登録ユーザーとしてSCR-002で会員登録する
2. 自動ログイン状態でSCR-003（プロジェクト一覧）へ遷移する
3. 「新規プロジェクト作成」からプロジェクトを作成する → 自動的にOwnerになる
4. プロジェクト設定（SCR-008）からメンバーをメールアドレスで招待する（既存の別ユーザー）
5. カンバン（SCR-004）で新規タスクを作成し、招待したメンバーを担当に設定する
6. タスク詳細（SCR-005）でコメントを投稿する
7. タスクにファイル（5MB PDF）を添付する
8. カンバンでタスクをtodo→in_progress→doneへドラッグ&ドロップで移動する
9. 招待されたメンバーで再ログインし、通知一覧（SCR-009）で「担当割当」「コメント」の通知を確認し既読化する
期待結果: 全ステップが例外なく完了する。作成したプロジェクト・タスク・コメント・ファイル・通知がそれぞれの画面に正しく反映される
ST-SCN-002 Ownerによるメンバー・プロジェクト運用
1. ST-SCN-001のOwnerでログインする
2. メンバーのロールをmember→ownerへ変更する（複数Owner体制にする）
3. 一方のOwnerが自主脱退する
4. 残るOwnerがプロジェクトをアーカイブする → カンバンがArchived-banner表示になり書込操作が無効化される
5. プロジェクトを復元する → 書込操作が復活する
期待結果: ロール変更・自主脱退・アーカイブ・復元が画面上で正しく反映され、アーカイブ中の書込UIが無効化される
ST-SCN-003 期限接近通知〜リアルタイム受信（bonus含む）
1. Ownerがタスクを作成し、期限を「明日」に設定してMemberを担当に設定する
2. バッチ（SendDueSoonNotifications）の実行を待つ、または手動実行する
3. Memberでログインし、通知一覧に task_due_soon が現れることを確認する
4. （bonus）Owner・Member双方をブラウザで開いた状態でOwnerがタスクをカンバン上で移動し、Member側の画面がリロードなしで同期することを確認する
期待結果: 期限接近通知が作成され、（bonus環境が正常な場合）カンバンの変更がリアルタイムで反映される
ST-SCN-004 非メンバーによるアクセス試行（IDOR実地確認）
1. TD-PRJ-003のOwner（TD-PRJ-001の非メンバー）でログインする
2. ブラウザのアドレスバーに直接 `/projects/{TD-PRJ-001のID}` を入力する
期待結果: プロジェクトの存在を示す情報が一切表示されず、「見つかりません」等の画面（G-07相当）またはボード一覧へ誘導される。ネットワークタブでAPI応答がE007 (404) であることを確認する
■ 5. ロール別シナリオ試験

■ 10. まとめ

本書は15_単体試験仕様書.md（ロジック網羅）・16_結合試験仕様書.md（実物境界の確認）に続く最終段階として、実際のチーム運用シナリオでシステム全体を通しで確認する。特にST-SCN-004（非メンバーのIDOR実地確認）は14_セキュリティ設計.mdのIDOR対策がユーザー視点でも機能することの最終証跡であり、ST-REC-002（Reverb障害時の継続）はbonus機能の隔離設計（00_開発計画書.mdのリスク対策）が実際に効くことの証跡である。
