■ 1. 本書の目的

本書はProject Management SystemをAWS本番環境（EC2 + RDS + S3）へリリースする手順を定義する。Project 02（EC）と異なり、SPA（Vue 3）とAPI（Laravel）のビルド・デプロイが2系統に分かれる点、および外部決済連携（Stripe）を持たない代わりにpresigned URLの設定確認が必要な点が本プロジェクト固有の手順である。

■ 2. リリース方針
方針ID方針内容
REL-POL-001試験Pass前提15/16/17_試験仕様書.mdが全てPassし、18_UAT.mdがサインオフ済みであることをリリースの前提とする。
REL-POL-002FE/BE分離ビルドフロントエンドはCI上でビルドし、本番EC2にはビルド成果物（静的ファイル）のみを配置する（Nodeランタイムを本番に置かない — 13_インフラ設計.md 7.2節 INF-POL-005）。
REL-POL-003環境ごとの再ビルド必須VITE_*環境変数はビルド時に焼き込まれるため、環境（Local/Production）を跨いだビルド成果物の使い回しを禁止する（13_インフラ設計.md 14.2節）。
REL-POL-004bonus隔離Reverbの起動失敗はコア機能のリリース可否に影響させない（00_開発計画書.mdのリスク対策方針）。

■ 5. リリース手順
1. mainブランチの最新コミットを確認する
2. GitHub Actionsのfrontendジョブでビルド成果物（frontend/dist）を生成する（REL-POL-002）
3. Production EC2にSSH接続する
4. git pull で最新コード（backend）を取得する
5. ビルド成果物（dist/）をEC2の配信ディレクトリへ配置する（7章参照）
6. docker compose build（backend/scheduler/reverbイメージに変更がある場合）
7. docker compose up -d
8. php artisan migrate --force（6章参照）
9. php artisan config:cache / route:cache
10. Reverbコンテナの起動確認（8章参照、bonus）
11. 9章のSmoke Testを実施する
■ 6. Migration手順
1. RDSの手動スナップショットを取得する（念のためのバックアップ）
2. Development環境で同一Migrationが成功することを事前確認済みであることを再確認する
3. php artisan migrate --force を実行する
4. php artisan migrate:status で全Migrationが実行済みであることを確認する

7テーブルすべてに外部キー制約があるため、Migration順序は09_テーブル定義.md 3章のテーブル一覧順（依存関係の浅い順: users→projects→project_members→tasks→task_comments→task_files→notifications）を厳守する。

■ 7. フロントエンドビルド・配置手順

Project 02（Blade、サーバーサイドレンダリング）には存在しない、本プロジェクト固有の手順である。

1. CI（frontendジョブ）で本番用環境変数（.env.production）を用いて vite build を実行する
   → frontend/dist/ に静的ファイル一式が生成される
2. dist/ をEC2の配信ディレクトリ（nginxコンテナがマウントするパス）へ配置する
3. Nginx設定（13_インフラ設計 8章）に try_files $uri /index.html が有効であることを確認する
   → 無効な場合、SPAのディープリンク（例: /projects/1/tasks/5 への直接アクセス）が404になる
4. ブラウザのキャッシュ影響を避けるため、ビルド成果物のファイル名にハッシュが含まれていること（Viteのデフォルト挙動）を確認する

注意（REL-POL-003の実務化）: Local開発用に一度ビルドしたdist/を誤って本番へ配置しないこと。VITE_API_BASE_URLがLocalの値のまま焼き込まれた成果物を配置すると、SPAは本番APIではなくLocalへ接続しようとし、画面が真っ白または全API失敗になる（Smoke Testの9章で検知可能だが、事前確認を優先する）。

■ 8. Reverb起動手順（bonus）
1. docker compose up -d でreverbコンテナが起動していることを確認する（docker compose ps）
2. Nginx経由の wss:// 接続がハンドシェイクに成功することを確認する（ブラウザのDevToolsまたはwscat）
3. 失敗した場合: REL-POL-004に従いコアリリースは続行する。SPAはポーリング（30秒間隔）へ自動縮退することを確認し（BR-NTF-006）、
   Reverbの復旧は別途 `20_運用保守手順書.md` の手順で対応する
■ 9. リリース後確認（Smoke Test）
□ /api/health が200を返す（DB接続含む）
□ SPAのトップ（/login）が表示される
□ テストユーザーでログインできる（トークンが発行され、以後のAPIが認証される）
□ プロジェクト一覧・カンバンが表示される
□ タスク作成→カンバン移動→コメント投稿が成功する
□ ファイルアップロード→ダウンロード（presigned 302）が成功する
□ 通知一覧が表示される
□ Admin管理画面（ユーザー一覧・全プロジェクト一覧）にログインできる
□ （bonus）Reverb接続が確立する、または失敗時はポーリングへ縮退する
□ ディープリンク（/projects/{id}/tasks/{id}）への直接アクセスが正しく表示される（try_filesの実地確認）
■ 10. ロールバック手順
1. docker compose down
2. 直前の安定版タグへ git checkout（backend）
3. 直前の安定版のビルド成果物（dist/）へ配信ディレクトリを戻す
4. docker compose up -d
5. Migrationをロールバックしたテーブルがある場合、RDSスナップショットから復元を検討する
   （タスク・コメント・ファイルの履歴データを含むため、Migrationのロールバックは原則
    as new columnの追加のみに限定し、破壊的変更は避ける方針とする — Project 02と同一方針）
6. S3のオブジェクトはロールバック対象外とする（ロールバック後もファイル実体は保持され、
   孤児化した場合は 20_運用保守手順書.md の棚卸で対応する）

■ 13. まとめ

本書のリリース手順で最も注意すべきはREL-POL-002/003（FE/BE分離ビルドと環境ごとの再ビルド必須）であり、Blade構成のProject 02にはなかった「ビルド成果物の環境取り違え」という新しい失敗モードに対応する。Reverb（bonus）はREL-POL-004により切り離されており、単独の起動失敗がリリース全体をブロックしない設計になっている。
