■ 1. 本書の目的

本書は08_ER図の論理設計を、MySQL 8上で実行可能な物理テーブル定義に落とし込む。本書の型・制約定義がそのままLaravel Migrationの実装根拠となる。

■ 2. テーブル設計方針
方針ID方針内容
TBL-POL-001全テーブルにタイムスタンプcreated_at（+更新がありうるテーブルはupdated_at）を必須とする。例外3件: task_comments（編集機能なしで不変、BR-CMT-003）とtask_files（アップロード後不変）はcreated_atのみ。notificationsは状態変化（既読化）をread_atが記録するためupdated_atを持たない。
TBL-POL-002外部キーは物理制約で強制Laravel側のリレーション定義だけでなく、DBのFOREIGN KEY制約でも整合性を強制する（Project 01/02と同一方針）。
TBL-POL-003無効化と物理削除の使い分けusers（inactive化）・projects（archived化）は物理削除しない。タスク系（tasks/task_comments/task_files）はタスク削除時に連動物理削除する（BR-TSK-007。Project 02の「会計証憑は永続保持」との対比を意図した設計）。
TBL-POL-004外部キー命名の例外原則[参照先単数形]_idだが、役割名を優先する3例外（assignee_id / created_by / uploaded_by、いずれもusers参照）を許容する。詳細は10章。
TBL-POL-005DB制約とアプリ層の役割分担の明示UNIQUE・CHECK・FKで守れるものはDBで守る。DBで表現できない2件（positionの一意性、task_due_soonの重複防止）は理由と代替保証をセットで11章に記録する。

■ 8. CHECK制約
テーブル制約目的
task_filessize_bytes <= 1048576010MB制限（BR-FIL-001）の最終防衛線。アプリ検証をすり抜けた場合もDBが拒否する（Project 02の在庫CHECKと同じ「二重防御」哲学）
notifications(is_read = FALSE AND read_at IS NULL) OR (is_read = TRUE AND read_at IS NOT NULL)「既読なのに日時がない/未読なのに日時がある」矛盾状態の物理的防止

ステータス系の許容値はENUM型自体が制約として機能するため、Project 02のような別途CHECKは設けない（orders.statusはBR上の遷移制限があったためCHECKを重ねたが、本システムのステータスは全遷移自由でありENUMで十分）。

■ 11. 設計判断（設計判断メモ）

00_開発計画書 9章の設計論点への最終回答を記録する。

  1. positionの採番方式 — 隙間法（gap method）を採用し、UNIQUEは張らない:
  • 採番: 新規タスクは列末尾に最大position + 1024で追加。D&Dによる挿入は前後タスクのpositionの中間値（floor((prev + next) / 2)）を採る。中間値が作れなくなった（前後差が1になった）時点で、その列（project_id × status）全体を1024刻みで振り直す（リナンバリング）。
  • UNIQUE(project_id, status, position)を張らない理由: リナンバリングや隣接タスクの入れ替えでは更新途中に一時的な重複が発生し得るが、MySQLのUNIQUE制約は文の実行ごとに即時検査されるため（遅延制約非対応）、正当な並び替え操作自体が失敗してしまう。
  • 代替保証: 並び替えはService層で列単位にSELECT ... FOR UPDATE（対象列のタスク行をロック）した上で実行し、同時ドラッグを直列化する（Project 02の在庫ロックで習得した悲観的ロックの再適用）。万一positionが重複しても、表示はORDER BY position, idで決定的であり業務影響は「見た目の順序が1回ズレる」に留まる — 在庫のような金銭的損害はないため、この保証レベルで十分と判断した。
  1. task_due_soonの重複防止 — アプリ層の存在チェックで保証: 「1タスク1回」（BR-NTF-003）をUNIQUE(task_id, type)で表現すると、task_assigned/task_commented（同一タスクに複数回発生し得る）まで巻き込まれる。type条件付きの部分UNIQUEはMySQLでは表現できない（Project 02の「1会員1activeカート」と同じ制約事情）。期限バッチは毎時単一実行（withoutOverlapping）で並行性がないため、INSERT前の存在チェック（INDEX(task_id, type)使用）で十分と判断した。
  2. 通知の生存戦略（物理層）: task_id SET NULL + task_titleスナップショット（08_ER図 REL-011/10章の物理実装）。遷移時にtask_idがNULLなら、フロントはE007相当の「タスクが見つかりません」を表示する（G-07）。
  3. task_files.size_bytesのCHECK制約: FormRequest（アプリ層）の検証が正だが、直接SQLやバグをすり抜けた場合の最終防衛線としてDB側でも拒否する。10485760 = 10 × 1024 × 1024。
  4. projectsに集計列・作成者列を持たない: 08_ER図 10章の正規化判断を物理層でも維持する。未完了タスク数（SCR-003）はCOUNT導出、Ownerはproject_membersから導出。
■ 12. トレーサビリティ

08_ER図のリレーション定義（REL-ID）→ 本書のテーブル定義（TBL-ID）→ Laravel Migrationの物理実装、の順に一意に追跡できる。業務ルール（BR-ID）は各テーブル・カラムの「説明」列に注記した。

■ 13. まとめ

7テーブルの物理設計を確定した。本書の核心は11章の2つの設計判断 — ①position隙間法（UNIQUEを張らない理由と代替保証のセット）、②task_due_soon重複防止のアプリ層委譲 — であり、いずれも「DB制約で守れないものは、守れない理由と代替保証を明文化する」（TBL-POL-005）の実践である。実装時はproject_membersのUNIQUE複合キーとnotificationsのCHECK制約を必ずMigrationに含め、15_単体試験仕様書でDB制約レベルの検証（直接INSERT試験）を行うこと。
