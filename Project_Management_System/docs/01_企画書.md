# 1. 企画背景

Portfolio Roadmap（SE/BrSE 2026-2027）における10プロジェクト構想のうち、Project 03 として位置づけられる案件。Project 01（HR & Attendance System）でCRUD・認証・グローバルRBACを、Project 02（EC Site）でトランザクション・決済・在庫整合性を習得した。次の段階として、実務のB2B SaaS開発で最も普遍的なドメインである「チームコラボレーションツール」に取り組み、**複数ユーザーが同じリソースを共有する**システム特有の設計課題（リソース単位の認可、通知、リアルタイム性）を経験する。

日本のIT企業（特にBrSEとして働く場合）では、Backlog・Redmine・Jira等のプロジェクト管理ツールが開発現場の共通言語であり、その仕組みを「使う側」から「作る側」の視点で理解していることは、オフショア開発のブリッジ業務（タスク分解・進捗管理・仕様調整）において直接的な強みになる。

---

# 2. 現状の課題

- Project 01/02のフロントエンドはいずれもBlade（サーバーサイドレンダリング）であり、実務案件で主流のSPA + API分離構成（CORS・トークン認証・ビルド分離）を経験していない
- Project 01/02の認可は「ユーザーのグローバルロール」のみで判定できた。**「同じユーザーがリソースAでは管理者、リソースBでは一般参加者」という文脈依存の認可**（プロジェクト単位のロール）を設計・実装した経験がない
- 「あるユーザーの操作が別のユーザーへの通知を生む」という**ユーザー間相互作用**（fan-out・既読管理・リアルタイム配信）を扱った経験がない
- ファイルという非構造化データの保存・検証・アクセス制御（S3上のオブジェクトを権限判定つきで取得させる）を本格的に扱った経験がない（Project 02の商品画像は公開読み取りのみ）
- 個人開発が続いており、featureブランチ・Pull Request・レビューというチーム開発の作法が習慣化していない

---

# 3. 企画目的

上記の課題を解消するため、以下を満たすプロジェクト管理システムを自主開発する。

1. ユーザーがプロジェクトを作成し、他のユーザーを招待してチームでタスクを管理できる
2. プロジェクトのメンバーでない者は、そのプロジェクトのタスク・コメント・ファイルに一切アクセスできない（2層ロールモデルによる認可の徹底）
3. タスクの割当・コメント・期限接近が、関係するユーザーへの通知として届き、既読管理できる
4. カンバンボードでタスクの状態と順序を直感的に操作できる
5. タスクにファイルを添付でき、メンバーのみがダウンロードできる

---

# 4. 対象ユーザー

Admin（グローバル管理者）、Owner（プロジェクト内管理者）、Member（プロジェクト参加者）の2層3種別。詳細は 00_開発計画書 3章（2層ロールモデル）、権限マトリクスは 02_要件定義書 で確定する。

# 5. ステークホルダー

| Stakeholder | 責任 | 期待 | リスク | コミュニケーション方法 |
|---|---|---|---|---|
| Sponsor（自分自身＝学習投資者） | 学習方針決定、時間配分 | Mid-levelスキル（SPA・認可・通知）の獲得 | 学習範囲の逸脱（Reverb深追い等） | 週次の進捗振り返り（日報） |
| Project Manager（自分） | 進捗管理、スコープ管理 | Roadmap通りの進行 | スケジュール遅延 | 日報1行 + PRベースの進捗可視化 |
| Business User（想定：開発チームリーダー） | 業務要件の正確性確認 | 現実のタスク管理業務に沿った仕様 | 要件の抜け漏れ（権限の例外ケース） | 本ドキュメント一式によるセルフレビュー |
| System User（想定：チームメンバー） | サービス利用 | 迷わないカンバン操作・煩わしくない通知 | 通知過多による無視・UX離脱 | 画面設計・通知条件の絞り込み（3種限定） |
| Operation（自分） | 本番運用（想定） | 障害の少ない安定稼働 | WebSocket接続障害・S3コスト超過 | ログ設計・監視設計・Runbook |

# 6. 解決方針

| 課題 | 解決方針 |
|---|---|
| 文脈依存の認可（プロジェクト単位ロール） | `project_members`（`UNIQUE(project_id, user_id)` + `role`列）で2層目のロールをDB表現し、判定をLaravel Policyに一元化。Controllerに権限分岐を散在させない |
| プロジェクト横断のIDOR | 全リソース（タスク・コメント・ファイル）の取得を必ず所属プロジェクトのメンバーシップ経由でスコープする。権限マトリクス×全APIを試験仕様書で網羅 |
| 通知の設計 | 通知種別を3種（`task_assigned` / `task_commented` / `task_due_soon`）に限定し、「発火条件・通知先・除外条件（本人除外）」を要件定義書で仕様化してから実装する |
| リアルタイム配信のリスク | Laravel Reverbはbonusとして隔離し、失敗時はポーリングにフォールバック。コア機能（DB通知）はWebSocketなしで完結させる |
| SPA分離の複雑さ | Sanctumトークン認証 + fetchベース自作クライアント。認証方式のPoCを基盤構築フェーズ冒頭で実施（14_セキュリティ設計で確定） |
| ファイルのアクセス制御 | S3バケットはPrivateとし、権限判定を通したダウンロード経路のみ許可（方式は12_詳細設計書/14_セキュリティ設計で確定） |

# 7. システム概要

チーム向けのプロジェクト・タスク管理システム（Kanban方式）。ユーザーはプロジェクトを作成してメンバーを招待し、タスクの作成・割当・進捗管理・コメント・ファイル共有を行う。関係する操作はアプリ内通知として届く。詳細構成は 00_開発計画書 4章の構成図を参照。

---

# 8. 主要機能

認証・プロジェクト管理・メンバー管理・タスク管理・カンバン・コメント・ファイル添付・通知の8系統。詳細は 00_開発計画書 5章および 07_機能一覧（作成予定）を参照。

# 9. 開発範囲

## 9.1 開発対象

- 会員登録・ログイン・ログアウト（Sanctum、SPA向けトークン認証）
- プロジェクト管理（作成・編集・アーカイブ）
- メンバー管理（招待・ロール変更・除名、最後のOwner保護）
- タスク管理（CRUD、担当者、期限、優先度、ステータス）
- カンバンボード（D&Dによるステータス変更・列内並び替え）
- コメント（投稿・削除、担当者への通知発火）
- ファイル添付（S3保存、検証、権限付き取得）
- アプリ内通知（3種、既読管理）+ 期限接近バッチ
- （bonus）Laravel Reverbによるリアルタイム配信

## 9.2 開発対象外

- ガントチャート・工数管理・レポート集計
- 複数ワークスペース／組織（テナント）機能
- メール通知・Slack等の外部サービス連携
- コメントの@メンション解析（将来拡張）
- 全文検索（タスク検索はタイトル部分一致のみ）
- ゲスト共有リンク（未ログイン閲覧）
- モバイルアプリ・多言語UI（日本語のみ、内部文書はベトナム語併記）

00_開発計画書 2.1/2.2節と同一。

# 11. KPI

学習プロジェクトのため売上KPIではなく、以下を達成条件とする。

| KPI | 目標値 |
|---|---|
| 権限マトリクスの試験網羅 | 全ロール（Admin/Owner/Member/非メンバー）× 主要操作の組合せを単体試験で100%網羅し、全件Pass |
| IDOR防御 | 非メンバーによるプロジェクト資源への全アクセス経路（タスク・コメント・ファイル・通知）が拒否されること |
| 通知の正確性 | 3種の通知が「発火すべき時に発火し、発火すべきでない時（本人操作等）に発火しない」ことをテストで保証 |
| フロントエンド品質 | TypeScript strictモードでビルドエラー0、主要画面のLoading/Empty/Error 3状態実装 |
| チーム開発プラクティス | 実装期間中のmain直接push 0件（全変更がPR経由） |
| デプロイ | AWS EC2 + RDS + S3 構成での稼働確認（SPA + API分離デプロイ） |

| 項目 | 概算（月額） |

# 14. 期待効果

- SPA + API分離という実務標準構成を、認証設計（CORS・トークン）込みで独力構築できる力の獲得
- 「リソース単位の認可」という、B2B SaaS開発のほぼ全案件に登場する設計パターンの言語化・実装能力の獲得
- 通知・リアルタイムという「ユーザー間相互作用」の設計経験 — Project 04（災害安否確認の通知）・Project 10（モバイルpush通知）への直接の布石
- プロジェクト管理ツールを「作る側」から理解することによる、BrSEとしてのブリッジ業務（タスク分解・進捗管理）の解像度向上

# 16. まとめ

本企画は、Project 01/02で完成させた「Laravelレイヤー構成 + 業務ルール駆動の設計プロセス」を、初のSPA分離構成と2層ロールモデルへ拡張する挑戦である。核心は「同じユーザーでも文脈（プロジェクト）によって権限が変わる」認可設計であり、この判定を漏れなく仕様化（権限マトリクス）→一元実装（Policy）→網羅試験（KPI）まで一気通貫させることを、本プロジェクト最大の成果物とする。
