# 1. 本書の目的

本書は、Project Management Systemが支える業務の流れを、アクター（スイムレーン）ごとの時系列で定義する。02_要件定義書の業務ルール（BR-ID）と03_ユースケースのUC-IDを業務の文脈に配置し、05_画面遷移図以降の設計の基準とする。

本システムの業務フローの特徴は、**1人の操作が他のメンバーへの通知として波及する**（fan-out）ことである。各フローでは「誰の操作が」「誰に何を届けるか」を明示する。

# 3. AS-IS 業務フロー（想定：システム化以前）

```mermaid
flowchart TD
    A["リーダーが朝会でタスクを口頭・チャットで割り振る"] --> B["各メンバーがExcelの進捗表を各自更新"]
    B --> C{"進捗を知りたい?"}
    C -->|Yes| D["リーダーがチャットで個別に確認・催促"]
    D --> E["メンバーが作業を中断して状況報告"]
    C -->|No| F["Excelが更新されず実態と乖離"]
    F --> G["期限切れタスクに誰も気づかない"]
    E --> H["関連ファイルはチャットに流れて行方不明"]
```

## 3.1 AS-IS 課題

- タスクの割当・進捗がチャットとExcelに分散し、「今の状態」の正が存在しない
- 進捗確認が人力（催促）で行われ、確認する側もされる側も時間を奪われる
- 期限切れに気づく仕組みがなく、遅延の発見が事後になる
- ファイルがチャットのタイムラインに埋もれ、後から探せない
- 誰がどのタスクの情報を見てよいかの管理がなく、プロジェクト外への情報漏れに気づけない

# 4. TO-BE 業務フロー（本システム導入後）

```mermaid
flowchart TD
    A["Ownerがプロジェクトを作成しメンバーを招待"] --> B["タスクを作成し担当者を割当（自動で通知）"]
    B --> C["担当者がカンバンで着手を宣言（todo→in_progress）"]
    C --> D["進捗はカンバンボードで常時可視化（催促不要）"]
    D --> E["コメント・ファイルはタスクに紐づけて集約"]
    E --> F["期限24時間前にSystemが自動通知"]
    F --> G["完了したらdoneへ移動、チーム全員が即座に把握"]
```

## 4.1 TO-BE 改善点

| 項目 | AS-IS | TO-BE |
| --- | --- | --- |
| タスクの正 | チャットとExcelに分散 | `tasks`テーブルに一元化、カンバンで常時可視化 |
| 割当の伝達 | 口頭・チャット（見落とし発生） | `task_assigned`通知が自動で届く（BR-NTF-001） |
| 進捗確認 | リーダーが個別に催促 | カンバンの列（BR-TSK-001）を見るだけ。催促という業務自体を消す |
| 期限管理 | 人間の記憶頼み | Systemが毎時監視し期限24時間前に自動通知（BR-NTF-003） |
| ファイル | チャットに流れて散逸 | タスクに添付しS3へ集約（BR-FIL-004）、メンバーのみ取得可（BR-FIL-002） |
| 情報の境界 | 管理なし | メンバーシップによるアクセス制御（BR-PRM-002/006） |

# 5. プロジェクト立ち上げ業務フロー（BF-002）

```mermaid
flowchart TD
    subgraph 発起人["発起人（作成後はOwner）"]
        A["プロジェクトを作成（名称・説明）"]
        C["メンバー管理画面でメールアドレスを指定し招待"]
        E["必要に応じて共同Ownerへ昇格させる"]
    end
    subgraph System
        B["projects作成 + 発起人をowner登録（同一トランザクション、BR-PRJ-001）"]
        D["project_membersへrole=memberで追加（即時参加、BR-PRJ-004）"]
        F["ロール変更を保存 + 操作ログ記録"]
    end
    subgraph 招待されたユーザー["招待されたユーザー"]
        G["プロジェクト一覧に新プロジェクトが表示される"]
        H["カンバンボードへ参加"]
    end

    A --> B --> C --> D --> G --> H
    C --> E --> F
```

**業務上のポイント**: 招待は承諾フローなしの即時参加（BR-PRJ-004）のため、誤招待は除名（BF-007）で取り消す運用となる。招待相手は事前にシステム登録が必要（未登録メールはE007）。

# 6. タスク運用業務フロー（BF-003）

タスクの誕生から完了までの基本サイクル。**通知のfan-out（誰の操作が誰に届くか）**が本フローの中心である。

```mermaid
flowchart TD
    subgraph 起票者["起票者（Owner / Member）"]
        A["タスクを作成（タイトル・期限・優先度）"]
        B["担当者にメンバーBを指定"]
    end
    subgraph System
        C["tasksをstatus=todoで作成"]
        D{"担当者は起票者本人?"}
        E["task_assigned通知を作成（BR-NTF-001）"]
        F["（bonus）Reverbでリアルタイム配信（UC-021）"]
    end
    subgraph 担当者["担当者（メンバーB）"]
        G["通知を受信・確認（UC-015）"]
        H["タスク詳細を確認し、カンバンでin_progressへ移動"]
        I["作業完了後、doneへ移動"]
    end

    A --> B --> C --> D
    D -->|"No（他人に割当）"| E --> F --> G --> H --> I
    D -->|"Yes（自分に割当）"| H
```

**業務上のポイント**:
- 通知は「他人への割当」でのみ発火する（本人除外の原則）。自己割当で自分に通知が来る煩わしさを設計段階で排除している
- 担当者の変更（UC-009）も同じ経路で新担当者へ`task_assigned`を発火する
- in_progress→doneの移動に承認は不要（BR-TSK-001の自由遷移）。品質ゲートが必要なチームはdone列への移動をレビュー完了の合図とする運用でカバーする

# 7. カンバン操作フロー（BF-004）

```mermaid
flowchart TD
    subgraph メンバーA["メンバーA"]
        A["タスクカードをドラッグ"]
        B["別列（または同列内の別位置）へドロップ"]
        C["画面は楽観的に即時反映"]
    end
    subgraph System
        D["status / positionを更新（BR-TSK-002）"]
        E{"更新成功?"}
        F["（bonus）他メンバーへboardイベントを配信"]
        G["エラー応答（E006/E007等）"]
    end
    subgraph メンバーB["メンバーB（同じボードを閲覧中）"]
        H["（bonus）ボードが自動更新される"]
        I["（Reverbなし）次回取得時に最新化"]
    end

    A --> B --> C --> D --> E
    E -->|Yes| F --> H
    E -->|No| G -->|"カードを元の位置に戻す"| A
    F -.-> I
```

**業務上のポイント**: 同じ列を複数メンバーが同時に並び替える競合は日常的に起きる。整合の最終責任はDB側のposition設計（09_テーブル定義で確定）が持ち、クライアントは失敗時に巻き戻して再取得する、が原則。

# 8. コラボレーション業務フロー（BF-005）

```mermaid
flowchart TD
    subgraph メンバーA["メンバーA（コメント・ファイル投稿者）"]
        A["タスク詳細でコメントを投稿"]
        B["関連ファイルを添付（10MB以下・許可種別）"]
    end
    subgraph System
        C["task_comments作成"]
        D{"担当者がいる AND 投稿者本人ではない?"}
        E["task_commented通知を作成（BR-NTF-002）"]
        F["ファイルを検証しS3へ保存（BR-FIL-001/004）"]
    end
    subgraph 担当者["担当者（メンバーB）"]
        G["通知からタスクへ遷移"]
        H["コメントを確認し、ファイルをダウンロード（メンバーのみ、BR-FIL-002）"]
    end

    A --> C --> D
    D -->|Yes| E --> G --> H
    D -->|No| Z["通知なし（本人除外/担当者なし）"]
    B --> F --> H
```

**業務上のポイント**: 議論と成果物がタスクに紐づくため、後から参加したメンバーも文脈を追える（AS-IS課題「ファイルの散逸」の解消）。ダウンロードは毎回メンバーシップ判定を通る — チャットのように「URLを知っていれば誰でも見られる」状態を作らない。

# 9. 期限接近通知バッチフロー（BF-006）

```mermaid
flowchart TD
    subgraph System["System（毎時バッチ、UC-019）"]
        A["期限まで24時間以内 AND status != done AND 担当者あり のタスクを抽出"]
        B{"このタスクは通知済み?（BR-NTF-003）"}
        C["task_due_soon通知を作成"]
        D["スキップ"]
        E["失敗タスクは飛ばして継続（1件の失敗を波及させない）"]
    end
    subgraph 担当者["担当者"]
        F["通知を受信し、タスクの完了 or 期限の見直しを判断"]
    end

    A --> B
    B -->|No| C --> F
    B -->|Yes| D
    C -.-> E
```

**業務上のポイント**: 通知は1タスク1回のみ（毎時再送しない — 通知過多はAS-ISの「催促」の再来になる）。期限を過ぎたタスクへの追加通知も行わず、ボード上の強調表示（06_画面設計）に委ねる。

# 10. 運用業務フロー（BF-007: 除名・アーカイブ・Admin）

## 10.1 メンバー除名フロー

```mermaid
flowchart TD
    subgraph Owner
        A["メンバー管理画面で除名を実行"]
    end
    subgraph System
        B{"対象は最後のOwner?（BR-PRJ-002）"}
        C["E006で拒否"]
        D["project_members行を削除"]
        E["当該ユーザーの担当タスクをassignee NULL化（BR-PRJ-005、同一トランザクション）"]
        F["操作ログ記録（INC-008）"]
    end
    subgraph 除名されたユーザー["除名されたユーザー"]
        G["プロジェクトが一覧から消える。以後の資源アクセスはE007（存在秘匿）"]
        H["過去のコメント・ファイルは履歴として残る（投稿者名表示は保持）"]
    end

    A --> B
    B -->|Yes| C
    B -->|No| D --> E --> F --> G
    D -.-> H
```

## 10.2 アーカイブ・Admin運用フロー

```mermaid
flowchart TD
    subgraph Owner
        A["プロジェクトをアーカイブ"]
        E["必要になったら復元"]
    end
    subgraph Admin
        B["全プロジェクト一覧で放置プロジェクトを発見"]
        C["強制アーカイブ（中身のタスク等は閲覧不可、BR-PRM-004）"]
    end
    subgraph System
        D["status=archived: 全書込をE006で拒否、閲覧のみ許可（BR-PRJ-003）"]
        F["status=activeへ戻す（復元はOwnerのみ）"]
    end

    A --> D
    B --> C --> D
    D --> E --> F
```

# 11. 例外フロー

| 例外 | 発生フロー | システムの挙動 | 業務上の対応 |
| --- | --- | --- | --- |
| 非メンバーがプロジェクト資源へアクセス | 全BF | E007（存在秘匿、BR-PRM-006） | 参加が必要ならOwnerに招待を依頼する |
| Memberが Owner専用操作を実行 | BF-002 / 007 | E002 | Ownerに操作を依頼、または昇格してもらう |
| archivedプロジェクトへの書込 | BF-003〜005 | E006（BR-PRJ-003） | Ownerが復元してから操作する |
| 最後のOwnerの降格・除名・脱退 | BF-007 | E006（BR-PRJ-002） | 先に他メンバーをOwnerへ昇格させる |
| 既参加メンバーへの再招待 | BF-002 | E011 | メンバー一覧を確認する |
| ファイル制限違反（サイズ・種別・件数） | BF-005 | E003（BR-FIL-001） | ファイルを分割・変換して再添付 |
| カンバン同時並び替えの競合 | BF-004 | DB側で整合を保証し、クライアントは再取得 | 特別な対応不要（自動回復） |
| Reverb停止 | BF-003〜005 | 配信のみ停止、DB通知は継続（BR-NTF-006） | ポーリングで代替。運用手順は20_運用保守手順書 |
| 担当者が無効化（inactive）されたユーザー | BF-003 / 006 | メンバーシップ・担当は保持（UC-017）。通知は作成されるがログイン不可のため未読のまま | Ownerが担当を付け替える |

# 14. まとめ

本業務フローの核心は、AS-ISの「人力による伝達・確認・催促」を、**通知のfan-out（BF-003/005/006）とカンバンの常時可視化（BF-004）に置き換える**ことである。同時に、メンバーシップ境界（BR-PRM）が全フローの前提として機能し、「見える人にしか見えない」ことを業務の自然な流れの中で保証する。次工程の05_画面遷移図 06_画面設計は、本書のフローが画面上で迷いなく実行できることを基準に設計する。
