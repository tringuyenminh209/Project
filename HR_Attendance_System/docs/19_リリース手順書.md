# 1. 本書の目的

本書は、HR & Attendance SystemをDevelopment環境からProduction環境（AWS EC2 + Docker + RDS）へリリースする際の手順を定義する。

インフラ設計（13）のCI/CD設計・Deploy方針を実行可能な手順に落とし込み、システム試験（17）・UAT（18）のPass後、安全かつ再現性のある方法でリリースできるようにする。

---

# 2. リリース方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| REL-POL-001 | Gate Keeping | 単体・結合・システム試験・UATがすべてPassするまでProductionへリリースしない。 |
| REL-POL-002 | Snapshot First | DBに影響するリリースは、必ず事前にRDSスナップショットを取得する。 |
| REL-POL-003 | Reproducible | リリース手順はGitのタグ・コミットIDで一意に再現できるようにする。 |
| REL-POL-004 | Zero Silent Change | `.env`や本番設定の変更は、本書またはIssueに記録してから行う。 |
| REL-POL-005 | Rollback Ready | ロールバック手順を事前に確認したうえでリリースを実施する。 |
| REL-POL-006 | Health Check Mandatory | リリース後は必ずヘルスチェック・主要機能のスモークテストを実施する。 |

# 6. リリース手順

`13_インフラ設計.md` 15章のCI/CDパイプラインに対応する具体的な実行ステップを以下に示す。

| Step | コマンド／操作 | 内容 |
| --- | --- | --- |
| 1 | `git checkout <release-tag>` | リリース対象のタグをチェックアウトする |
| 2 | GitHub Actions起動 | push/tagトリガーでCIパイプラインを起動する |
| 3 | `composer install --no-dev --optimize-autoloader` | 本番用依存パッケージをインストールする |
| 4 | `npm ci && npm run build` | Viteでフロントエンドassetをビルドする |
| 5 | `docker build` | Nginx / PHP-FPM+Laravelのイメージをビルドする |
| 6 | EC2へDeploy | SSH deployまたはpull deployでEC2上のコンテナを更新する |
| 7 | `php artisan migrate --force` | マイグレーションを実行する（7章参照） |
| 8 | `php artisan config:cache` `route:cache` `view:cache` | Laravelキャッシュを再生成する |
| 9 | `docker compose restart` | nginx / app コンテナを再起動する |
| 10 | ヘルスチェック | `/health`および主要APIが200を返すことを確認する（8章） |

# 7. Migration手順

| 項目 | 内容 |
| --- | --- |
| 事前確認 | migrationファイルに破壊的変更（DROP COLUMN、NOT NULL化等）が含まれる場合は特に注意し、影響範囲をテーブル定義書（09）と突き合わせる |
| 実行前 | RDSスナップショットを取得する（`Release前、Migration前に取得`、13_インフラ設計 11.2準拠） |
| 実行 | `php artisan migrate --force`をメンテナンスウィンドウ中に実行する |
| 実行後 | `php artisan migrate:status`で全migrationが`Ran`になっていることを確認する |
| 失敗時 | 9章のロールバック手順に従う |

# 8. リリース後確認（Smoke Test）

リリース直後に以下の主要フローを実施し、本番環境で最低限の機能が動作することを確認する。システム試験（17）の全項目を再実施するものではない。

| # | 確認項目 | 期待結果 | 関連 |
| --- | --- | --- | --- |
| 1 | `/health` 応答 | 200 OKが返る | 13_インフラ設計 9.3 |
| 2 | ログイン（API-001） | User/Manager/Adminそれぞれでログインできる | ST-FUNC-001 |
| 3 | 出勤打刻（API-004） | 打刻が成功し、DBに反映される | ST-FUNC-004 |
| 4 | 勤怠履歴取得（API-007） | 自分の履歴が表示される | ST-FUNC-007 |
| 5 | 休暇申請（API-009） | 申請が成功する | ST-FUNC-009 |
| 6 | 月次レポート（API-012） | 集計が表示される | ST-FUNC-012 |
| 7 | HTTPS | 本番URLがHTTPSでアクセスできる | NFR-010 |
| 8 | ログ出力 | Application Log / Access Logが正常に出力されている | 13_インフラ設計 14.2 |

すべてPassした場合のみ、リリース完了とみなす。1件でもFailした場合は9章に従いロールバックを検討する。

# 9. ロールバック手順

| 障害パターン | ロールバック手順 |
| --- | --- |
| Application起因（画面/API異常） | 直前のGitタグへ`git checkout`し、`docker compose restart`で再起動する（13_インフラ設計 16.2準拠） |
| Migration失敗・データ不整合 | 6章実行前に取得したRDSスナップショットから復元する。復元後、アプリケーションも直前バージョンへ戻す |
| Deploy自体の失敗（Container起動不可） | `docker compose ps` / `logs`でコンテナ状態を確認し、直前のDockerイメージで再起動する |
| ロールバック後 | 8章のSmoke Testを再実施し、正常性を確認したうえで関係者へ報告する（11章） |

ロールバック実施の判断は、リリース責任者が9章の情報をもとに決定する（判定基準は10章と対称）。

---

# 10. リリース判定基準

| 判定 | 条件 |
| --- | --- |
| Go | 5章のチェックリストがすべて完了し、8章のSmoke Testが全項目Passしている |
| No-Go | 5章のチェックリストに未完了項目がある、またはCritical Bugが未修正である |
| Rollback | リリース後にSmoke TestでFailが発生し、9章のロールバック手順を実施する必要がある |

| タイミング | 内容 | 宛先 |
| --- | --- | --- |
| リリース前日 | リリース予定日時、対象機能、想定影響（メンテナンス時間等）を通知する | 関係者（ステークホルダー） |

# 13. まとめ

本書では、HR & Attendance Systemのリリースにあたり、リリース前チェックリスト、具体的なデプロイ手順、Migration手順、リリース後のSmoke Test、ロールバック手順、リリース判定基準、連絡・報告方法を定義した。
