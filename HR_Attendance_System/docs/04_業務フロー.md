# 1. 本書の目的

本書は、HR & Attendance Systemにおける勤怠管理業務の流れを整理し、AS-ISとTO-BEの違いを明確にするものである。

本書で定義した業務フローは、画面遷移図、画面設計、API設計、テスト仕様書の入力情報として利用する。

# 3. AS-IS 業務フロー

現状は、紙またはExcelによる勤怠管理を想定する。

```mermaid
flowchart TD
    A["社員が出勤・退勤時刻を紙/Excelに記入"] --> B["社員が休暇申請を口頭/メールで連絡"]
    B --> C["管理者が内容を確認"]
    C --> D{"承認可能?"}
    D -->|Yes| E["管理者が承認結果を連絡"]
    D -->|No| F["管理者が却下理由を連絡"]
    E --> G["月末に管理者が勤怠情報を集計"]
    F --> G
    G --> H["Excelでレポート作成"]
    H --> I["入力ミス・確認漏れを手作業で修正"]
```

## 3.1 AS-IS 課題

| 課題ID | 課題 | 影響 |
| --- | --- | --- |
| ASIS-001 | 打刻情報が手入力である | 入力漏れ、記録ミスが発生しやすい |
| ASIS-002 | 休暇申請の状態が見えにくい | 承認漏れ、確認漏れが発生する |
| ASIS-003 | 月次集計が手作業である | 管理者の作業負担が大きい |
| ASIS-004 | 権限管理が明確でない | 不適切な閲覧・編集が発生する可能性がある |
| ASIS-005 | 操作履歴が残りにくい | 問題発生時の原因調査が難しい |

# 4. TO-BE 業務フロー

TO-BEでは、打刻、休暇申請、承認、レポート確認をWebシステム上で一元管理する。

```mermaid
flowchart TD
    A["User logs in"] --> B{"権限確認"}
    B -->|User| C["出勤・退勤打刻"]
    B -->|User| D["勤怠履歴確認"]
    B -->|User| E["休暇申請"]
    B -->|Manager| F["休暇申請確認"]
    F --> G{"承認判断"}
    G -->|Approve| H["Approvedに更新"]
    G -->|Reject| I["Rejectedに更新"]
    B -->|Manager/Admin| J["月次レポート閲覧"]
    J --> K["CSV出力"]
    B -->|Admin| L["社員・部署・シフト管理"]
    C --> M["操作ログ記録"]
    H --> M
    I --> M
    K --> M
    L --> M
```

## 4.1 TO-BE 改善点

| 改善ID | 改善内容 | 対応課題 |
| --- | --- | --- |
| TOBE-001 | 出退勤を画面から登録する | ASIS-001 |
| TOBE-002 | 休暇申請状態をPending / Approved / Rejectedで管理する | ASIS-002 |
| TOBE-003 | 月次レポートとCSV出力をシステム化する | ASIS-003 |
| TOBE-004 | User / Manager / Adminで権限制御する | ASIS-004 |
| TOBE-005 | 重要操作をaudit logに記録する | ASIS-005 |

# 5. 打刻業務フロー

## 5.1 出勤打刻

```mermaid
flowchart TD
    A["User opens 打刻画面"] --> B{"ログイン済み?"}
    B -->|No| E010["E010 Session Timeout"]
    B -->|Yes| C{"当日出勤打刻済み?"}
    C -->|Yes| E004["E004 Duplicate Attendance"]
    C -->|No| D["Userが入力したcheck_in_timeを登録"]
    D --> E["attendance_recordsを保存"]
    E --> F["登録完了メッセージ表示"]
```

## 5.2 退勤打刻

```mermaid
flowchart TD
    A["User opens 打刻画面"] --> B{"ログイン済み?"}
    B -->|No| E010["E010 Session Timeout"]
    B -->|Yes| C{"出勤打刻済み?"}
    C -->|No| E005["E005 Check Out Without Check In"]
    C -->|Yes| D{"退勤打刻済み?"}
    D -->|Yes| E004["E004 Duplicate Attendance"]
    D -->|No| E["Userが入力したcheck_out_timeを登録"]
    E --> F["勤務時間を自動計算"]
    F --> G["attendance_recordsを保存"]
    G --> H["登録完了メッセージ表示"]
```

## 5.3 打刻業務ルール

| ルール | 内容 |
| --- | --- |
| BR-001 | 同一勤務日に出勤打刻は原則1回のみ登録できる。 |
| BR-002 | 退勤打刻は出勤打刻後にのみ登録できる。 |
| BR-003 | 勤務時間は退勤時刻から出勤時刻を差し引いた時間から、シフトの休憩時間（break_minutes）を差し引いて算出する。 |

# 6. 休暇申請・承認業務フロー

## 6.1 休暇申請

```mermaid
flowchart TD
    A["User opens 休暇申請画面"] --> B{"ログイン済み?"}
    B -->|No| E010["E010 Session Timeout"]
    B -->|Yes| C["申請種別・対象日・理由を入力"]
    C --> D{"入力チェックOK?"}
    D -->|No| E003["E003 Validation Error"]
    D -->|Yes| E{"重複申請なし?"}
    E -->|No| F["重複申請エラー表示"]
    E -->|Yes| G["leave_requestsをPendingで登録"]
    G --> H["申請完了メッセージ表示"]
```

## 6.2 休暇承認

```mermaid
flowchart TD
    A["Manager opens 休暇承認画面"] --> B{"ログイン済み?"}
    B -->|No| E010["E010 Session Timeout"]
    B -->|Yes| C{"Manager/Admin権限あり?"}
    C -->|No| E002["E002 Permission Denied"]
    C -->|Yes| D["承認待ち申請一覧を表示"]
    D --> E["申請内容を確認"]
    E --> F{"承認する?"}
    F -->|Approve| G["状態をApprovedに更新"]
    F -->|Reject| H["状態をRejectedに更新"]
    G --> I["操作ログ記録"]
    H --> I
    I --> J["処理完了メッセージ表示"]
```

## 6.3 休暇申請状態遷移

DBに保存する状態はPending / Approved / Rejectedの3つのみとする。対象日を経過したApproved申請の状態は変更せず、画面表示時にend_dateと現在日付を比較して「済」を付加表示する。

```mermaid
stateDiagram-v2
    [*] --> Pending: User submits request
    Pending --> Approved: Manager approves
    Pending --> Rejected: Manager rejects
    Approved --> [*]
    Rejected --> [*]
```

# 8. マスタ管理業務フロー

Adminは、社員、部署、シフトを管理する。

```mermaid
flowchart TD
    A["Admin opens 管理画面"] --> B{"ログイン済み?"}
    B -->|No| E010["E010 Session Timeout"]
    B -->|Yes| C{"Admin権限あり?"}
    C -->|No| E002["E002 Permission Denied"]
    C -->|Yes| D{"管理対象を選択"}
    D -->|社員| E["社員情報を登録・編集・無効化"]
    D -->|部署| F["部署情報を登録・編集・無効化"]
    D -->|シフト| G["シフト情報を登録・編集・無効化"]
    E --> H{"入力チェックOK?"}
    F --> H
    G --> H
    H -->|No| E003["E003 Validation Error"]
    H -->|Yes| I["DB保存"]
    I --> J{"保存成功?"}
    J -->|No| E009["E009 Database Error"]
    J -->|Yes| K["操作ログ記録"]
    K --> L["完了メッセージ表示"]
```

## 8.1 マスタ管理方針

| 対象 | 方針 |
| --- | --- |
| 社員 | 退職・利用停止の場合は物理削除より無効化を優先する。 |
| 部署 | 使用中部署は削除せず、無効化を優先する。 |
| シフト | 勤怠履歴と関連するため、使用中シフトは無効化を優先する。 |

# 9. 例外フロー

| エラーID | エラー名 | 発生業務 | 処理 |
| --- | --- | --- | --- |
| E001 | Login Failed | ログイン | ログイン失敗メッセージを表示する。 |
| E002 | Permission Denied | 承認、レポート、マスタ管理 | アクセス権限がない旨を表示する。 |
| E003 | Validation Error | 休暇申請、検索、マスタ管理 | 対象項目ごとにエラーを表示する。 |
| E004 | Duplicate Attendance | 出勤・退勤打刻 | 二重打刻できない旨を表示する。 |
| E005 | Check Out Without Check In | 退勤打刻 | 先に出勤打刻が必要である旨を表示する。 |
| E006 | Leave Request Already Processed | 休暇承認 | 最新状態を再表示する。 |
| E007 | Data Not Found | 勤怠確認、レポート | データが見つからない旨を表示する。 |
| E008 | CSV Export Failed | CSV出力 | エラーメッセージを表示し、ログを記録する。 |
| E009 | Database Error | 登録・更新・検索 | システムエラーを表示し、ログを記録する。 |
| E010 | Session Timeout | 全業務画面 | ログイン画面へ遷移する。 |

# 10. 業務ルール対応

| 業務ルールID | 対応業務 | 対応内容 |
| --- | --- | --- |
| BR-001 | 出勤打刻 | 同一勤務日の二重出勤打刻を防止する。 |
| BR-002 | 退勤打刻 | 出勤打刻なしの退勤打刻を防止する。 |
| BR-003 | 退勤打刻 | 退勤登録時に、退勤時刻と出勤時刻の差からシフトの休憩時間を差し引いて勤務時間を計算する。 |
| BR-004 | 休暇申請 | Pending、Approved、Rejectedで状態管理する。 |
| BR-005 | 休暇申請 | 承認済み申請はUserが直接変更できない。 |
| BR-006 | 休暇承認 | ManagerまたはAdminのみ承認・却下できる。 |
| BR-007 | ログイン | 無効化された社員のログインを拒否する。 |
| BR-008 | CSV出力 | 権限範囲内の勤怠データのみ出力する。 |
