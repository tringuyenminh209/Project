# 1. 本書の目的

本書は、HR & Attendance Systemのリリース（19）後、Production環境を継続的に安定稼働させるための日常運用・保守手順を定義する。

インフラ設計（13）で定義した監視対象・バックアップ方針・障害パターンを、実際に「誰が」「いつ」「何を」行うかという運用可能な手順に落とし込む。

---

# 2. 運用方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| OPS-POL-001 | Prevention First | 監視・アラートにより障害を未然に検知する。 |
| OPS-POL-002 | Documented Response | 障害発生時の対応はすべて記録し、再発防止に活用する。 |
| OPS-POL-003 | Least Change | 本番環境への変更は最小限とし、事前レビューを経て実施する。 |
| OPS-POL-004 | Data Integrity First | 復旧作業ではデータ整合性をパフォーマンスより優先する。 |
| OPS-POL-005 | Business Hours Aware | 学習用・社内利用想定のため、平日日中の安定稼働を最優先とする（NFR-004準拠）。 |

# 7. 監視・アラート対応

`13_インフラ設計.md` 14章の監視対象・Alert方針に対応する一次対応手順を定義する。

| Alert | 検知内容 | 一次対応 |
| --- | --- | --- |
| CPU High | EC2/RDSのCPU使用率が80%以上継続 | 高負荷の原因（バッチ的な大量アクセス等）を確認し、必要に応じてインスタンスタイプ変更を検討する |
| Disk High | EC2 Disk使用率80%以上 | 不要ログ・一時ファイルを削除し、根本的に不足する場合はEBS拡張を計画する |
| DB Error | E009（Database Error）が短時間に複数発生 | Application Logで原因（接続断・クエリエラー等）を確認し、必要に応じてRDS状態を確認する |
| CSV Error | E008（CSV Export Failed）発生 | 対象月・データ量を確認し、再実行する。継続する場合はメモリ・タイムアウト設定を見直す |
| Application Down | `/health`応答なし | 8〜9章の障害対応手順に従う |

# 8. バックアップ・リストア手順

| 項目 | 内容 |
| --- | --- |
| RDSバックアップ | Automated Backup（保持7日目安）+ Release前・Migration前のManual Snapshot（13_インフラ設計 11章準拠） |
| Sourceバックアップ | GitHub Repositoryを正とする |
| `.env`バックアップ | Secret管理領域に別途保管し、Gitには含めない |
| リストア手順（DB） | ① 対象Snapshotを選定 → ② 新規RDSインスタンスへ復元 → ③ アプリケーションの接続先を切替 → ④ Smoke Test（19_リリース手順書 8章）を実施 |
| リストア手順（Application） | ① 直前の正常タグへ`git checkout` → ② Docker再ビルド・再起動 → ③ Smoke Test実施 |
| RTO / RPO | 24時間以内の復旧、最大1日以内のデータ損失（NFR-006 / NFR-007準拠） |

# 9. 障害対応手順（障害レベル別）

| レベル | 定義 | 対応時間目安 | 対応手順 |
| --- | --- | --- | --- |
| Sev1（Critical） | 全機能停止、データ破損、権限制御突破 | 即時対応 | ①一次報告（第一報） → ②影響範囲特定 → ③暫定対応（ロールバック等） → ④恒久対応の計画 → ⑤障害報告書作成 |
| Sev2（Major） | 一部機能停止（例：CSV出力のみ不可） | 当日中 | ①影響機能の切り分け → ②回避策の周知 → ③修正・デプロイ |
| Sev3（Minor） | 表示崩れ、軽微な不具合 | 次回定期リリースまで | ①Issue登録 → ②優先度付け → ③次回リリースに含める |

障害報告は「第一報→影響範囲→暫定対応→恒久対応」の順で記載する（`plan/07_BRSE_TRACK_WEEKLY.md`の障害報告メール型と統一）。

# 10. マスタ運用（入社・異動・退職）

| ケース | 手順 | 関連 |
| --- | --- | --- |
| 入社 | ①Adminが社員管理画面から登録（employee_id, name, email, department_id, shift_id, role_id） ②発行した初期パスワードを本人へ安全な方法で通知する ③初回ログイン後にパスワード変更を促す | API-014 / UAT-ADM-001 |
| 部署異動 | ①Adminが対象社員のdepartment_idを変更する ②異動日以降の勤怠・レポート集計が新部署に反映されることを確認する | API-015 |
| シフト変更 | ①Adminが対象社員のshift_idを変更する ②変更前の打刻データのwork_hoursは再計算されないことを運用者として認識しておく（12_詳細設計書 5.3のスナップショット方針） | API-015 |
| 退職 | ①Adminが対象社員をinactiveへ無効化する（物理削除は行わない） ②退職後もレポート・履歴参照は可能なままであることを確認する | API-016 / BR-EMP-001 |

# 11. セキュリティ運用

| # | 項目 | 内容 |
| --- | --- | --- |
| 1 | パスワードポリシー遵守確認 | 新規登録時に8〜20文字ルールが守られていることを定期確認する |
| 2 | Secret棚卸し | `.env`、DB_PASSWORD、APP_KEYがGit管理・ログに露出していないことを月次で確認する（14_セキュリティ設計 SEC-CHK-007準拠） |
| 3 | audit_logs監査 | 社員管理・承認・CSV出力の操作履歴を定期的にレビューし、不自然な操作がないか確認する |
| 4 | 脆弱性対応 | OS・PHP・Laravel・npmパッケージの既知脆弱性情報を確認し、パッチを計画的に適用する |
| 5 | アクセス制御見直し | Security GroupのSSH許可IPリストなど、インフラ側のアクセス制御を定期的に見直す |

# 15. まとめ

本書では、HR & Attendance Systemの運用保守として、日次・週次・月次の運用タスク、監視・アラート対応、バックアップ・リストア手順、障害レベル別の対応手順、マスタ運用（入社・異動・退職）、セキュリティ運用、定期メンテナンス、変更管理・エスカレーションを定義した。

本書をもって、開発計画書（00）で定めた全工程（企画→要件定義→設計→実装→試験→リリース→運用保守）の設計書一式が完了する。
