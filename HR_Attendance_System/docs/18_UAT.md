# 1. 本書の目的

本書は、HR & Attendance Systemが実際の業務で問題なく利用できるかを、開発者ではなく利用者（またはその代理）の視点で確認するUAT（User Acceptance Test）の実施内容を定義する。

システム試験仕様書（17）が要件・非機能・セキュリティを技術的に確認するのに対し、本書は**実際の業務の言葉・操作感**でシステムを通し、要件定義書（02）の受入条件（AC-001〜AC-009）と企画書（01）の成功条件（SC-001〜SC-005）を最終的に満たすことを確認する。

---

# 2. UAT方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UAT-POL-001 | Business Language | 試験手順・結果は専門用語を避け、業務担当者が読める言葉で記述する。 |
| UAT-POL-002 | Role Play | User / Manager / Adminそれぞれの立場になりきって操作する。 |
| UAT-POL-003 | End-to-End | 単機能の確認だけでなく、入社〜退職、申請〜承認のような一連の業務を通しで確認する。 |
| UAT-POL-004 | Evidence Based | 各シナリオの実施結果はスクリーンショットまたは実施ログを残す。 |
| UAT-POL-005 | Sign-off Required | すべてのMustシナリオがPassし、実施者・承認者の署名（記名）をもって完了とする。 |

# 3. UAT実施体制

| 役割 | 担当 | 責務 |
| --- | --- | --- |
| UAT実施者（User役） | Nguyen Minh Tri（ロールプレイ） | User編シナリオの実施 |
| UAT実施者（Manager役） | Nguyen Minh Tri（ロールプレイ） | Manager編シナリオの実施 |
| UAT実施者（Admin役） | Nguyen Minh Tri（ロールプレイ） | Admin編シナリオの実施 |
| 承認者（Sponsor相当） | Nguyen Minh Tri | 全シナリオの実施結果を確認し、サインオフを行う |

本プロジェクトはポートフォリオ用の個人開発であるため、実際の複数ステークホルダーの代わりに、役割ごとに操作・確認の観点を分けてロールプレイ形式で実施する。

# 10. 不具合判定基準

| 重要度 | 定義 | リリース可否への影響 |
| --- | --- | --- |
| Critical | データ破損、認証・権限制御の突破、業務が完全に停止する不具合 | 修正必須。リリース不可 |
| Major | 主要業務（打刻・申請・承認・レポート）が正しい結果を返さない不具合 | 原則修正必須。リリース判定者の合意があれば次リリースへ延期可 |
| Minor | 表示崩れ、メッセージ文言の誤り、操作性の軽微な不便 | Conditional Passとして記録し、次リリースで対応可 |
| Cosmetic | 誤字、余白などの見た目のみの指摘 | 任意対応 |

# 11. 受け入れ判定基準（サインオフ条件）

| 条件ID | 内容 |
| --- | --- |
| UAT-AC-001 | 6〜9章のMustシナリオ（AC-001〜AC-009に対応するもの）がすべてPassしていること。 |
| UAT-AC-002 | Critical Bugが1件も残っていないこと（SC-004準拠）。 |
| UAT-AC-003 | Major Bugについて、修正またはリリース判定者との合意があること。 |
| UAT-AC-004 | 横断シナリオ（9章）がすべてPassしていること。 |
| UAT-AC-005 | 実施結果に承認者のサインオフ（記名・日付）があること。 |

本条件を満たした時点で、企画書（01）の成功条件SC-003「User、Manager、Adminの主要業務シナリオがUATでPassすること」を達成したとみなし、リリース手順書（19）に進む。

# 14. まとめ

本書では、HR & Attendance SystemのUATとして、User/Manager/Adminそれぞれの立場でのシナリオ、複数ロールをまたぐ横断シナリオ、不具合判定基準、受け入れ判定基準（サインオフ条件）を定義した。

本書がPassし、11章のサインオフ条件をすべて満たした時点で、リリース手順書（19）に基づく本番リリース判定に進む。
