■ 1. 本書の目的

本書は、HR & Attendance Systemが実際の業務で問題なく利用できるかを、開発者ではなく利用者（またはその代理）の視点で確認するUAT（User Acceptance Test）の実施内容を定義する。

システム試験仕様書（17）が要件・非機能・セキュリティを技術的に確認するのに対し、本書は実際の業務の言葉・操作感でシステムを通し、要件定義書（02）の受入条件（AC-001〜AC-009）と企画書（01）の成功条件（SC-001〜SC-005）を最終的に満たすことを確認する。

■ 2. UAT方針
方針ID方針内容
UAT-POL-001Business Language試験手順・結果は専門用語を避け、業務担当者が読める言葉で記述する。
UAT-POL-002Role PlayUser / Manager / Adminそれぞれの立場になりきって操作する。
UAT-POL-003End-to-End単機能の確認だけでなく、入社〜退職、申請〜承認のような一連の業務を通しで確認する。
UAT-POL-004Evidence Based各シナリオの実施結果はスクリーンショットまたは実施ログを残す。
UAT-POL-005Sign-off RequiredすべてのMustシナリオがPassし、実施者・承認者の署名（記名）をもって完了とする。
■ 3. UAT実施体制
役割担当責務
UAT実施者（User役）Nguyen Minh Tri（ロールプレイ）User編シナリオの実施
UAT実施者（Manager役）Nguyen Minh Tri（ロールプレイ）Manager編シナリオの実施
UAT実施者（Admin役）Nguyen Minh Tri（ロールプレイ）Admin編シナリオの実施
承認者（Sponsor相当）Nguyen Minh Tri全シナリオの実施結果を確認し、サインオフを行う

本プロジェクトはポートフォリオ用の個人開発であるため、実際の複数ステークホルダーの代わりに、役割ごとに操作・確認の観点を分けてロールプレイ形式で実施する。

■ 4. UAT対象・範囲
区分内容
対象要件定義書（02）Must要件、企画書（01）の主要機能（F001〜F009）
対象外給与計算、ICカード連携、Slack通知、多言語対応、モバイルアプリ（開発対象外と同一）
前提システム試験仕様書（17）のST-FUNC / ST-SCN / ST-AC がすべてPassしていること
■ 5. UAT環境
項目内容
環境Development環境（13_インフラ設計と同一構成、本番相当データ規模）
データシステム試験用データを流用し、UAT開始前に打刻・申請状況をリセットする
アカウントUser: EMP001、Manager: EMP002、Admin: EMP003（15_単体試験仕様書 TD-USER系と共通）
実施期間システム試験完了後、リリース判定前の1週間を想定
■ 6. UATシナリオ（User編）
シナリオID手順期待結果対応AC結果(Pass/Fail)実施者実施日

UAT-USER-002①「打刻」を選択 ②出勤ボタンを押す「出勤しました」等の完了メッセージが表示され、出勤時刻が画面に表示されるAC-002
UAT-USER-003①同日中に退勤ボタンを押す退勤時刻と勤務時間（休憩控除後）が表示されるAC-002
UAT-USER-004①同日にもう一度出勤ボタンを押す「既に打刻済みです」旨のエラーが表示され、二重登録されないREQ-004 / BR-001
UAT-USER-005①「勤怠履歴」を選択 ②対象月を指定する自分の出退勤記録のみが一覧表示される（他人のデータは表示されない）AC-003
UAT-USER-006①「休暇申請」を選択 ②種別（有給/欠勤/遅刻/早退）・期間・理由を入力して申請する「申請しました」旨のメッセージが表示され、自分の申請一覧に「承認待ち」として表示されるAC-004

シナリオID手順期待結果対応AC結果(Pass/Fail)実施者実施日
UAT-MGR-001①「休暇承認」を選択する自部署の承認待ち申請一覧が表示される（他部署の申請は表示されない）AC-005
UAT-MGR-002①承認待ち申請の内容を確認し「承認」を押すステータスが「承認済み」に変わり、申請者側の一覧にも反映されるAC-005

UAT-MGR-006①「レポート」から対象月を指定し月次レポートを表示する部署内社員の勤務日数・総勤務時間・休暇日数が表示されるAC-008
UAT-MGR-007①レポート画面からCSV出力を実行するCSVファイルがダウンロードされ、画面表示内容と一致するREQ-014

■ 8. UATシナリオ（Admin編）
シナリオID手順期待結果対応AC結果(Pass/Fail)実施者実施日
UAT-ADM-001①「社員管理」から新規社員を登録する（氏名・メール・部署・シフト・権限を入力）登録が完了し、一覧に反映されるAC-006
UAT-ADM-002①登録直後の社員アカウントでログインを試みる（別ブラウザ／シークレットウィンドウ）登録した認証情報で問題なくログインできるREQ-015
UAT-ADM-003①既存社員を選択し「無効化」を実行する対象社員がログインできなくなるが、過去の勤怠・申請データは参照可能なまま残るREQ-017
UAT-ADM-004①「部署管理」から新しい部署を登録する一覧に反映され、社員登録画面の部署選択肢にも表示されるAC-006
UAT-ADM-005①既存社員が所属する部署を無効化しようとする削除ではなく無効化のみ行われ、既存社員の所属情報はそのまま保持されるER-003
UAT-ADM-006①「シフト管理」から新しいシフト（開始・終了・休憩時間）を登録する一覧に反映され、社員登録画面のシフト選択肢に表示されるAC-006
UAT-ADM-007①開始時刻より前の終了時刻でシフトを登録しようとする入力エラーが表示され、登録されないREQ-019
■ 9. 横断シナリオ（複数ロール連携）
シナリオID手順期待結果対応AC/SC
UAT-CROSS-001入社対応：①AdminがEMP-NEWを登録 ②EMP-NEWでログイン ③出勤・退勤を打刻 ④勤怠履歴で確認登録直後から一連の操作がすべて問題なく完了するAC-001〜AC-003 / AC-006
UAT-CROSS-002休暇申請フロー：①Userが休暇申請 ②Managerが承認待ち一覧で確認 ③承認 ④Userが自分の一覧で「承認済み」を確認申請から承認までの状態がすべての画面で一貫しているAC-004 / AC-005
UAT-CROSS-003月次締めフロー：①複数社員が1ヶ月分打刻・休暇申請 ②Managerが月次レポートを確認 ③CSVを出力し関係者へ共有する想定で内容を確認する実運用と同じ流れで、レポートの数値に矛盾がないAC-008 / SC-003
UAT-CROSS-004権限逸脱フロー：①Userが承認・レポート・管理系の全画面へのアクセスを試みる（画面遷移・URL直打ちの両方）すべて一貫して拒否されるAC-007
■ 10. 不具合判定基準
重要度定義リリース可否への影響
Criticalデータ破損、認証・権限制御の突破、業務が完全に停止する不具合修正必須。リリース不可
Major主要業務（打刻・申請・承認・レポート）が正しい結果を返さない不具合原則修正必須。リリース判定者の合意があれば次リリースへ延期可
Minor表示崩れ、メッセージ文言の誤り、操作性の軽微な不便Conditional Passとして記録し、次リリースで対応可
Cosmetic誤字、余白などの見た目のみの指摘任意対応
■ 11. 受け入れ判定基準（サインオフ条件）
条件ID内容
UAT-AC-0016〜9章のMustシナリオ（AC-001〜AC-009に対応するもの）がすべてPassしていること。
UAT-AC-002Critical Bugが1件も残っていないこと（SC-004準拠）。
UAT-AC-003Major Bugについて、修正またはリリース判定者との合意があること。
UAT-AC-004横断シナリオ（9章）がすべてPassしていること。
UAT-AC-005実施結果に承認者のサインオフ（記名・日付）があること。

本条件を満たした時点で、企画書（01）の成功条件SC-003「User、Manager、Adminの主要業務シナリオがUATでPassすること」を達成したとみなし、リリース手順書（19）に進む。

■ 14. まとめ

本書では、HR & Attendance SystemのUATとして、User/Manager/Adminそれぞれの立場でのシナリオ、複数ロールをまたぐ横断シナリオ、不具合判定基準、受け入れ判定基準（サインオフ条件）を定義した。

本書がPassし、11章のサインオフ条件をすべて満たした時点で、リリース手順書（19）に基づく本番リリース判定に進む。
