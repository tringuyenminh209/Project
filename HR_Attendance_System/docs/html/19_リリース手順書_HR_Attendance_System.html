■ 1. 本書の目的

本書は、HR & Attendance SystemをDevelopment環境からProduction環境（AWS EC2 + Docker + RDS）へリリースする際の手順を定義する。

インフラ設計（13）のCI/CD設計・Deploy方針を実行可能な手順に落とし込み、システム試験（17）・UAT（18）のPass後、安全かつ再現性のある方法でリリースできるようにする。

■ 2. リリース方針
方針ID方針内容
REL-POL-001Gate Keeping単体・結合・システム試験・UATがすべてPassするまでProductionへリリースしない。
REL-POL-002Snapshot FirstDBに影響するリリースは、必ず事前にRDSスナップショットを取得する。
REL-POL-003Reproducibleリリース手順はGitのタグ・コミットIDで一意に再現できるようにする。
REL-POL-004Zero Silent Change.envや本番設定の変更は、本書またはIssueに記録してから行う。
REL-POL-005Rollback Readyロールバック手順を事前に確認したうえでリリースを実施する。
REL-POL-006Health Check Mandatoryリリース後は必ずヘルスチェック・主要機能のスモークテストを実施する。

■ 6. リリース手順

13_インフラ設計.md 15章のCI/CDパイプラインに対応する具体的な実行ステップを以下に示す。

Stepコマンド／操作内容
1git checkout <release-tag>リリース対象のタグをチェックアウトする
2GitHub Actions起動push/tagトリガーでCIパイプラインを起動する
3composer install --no-dev --optimize-autoloader本番用依存パッケージをインストールする
4npm ci && npm run buildViteでフロントエンドassetをビルドする
5docker buildNginx / PHP-FPM+Laravelのイメージをビルドする
6EC2へDeploySSH deployまたはpull deployでEC2上のコンテナを更新する
7php artisan migrate --forceマイグレーションを実行する（7章参照）
8php artisan config:cache route:cache view:cacheLaravelキャッシュを再生成する
9docker compose restartnginx / app コンテナを再起動する
10ヘルスチェック/healthおよび主要APIが200を返すことを確認する（8章）
■ 7. Migration手順
項目内容
事前確認migrationファイルに破壊的変更（DROP COLUMN、NOT NULL化等）が含まれる場合は特に注意し、影響範囲をテーブル定義書（09）と突き合わせる
実行前RDSスナップショットを取得する（Release前、Migration前に取得、13_インフラ設計 11.2準拠）
実行php artisan migrate --forceをメンテナンスウィンドウ中に実行する
実行後php artisan migrate:statusで全migrationがRanになっていることを確認する
失敗時9章のロールバック手順に従う
■ 8. リリース後確認（Smoke Test）

リリース直後に以下の主要フローを実施し、本番環境で最低限の機能が動作することを確認する。システム試験（17）の全項目を再実施するものではない。

#確認項目期待結果関連
1/health 応答200 OKが返る13_インフラ設計 9.3
2ログイン（API-001）User/Manager/AdminそれぞれでログインできるST-FUNC-001
3出勤打刻（API-004）打刻が成功し、DBに反映されるST-FUNC-004
4勤怠履歴取得（API-007）自分の履歴が表示されるST-FUNC-007
5休暇申請（API-009）申請が成功するST-FUNC-009
6月次レポート（API-012）集計が表示されるST-FUNC-012
7HTTPS本番URLがHTTPSでアクセスできるNFR-010
8ログ出力Application Log / Access Logが正常に出力されている13_インフラ設計 14.2

すべてPassした場合のみ、リリース完了とみなす。1件でもFailした場合は9章に従いロールバックを検討する。

■ 9. ロールバック手順
障害パターンロールバック手順
Application起因（画面/API異常）直前のGitタグへgit checkoutし、docker compose restartで再起動する（13_インフラ設計 16.2準拠）
Migration失敗・データ不整合6章実行前に取得したRDSスナップショットから復元する。復元後、アプリケーションも直前バージョンへ戻す
Deploy自体の失敗（Container起動不可）docker compose ps / logsでコンテナ状態を確認し、直前のDockerイメージで再起動する
ロールバック後8章のSmoke Testを再実施し、正常性を確認したうえで関係者へ報告する（11章）

ロールバック実施の判断は、リリース責任者が9章の情報をもとに決定する（判定基準は10章と対称）。

■ 10. リリース判定基準
判定条件
Go5章のチェックリストがすべて完了し、8章のSmoke Testが全項目Passしている
No-Go5章のチェックリストに未完了項目がある、またはCritical Bugが未修正である
Rollbackリリース後にSmoke TestでFailが発生し、9章のロールバック手順を実施する必要がある

■ 13. まとめ

本書では、HR & Attendance Systemのリリースにあたり、リリース前チェックリスト、具体的なデプロイ手順、Migration手順、リリース後のSmoke Test、ロールバック手順、リリース判定基準、連絡・報告方法を定義した。

リリースが完了した後は、運用保守手順書（20）に基づき日次・週次・月次の運用へ移行する。
