■ 1. 本書の目的

本書は、HR & Attendance Systemのシステム試験仕様を定義する。

単体試験がController/Service/Model単位、結合試験が画面⇔API⇔DBの連結を確認するのに対し、本書ではシステム全体をエンドユーザーの視点で通しで動かし、要件定義書の機能要件・非機能要件・セキュリティ要件・受入条件を満たしていることを確認する。

本書はUAT（18）の直前工程であり、Pass後にUATへ移行する。

方針ID方針内容
ST-POL-001Black Box内部実装ではなく、画面操作とAPI応答のみに基づいて検証する。

■ 4. 試験対象
対象内容
機能要件REQ-001〜REQ-021に対応する全機能（FUNC-001〜FUNC-021）。
非機能要件NFR-001〜NFR-019（性能、可用性、拡張性、セキュリティ、保守性、ユーザビリティ）。
セキュリティ要件SEC-001〜SEC-005、および14_セキュリティ設計のThreat Model（TH-001〜TH-008）。
業務シナリオ出退勤、休暇申請・承認、月次レポート、マスタ管理を組み合わせた実運用相当のシナリオ。
障害・異常系DB接続断、二重打刻、権限外アクセス、入力異常などシステム全体としての耐性。
受入条件要件定義書19章 AC-001〜AC-009。
■ 5. 試験環境
項目内容
構成AWS EC2 + Docker（Nginx / PHP-FPM+Laravel 12 / Node-Vite） + RDS相当のMySQL 8（13_インフラ設計と同一構成）
ネットワーク結合試験と分離した検証（Development）環境を想定
データ本番類似のマスタ・複数月分の勤怠・休暇データを投入
監視CloudWatch相当のCPU/Memory/Disk/DB監視を有効化（14_インフラ設計 14章準拠）
ユーザーUser / Manager / Admin 各ロールにつき2〜3アカウント（部署をまたぐ組み合わせを含む）
ブラウザPC Chrome（最新版）を主対象（UI-005 PC First）
■ 6. 試験データ
6.1 組織データ
項目内容
部署dept_admin（管理部） / dept_dev（開発部） / dept_hr（人事部）
シフトshift_normal（09:00〜18:00、休憩60分）/ shift_short（休憩30分の別シフトを追加投入）
社員各部署に3〜5名、Manager 1〜2名、Admin 1名を含む合計15名規模
6.2 業務データ
項目内容
勤怠直近2ヶ月分、正常打刻・二重打刻試行・未退勤などのパターンを含む
休暇申請Pending / Approved / Rejectedの各状態、承認待ちが複数部署にまたがる状態を含む
監査ログ上記操作から自然発生するaudit_logs
■ 7. 判定基準
判定条件
Pass画面表示、業務結果、非機能指標、セキュリティ、受入条件のすべてが要件どおりである。
Fail機能・非機能・セキュリティ・受入条件のいずれかで要件との不一致がある。
Blocked環境・データ・前提機能が未整備で実行できない。
Conditional Pass軽微な指摘があるがCritical/Major Bugではなく、リリース判定者の合意のもとで許容する。
