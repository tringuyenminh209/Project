# 1. 本書の目的

本書はDisaster Safety Report Systemのインフラ構成を定義する。Project 01〜03のDocker Compose / AWS基本構成を踏襲しつつ、本プロジェクト特有の2点 — ①Google Maps/Geocoding APIキーの管理（バックエンドを経由しない、フロントエンド直接呼び出し特有の考慮）、②S3を本初期リリースでは提供しないという判断 — を追加定義する。

---

# 2. インフラ設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| INF-POL-001 | Docker First | ローカル・本番ともにDocker Composeベース（Project 01〜03と統一）。 |
| INF-POL-002 | 段階的AWS移行 | 開発はローカルDocker、本番はAWS EC2 + RDS。 |
| INF-POL-003 | Secret外部化 | DBパスワードは環境変数で管理し、コードにもGitにも含めない。 |
| INF-POL-004 | 未使用機能にインフラを提供しない | S3は07_機能一覧にファイルアップロード機能が1つも存在しないため、本初期リリースではバケット・IAMロールを作成しない（11章で詳述）。使わないリソースを用意することは、無駄なコストだけでなく無駄な攻撃面（不要なIAM権限、放置されがちな設定）を生む。 |
| INF-POL-005 | FE/BEビルド分離 | SPAのビルド（Node）とAPIの実行（PHP）を分離し、本番EC2にNodeランタイムを置かない（ビルドはCIで実施）。 |
| INF-POL-006 | 外部APIキーは「秘匿」ではなく「用途制限」で守る | Google Maps/Geocoding APIキーはブラウザから直接呼び出す設計（00_開発計画書 4章）のため、原理的にクライアント側コード（配布物）に含まれ、秘匿できない。ReverbのAPP_SECRETのような「絶対に漏らさない」管理ではなく、Google Cloud ConsoleでのHTTPリファラ制限・APIごとの有効化・利用量アラートという「漏れても悪用範囲を制限する」設計にする（14章で詳述）。 |

# 11. S3設計（本初期リリースでは提供しない）

**確定事項（INF-POL-004）**: 07_機能一覧のFUNC-001〜019にファイルアップロード・ダウンロードを伴う機能が1つも存在しないため、本初期リリースではS3バケット・IAMロールを作成しない。00_開発計画書 4章・13章で「災害マニュアル等の添付ファイル」という将来拡張候補が挙げられているが、その機能を実際に設計・実装する段階で本章を改訂し、バケット命名・格納パス・IAMポリシー・削除方針（Project 03の`task_files`と同様の設計）を新設する。

**この判断の理由**: 使われないインフラを先回りして用意することは、①無駄なコスト（バケット自体は無料だが、IAMロールの管理対象が増える）、②無駄な攻撃面（設定ミスによるPublic化のリスクを、使われない機能のために抱える）を生む。00_開発計画書のアーキテクチャ図にS3が描かれていることと、実際にプロビジョニングすることは別の意思決定である — 図は「将来の拡張余地」を示すものであり、「今すぐ用意すべきもの」ではない。

# 14. 環境変数・Secret管理

## 14.1 バックエンド（実行時に参照、真のシークレット）

| 変数 | 内容 |
| --- | --- |
| DB_HOST / DB_DATABASE / DB_USERNAME / DB_PASSWORD | RDS接続情報 |
| SANCTUM関連 / CORS_ALLOWED_ORIGINS | SPA配信オリジンのみ許可（NFR-013、14_セキュリティ設計で確定） |

AWSキー（S3用）はINF-POL-004により本初期リリースでは存在しない。

## 14.2 フロントエンド（**ビルド時に焼き込み** — SPA特有の注意）

| 変数 | 内容 |
| --- | --- |
| VITE_API_BASE_URL | APIベースURL |
| VITE_GOOGLE_MAPS_API_KEY | Google Maps JavaScript API / Geocoding API共用キー |

**`VITE_`変数は全てビルド時にJSへ焼き込まれ、配布物として全ユーザーに見える**（Project 01〜03には存在しなかったSPA特有の制約）。特に`VITE_GOOGLE_MAPS_API_KEY`はINF-POL-006の対象であり、以下の2点で運用する（Reverbのシークレットのように「漏らさない」運用はそもそも成立しない）:

1. **Google Cloud ConsoleでHTTPリファラ制限**を設定し、本番ドメイン・ローカル開発ドメイン以外からのAPIキー利用を拒否する。
2. **利用量アラート**を設定し、無料枠（月$200クレジット、00_開発計画書 6章）超過の予兆を早期検知する（00_開発計画書 11章のリスク対策）。

環境ごとに異なるキー・リファラ制限を使うため、ビルド成果物の環境間流用は不可（本番用ビルドをローカルで動かすとリファラ不一致でMapsが表示されない）。

# 21. まとめ

Project 01〜03のインフラ資産（Docker Compose・VPC・Scheduler）を最大限流用しつつ、本システム特有の2点を確定した。①**S3を本初期リリースでは提供しない**（INF-POL-004） — 00_開発計画書のアーキテクチャ図に描かれた「将来の拡張余地」と「今すぐ用意すべきインフラ」を区別し、未使用機能のためにコスト・攻撃面を抱え込まない判断。②**Google Maps APIキーの管理方針**（INF-POL-006） — フロントエンド直接呼び出しという設計（00_開発計画書 4章）の帰結として、このキーは原理的に秘匿できないため、Reverbのシークレットとは異なり「リファラ制限＋利用量アラート」という被害限定型の対策で守る。結果として本システムのインフラはPMS（Reverbコンテナ・WebSocketポート・S3のPrivateバケット運用）より単純であり、これは機能不足ではなく、10_API設計・11_基本設計書で確定した設計判断（WebSocket不採用・ファイル機能なし）が正しくインフラに反映された結果である。
