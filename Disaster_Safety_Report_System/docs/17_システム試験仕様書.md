# 1. 本書の目的

実際のブラウザ操作に近い形で、災害発生〜通知〜安全報告〜集計確認までの一連の業務フローが、SPA・API・DB・Google Maps/Geocoding APIを含めたシステム全体として期待通り動作することを確認する。15/16章で分解確認した要素を、実際の災害対応の時系列に沿って結合する最終段階である。

---

# 2. 試験方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| ST-POL-001 | E2E | 実際の画面操作の連鎖で確認する（APIを直接叩かず、SPA経由）。 |
| ST-POL-002 | Role Based | Admin / 部門管理者 / 一般社員それぞれの視点でシナリオを作成する。 |
| ST-POL-003 | Recovery | Google Maps/Geocoding API障害・安全報告の同時送信等の異常系からの継続動線を確認する。 |
| ST-POL-004 | Multi-Actor | 災害対応の本質（複数従業員が同時に報告・複数の管理者が同時に閲覧）を反映し、可能な限り複数ブラウザ（Admin/部門管理者/一般社員）で並行実施する。 |

---

# 3. 試験環境

Docker Composeのフルスタック環境（Local）またはAWS Development環境。13_インフラ設計の構成（nginx/app/db/scheduler/vite build）を使用し、16_結合試験仕様書と同一の環境を流用する。開発用Google Maps APIキー（リファラ制限を試験環境ドメインに限定）を使用する。

# 4. E2Eシナリオ試験

## ST-SCN-001 災害発生〜報告〜集計確認の一気通貫

```
1. Adminとしてログインし、SCR-006（災害管理）で災害イベントを新規作成する（種別=地震、対象範囲=全社）
2. 作成確認ダイアログに表示される配信対象人数を確認し、作成を確定する → 同報通知fan-outが発火する
3. 一般社員として別ブラウザでログインする → SCR-002ホームにAlert Banner（安否報告のお願い）が表示される
4. ヘッダーの通知ドロップダウンに disaster_alert 通知が表示されていることを確認する
5. 通知をクリックする → SCR-003（安全報告フォーム）へ直リンク遷移する
6. 「安全です」を選択し、コメントは空欄のまま、現在地取得（GPS）を試みて送信する
7. 送信完了トーストを確認 → SCR-002へ戻り、当該災害カードの報告状況バッジが更新されていることを確認する
8. 部門管理者として別ブラウザでログインし、SCR-004（部署ダッシュボード）を開く → 手順6の一般社員の報告が反映されている（安全カウント+1、位置情報があれば地図に緑ピン表示）
9. Adminとして全社ダッシュボード（SCR-005）を開く → `by_department`に部門管理者が確認した内容と一致する内訳が表示される
期待結果: 全ステップが例外なく完了する。各画面の表示内容がAPI応答・DB状態と一致する
```

## ST-SCN-002 状況悪化の再報告と災害収束フロー

```
1. ST-SCN-001の一般社員が再度SCR-003を開く（「報告を更新する」から遷移） → 前回の報告内容（安全）が初期値表示される
2. 「支援が必要です」へ変更し、コメントに具体的な状況を記入、住所を入力してジオコーディングによる位置情報プレビューを確認して送信する
3. 部門管理者のSCR-004で要支援者一覧に当該従業員が最優先表示（Safety-needs-help色）されることを確認する
4. Adminが当該災害をSCR-006から収束へ切り替える（確認ダイアログ「新規の安否報告を受け付けなくなります」を確認）
5. 収束後、まだ報告していない別の一般社員がSCR-003を開こうとする → E006が「この災害は収束済みのため、新しい報告はできません」の文脈文言で表示され、SCR-011へ誘導される
期待結果: 状況変化（安全→要支援）が集計・地図に反映され、収束後の新規報告が画面上で正しく拒否される
```

## ST-SCN-003 未報告者への催促バッチ〜報告完了

```
1. 進行中の災害に対して未報告のままの一般社員を残しておく
2. 催促バッチ（SendReportReminders）の実行を待つ、または手動実行する
3. 当該一般社員でログインし、通知一覧に report_reminder が現れることを確認する
4. 通知をクリックしてSCR-003へ遷移し、報告を完了する
5. 再度バッチを実行する → 当該従業員は既に報告済みのため、次回の抽出対象から自然に外れる（催促が来ない）
期待結果: 催促通知が作成され、報告完了後は同一災害について再度催促されない（BR-NTF-002の実地確認）
```

## ST-SCN-004 部署スコープ越境の実地確認

```
1. 部門管理者（営業部所属）としてログインし、SCR-004部署ダッシュボードを開く
2. ブラウザの開発者ツールでAPI応答（`GET /dashboard/department`）を確認し、他部署（総務部等）の従業員・報告データが一切含まれないことを確認する
3. 同ブラウザのURLを直接編集して他部署を示唆するパラメータを付与してみる（例: クエリ文字列に部署IDを追加）→ サーバー側がそもそもパラメータを解釈しないため、応答内容に変化がない（自部署の結果のみ）ことを確認する
4. 一般社員としてログインし、他人の安全報告詳細を閲覧できる画面上の経路が存在しないことを確認する（SCR-011の集計サマリが自分の報告状況のみで、個人名一覧は表示されない）
期待結果: 部署スコープ・個人スコープの越境が画面操作・API直接確認のいずれでも発生しない
```

# 5. ロール別シナリオ試験

| ロール | シナリオ | 確認内容 |
| --- | --- | --- |
| Admin | 組織マスタ・災害の統括管理 | 部署/従業員/災害のCRUD、全社ダッシュボードの閲覧ができる。**安全報告フォームへの導線がUI上に一切存在しない**（BR-PRM-003、Report-CTAが表示されない） |
| 部門管理者 | 自部署の把握 + 一従業員としての報告 | SCR-004で自部署のみ把握でき、他部署のメニュー・データは一切表示されない。自分自身の安全報告（一従業員として）はSCR-002/003から通常どおり行える |
| 一般社員 | 通知受信〜報告〜再報告 | 通知受信→報告→状況変化時の再報告のみが行え、管理画面（部署/従業員/災害管理・ダッシュボード）へのメニュー自体が表示されない |

# 6. 障害回復試験

| Test ID | シナリオ | 確認内容 |
| --- | --- | --- |
| ST-REC-001 | Google Maps APIキーを試験的に無効化した状態で部署ダッシュボードを開く | 地図パネルのみ「地図を表示できません」のエラー表示となり、集計数値・要支援者一覧は通常どおり表示される（UI-005の実地確認） |
| ST-REC-002 | ネットワークを一時切断した状態で安全報告フォームから住所によるジオコーディングを試みる | ジオコーディング失敗の警告表示（エラートーストではない）が出るが、送信ボタンは活性のままであり、位置情報なしで送信が成功する（NFR-007の実地確認） |
| ST-REC-003 | 2つのブラウザで同一の従業員として同一災害への初回報告をほぼ同時に送信する | 一方が成功し、`safety_reports`に重複行が作成されない（atomic upsertの最終実地確認、15/16章の単体・結合試験の総仕上げ） |
| ST-REC-004 | トークン期限切れ（8時間経過）の状態で操作を続ける | 次のAPI呼び出しでログイン画面へ強制的に戻される |

# 10. まとめ

本書は15_単体試験仕様書（ロジック網羅）・16_結合試験仕様書（実物境界の確認）に続く最終段階として、実際の災害対応シナリオでシステム全体を通しで確認する。特にST-SCN-004（部署スコープ越境の実地確認）は14_セキュリティ設計の部署スコープ対策がユーザー視点でも機能することの最終証跡であり、ST-REC-001（Google Maps API障害時の継続）は「地図は補助であり必須ではない」という設計方針（UI-005）が実際に効くことの証跡である。ST-REC-003（同時送信）はatomic upsert設計の総仕上げとして、単体（UT-RPT-007）・結合（IT-DB-001）に続く3段階目の確認となる。
