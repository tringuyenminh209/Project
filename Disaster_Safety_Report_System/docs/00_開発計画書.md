# 1. 開発目的

本プロジェクトは、Portfolio Roadmap（SE/BrSE 2026-2027）における Project 04 として位置づけられる、企業内の防災・安否確認システムである。災害発生時に従業員が安否状況を報告し、管理者が組織全体の状況をリアルタイムに把握できることを目的とする。

Project 01（HR & Attendance System）で組織階層（部署・従業員）とCRUD・認証を、Project 02（EC Site）でトランザクション整合性を、Project 03（Project Management System）でSPA分離・通知fan-out・並行処理制御を習得したのに対し、本プロジェクトは以下の新しい難易度軸を目的とする。

- **GIS統合**: 初めて地図（座標・位置情報）を扱う。地図上への位置プロット、住所からの座標変換（ジオコーディング）、外部地図APIとの連携方式を習得する
- **大量同報通知（Broadcast Notification）**: Project 03の通知は「担当者1名への個別通知」だったのに対し、本プロジェクトは「災害発生時に全社員（または該当部署の全員）へ一斉送信」する必要がある。件数が数百〜数千件になり得るため、fan-outの実装方式（同期ループか、キュー経由の非同期処理か）を設計段階で検討する
- **リアルタイム集計ダッシュボード**: 「部署ごと・全社の安否報告状況」を刻々と変化する状態として集計表示する、読み取り負荷の高い画面を初めて設計する
- **組織階層に基づく業務システム**: Project 01の部署（`departments`）を土台に、「本人の安否報告」「部門管理者による自部署の状況把握」「Adminによる全社統括」という3層の業務要件を扱う

# 2. 開発範囲

## 2.1 開発対象（In Scope）

- ログイン・ログアウト（Sanctum トークン認証、Project 01/03と同一方式）
- 組織マスタ管理（Admin）: 部署・従業員のCRUD
- 災害管理（Admin）: 災害イベントの作成・進行中/収束の切替・詳細編集
- 安全報告（一般社員）: 進行中の災害に対して自分の安否状況（安全 / 要支援）を報告する。任意でコメント・現在地を添える
- 通知（同報配信）: 災害作成時に対象従業員へ一斉通知、未報告者への催促通知
- ダッシュボード: 部署別・全社の報告状況をリアルタイム集計（◯/◯人報告済み、要支援者一覧）
- 地図表示: 従業員の勤務地・報告時位置情報を地図上にプロットし、安否ステータスで色分け表示（Google Maps、6章参照）

## 2.2 開発対象外（Out Scope）

- マルチテナント／複数企業対応（3章・9章で確定 — 単一企業内システムとする）
- 気象庁 緊急地震速報API等、外部の災害検知サービスとの自動連携（将来拡張、13章）
- 従業員↔Admin間の双方向チャット（本システムは一斉通知＋一方向の報告のみ）
- オフライン地図キャッシュ・バックグラウンド位置同期（Bonus扱い。`README.md`で確定済み — オフライン対応特有の複雑さ（コンフリクト解決、同期キュー管理）が本プロジェクトのMid-levelという難易度を超えるため、コア機能安定後の拡張として切り離す）
- ネイティブモバイルアプリ（Vue.jsのレスポンシブWebのみ。ネイティブはProject 10で扱う）
- 多言語UI（日本語のみ。内部文書はベトナム語併記、Portfolio全体の方針を踏襲）

# 3. 対象ユーザー

**確定事項（持ち越し論点①）**: 本システムは**単一企業内の防災・安否確認システム**とする。複数企業が同一システムを共用するマルチテナントSaaSではない（理由は9章で詳述）。ユーザー層は以下の3層とする（確定事項②）。

| ユーザー種別 | 層 | 説明 |
|---|---|---|
| Admin | 全社 | BCP（事業継続計画）担当・危機管理部門。災害イベントの作成・管理、全社ダッシュボード閲覧、組織マスタ（部署・従業員）管理を行う |
| 部門管理者 | 部署単位 | 各部署の管理職。自部署の安否報告状況をリアルタイムに把握し、未報告者への個別フォロー（電話等、システム外の対応）に活用する |
| 一般社員 | 個人 | 災害発生時に通知を受け取り、自分の安否状況を報告する。他の従業員の報告は閲覧できない（自部署の集計値のみ部門管理者が見られる） |

**3層にした理由**: 「安全報告」「災害管理」「ダッシュボード」というREADME記載の3機能を素直に実装すると、報告する人（一般社員）・報告を集計して現場対応する人（部門管理者）・災害そのものを管理する人（Admin）は明確に異なる関心事を持つ。日本の実際の安否確認システム（Biz安否確認、セコム安否確認等）も同様の3層構成を取っており、Project 01のUser/Manager/Adminという役割分担パターンとも一貫性がある。2層（社員／Admin）に単純化する案も検討したが、部門管理者による部署単位の把握はBCPの実務上省略できないため採用しない。

# 4. システム概要

## システム構成

```
[Browser]
   │ HTTPS
   ▼
[Nginx (Reverse Proxy)]
   ├── /            → Vue.js SPA（ビルド済み静的ファイル配信）
   ├── /api         → [Laravel API (PHP-FPM)] ──► [AWS S3（今後の拡張: 災害マニュアル等の添付ファイル）]
   │                        │
   │                        ▼
   │                  [MySQL 8 (RDS)]
   └── （ブラウザ→外部直接） [Google Maps JavaScript API / Geocoding API]
                              ※フロントエンドから直接呼び出す。APIキーはHTTPリファラ制限必須（14章）
```

Project 03（PMS）で確立した「Vue.js SPA + Laravel API」の分離構成を踏襲する（フロントエンド/バックエンド完全分離、Nginxが`/`をSPA、`/api`をLaravelへ振り分け）。地図表示は、位置情報を含むレスポンス自体はLaravel APIが返し、地図タイル・ジオコーディングの呼び出しはブラウザからGoogle Maps APIへ直接行う（サーバー側でAPIキーを扱う必要がなく、Google側の利用量ダッシュボードでも計測しやすいため）。

# 5. 主な機能

## 5.1 認証機能
ログイン・ログアウト。Sanctumトークン認証。業務システムのためトークン期限は8時間（Project 01/03と同一基準）。

## 5.2 組織マスタ管理機能（Admin）
部署（`departments`）・従業員（`employees`）のCRUD。Project 01の部署・従業員管理パターンを踏襲する。

## 5.3 災害管理機能（Admin）
災害イベント（`disasters`）の作成（種別・発生日時・対象部署範囲）、進行中→収束のステータス切替、詳細編集。災害作成時に5.5の通知を発火する。

## 5.4 安全報告機能（一般社員）
進行中の災害（`disasters`）に対し、自分の安否状況を`safety_reports`へ記録する。ステータスは`safe`（安全）/ `needs_help`（要支援）の2値（02_要件定義書 BR-RPT-001で確定。「未確認」は報告が存在しないことから導出される表示上の状態であり、選択肢ではない）。任意でコメント・現在地（ジオコーディングまたはGPS座標）を添付できる。1つの災害につき1従業員1報告とし、状況変化時の再報告は既存行のUPDATEで反映する（02_要件定義書 BR-RPT-002で確定）。

## 5.5 通知機能（同報配信）
災害作成時、対象従業員へ一斉通知（`notifications`）を作成する。件数が多くなり得るため、Project 03のfan-out実装（1件ずつループ）をそのまま流用できるかを12_詳細設計書で検証し、必要ならキュー経由の非同期処理（Laravel Queue）へ切り替える。未報告者への定期催促通知（バッチ）も設ける。

## 5.6 ダッシュボード機能
部署別・全社の報告状況をリアルタイム集計表示（◯/◯人報告済み、要支援者一覧）。部門管理者は自部署のみ、Adminは全社を閲覧できる。

## 5.7 地図機能
Google Maps上に、従業員の勤務地（`locations`マスタ、通常時）または安全報告時の位置情報（`safety_reports`、災害時）をピン表示し、安否ステータスで色分けする。

# 6. 使用技術

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | Vue 3 + TypeScript + Vite + Pinia + Vue Router | Project 03と統一。SPA分離構成の知見をそのまま活用できる |
| CSS | Tailwind CSS | Project 02/03と同一 |
| HTTPクライアント | fetchベースの自作APIクライアント | axiosは採用しない（Portfolio全体の方針） |
| バックエンド | Laravel 12 / PHP 8.4 | Project 01〜03と統一 |
| データベース | MySQL 8 | Project 01〜03と統一 |
| 地図 | Google Maps JavaScript API + Geocoding API | 下記「地図API選定理由」参照（ADR相当、詳細は`docs/adr/`に別途記録） |
| インフラ | Docker Compose → AWS EC2 / RDS / S3 | Project 01〜03の構成を踏襲 |
| CI/CD | GitHub Actions | push時にバックエンドテスト + フロントエンドビルド/型チェックを実行 |
| （Bonus）オフライン対応 | Service Worker（Workbox）+ IndexedDB | `README.md`確定済み。コア機能安定後に着手する拡張枠 |

**地図API選定理由（Google Maps vs Mapbox — ADR-001相当、確定事項①）**:

- **候補**: Google Maps API / Mapbox（Roadmapでは両論併記）
- **決定**: Google Maps API（Maps JavaScript API + Geocoding API）
- **理由**: ①本システムの入力は「住所」（従業員の勤務地、安全報告時の現在地）であることが多く、日本語住所のジオコーディング精度はGoogleが実務上安定している。②日本国内での知名度・導入実績が高く、BrSEとしての就業先候補企業でも採用例が多いため、ポートフォリオとしての説明コストが低い。③無料枠（月$200クレジット）は本プロジェクトの学習規模（デモ用データ、実運用しない）であれば十分。
- **トレードオフ**: 無料枠を超えた本番運用では従量課金が発生する（Mapboxの方が地図タイル表示自体は無料枠が広い）。本プロジェクトは学習用でありスケールしないため許容する。実運用移行時に再検討する前提をADRに明記する
- **対策**: フロントエンドのAPIキーはHTTPリファラ制限を必須とし、`.env`経由でキーを注入する（Project 02のStripeキー管理と同じ原則 — シークレットを直書きしない）

# 9. データベース概要

## テーブル一覧（7テーブル + フレームワーク管理分）

`companies`, `departments`, `employees`, `disasters`, `safety_reports`, `locations`, `notifications`

Roadmap記載の7テーブルをそのまま採用する。Sanctumの`personal_access_tokens`等フレームワーク管理テーブルは数に含めない。

## `companies`テーブルの位置づけ（確定事項①の補足）

`companies`という名前から複数企業（マルチテナント）を連想しやすいが、本プロジェクトでは**単一企業のプロフィール情報（本社所在地・BCP連絡先等）を保持する1行のみのマスタテーブル**として扱う。理由:

1. Project 03（PMS）で「プロジェクト単位のリソーススコープ制御」（`project_members`によるテナントに近い分離）を既に習得済みであり、本プロジェクトで本格的なマルチテナントSaaS（`company_id`によるあらゆるクエリのスコープ制御、テナント跨ぎのセキュリティ試験）を再度実装しても学習の新規性が薄い
2. 実際の企業向け安否確認システムは、企業ごとに専用契約（オンプレミス/専用SaaSインスタンス）で導入されることが多く、マルチテナント共用は必須要件ではない
3. `companies`テーブル自体は残す — 会社名・本社住所（地図の初期表示位置に使う）等、将来的に本当に複数拠点/複数法人へ拡張する余地は残しつつ、**今回のスコープでは`company_id`によるアプリ層でのテナント分離ロジックは実装しない**（全クエリは部署・従業員単位のみでスコープする）

## ER概要

中心エンティティは`disasters`と`employees`。`companies`(1) → `departments`(N) → `employees`(N)という組織階層を持つ。`disasters`(1) → `safety_reports`(N)で、1つの災害に対し複数の安全報告が紐づく。`safety_reports`は`employees`を参照する（誰の報告か）。`locations`は`employees`の通常の勤務地マスタとして参照される想定だが、安全報告時の実際の位置情報（ジオコーディング結果やGPS座標）を`safety_reports`自体に持たせるか`locations`を都度作成するかは、08_ER図で設計判断として確定する（現時点の論点として記録）。`notifications`は災害通知の宛先・既読状態を管理する（Project 03の`notifications`設計を踏襲、宛先が1件でなく大量になる点が差分）。

設計上の主要論点（08/09で判断を記録する）:

- ~~`locations`と`safety_reports`の位置情報の持ち方~~ → **確定済み**: 02_要件定義書 BR-RPT-003で「`locations`はマスタ参照、`safety_reports`は報告時点のスナップショットを自身に保存」に確定。物理設計（列・制約）は08_ER図/09_テーブル定義で確定する
- 通知の同報配信をアプリ層の同期ループで実装するか、Queueで非同期化するかの閾値（対象人数がどの程度からQueueへ切り替えるか）→ 02_要件定義書 NFR-001で性能要件（1,000人・60秒以内）を確定。実装方式自体は12_詳細設計書で判断する
- ~~1つの災害に対する従業員の再報告を許可するか~~ → **確定済み**: 02_要件定義書 BR-RPT-002で「許可する。UNIQUE(disaster_id, employee_id)のUPDATE方式、履歴は残さない」に確定

# 10. テスト計画

## 単体試験
災害ステータス遷移（進行中→収束）、安全報告のバリデーション（未進行中の災害への報告拒否等）、通知の同報配信件数、ジオコーディング失敗時のフォールバック。

## 結合試験
API ↔ DB、API ↔ Google Maps API（ジオコーディング呼び出し）、災害作成 ↔ 通知fan-out ↔ 未報告者集計。

## システム試験
Admin: 災害作成→通知配信確認 → 一般社員: 通知受信→安全報告→地図反映確認 → 部門管理者: 自部署ダッシュボード確認、の一連シナリオ。

## テスト項目（概要）
- 収束済み・未作成の災害への安全報告拒否
- 部門管理者が他部署のダッシュボードを閲覧できないこと（IDOR）
- 大量通知配信時のパフォーマンス（同報件数がN件を超えた場合の応答時間）
- ジオコーディングAPI障害時、報告自体は失敗させない（位置情報なしでも報告を受理する）

詳細は`15〜17`の各試験仕様書で定義する。

# 11. リスク管理

| リスク種別 | リスク内容 | 影響度 | 発生確率 | 対策 | 担当 |
|---|---|---|---|---|---|
| 技術リスク | Google Maps APIの利用量課金・キー管理に不慣れ | 中 | 中 | 無料枠内でのデモ運用に限定し、APIキーにリファラ制限をかける（14_セキュリティ設計で確定） | 自分 |
| 技術リスク | 日本語住所のジオコーディング精度 | 低 | 低 | Google Geocoding APIを採用（6章の選定理由）。失敗時は位置情報なしで報告を受理するフォールバックを設計 | 自分 |
