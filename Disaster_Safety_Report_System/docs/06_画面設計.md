# 2. 画面設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UI-001 | 報告系=TOUI / 管理系=OOUIのハイブリッド | 一般社員・部門管理者向けの報告系画面（SCR-002/003）は「安否を報告する」というタスクを最短で完了させるタスク指向UI（TOUI — ATM・申請フォーム型）とする。Admin向けの管理系画面（SCR-005〜008）は災害・部署・従業員という対象物を選んでから操作するオブジェクト指向UI（OOUI）とする。緊急時の報告者と平時の管理者では最適なUI原則が異なるため、1方式に統一しない。 |
| UI-002 | Mobile First | 安全報告は緊急時に社員個人のスマートフォンから行われることを想定し、モバイル（375px基準）を最優先で最適化する（NFR-020。Project 02と同方針、Project 03のPC Firstとは異なる）。ダッシュボード・Admin管理画面はlg（1024px）以上で情報密度を上げるが、モバイルでも全機能が操作可能であること — 災害時に部門管理者・AdminがPCの前にいる保証はない。 |
| UI-003 | 報告フォームの最小化 | SCR-003は必須入力を安否ステータス1項目のみとし、大ボタン2択（安全 / 要支援）+任意項目（コメント・位置情報）の構成とする（NFR-021: 数タップで送信完了）。 |
| UI-004 | 権限による表示制御はUIの配慮、防御はAPI | Admin専用メニューは一般社員・部門管理者に非表示にするが、権限の実体は常にAPI側の判定（BR-PRM）。UIの出し分けをセキュリティと混同しない。 |
| UI-005 | 外部API障害の隔離表示 | Google Maps関連（地図パネル・ジオコーディング）の障害は該当パネルのみのエラー表示に留め、集計数値・報告送信という本質機能のUIは通常表示を継続する（NFR-007）。 |
| UI-006 | Traceability | 各画面はSCR・UC・REQ・エラーIDと対応させる。 |
| UI-007 | 位置情報のない報告を地図に描画しない（v1.1、設計監査で追加） | 安否報告に緯度経度が伴わない場合（ジオコーディング/GPS失敗、NFR-007で受理自体は継続）、その報告を`locations`マスタの通常勤務地へフォールバックして地図にピン表示することはしない。`needs_help`報告が実際とは異なる場所（オフィス）に表示されると、BCP担当者・救助側の初動判断を誤らせる危険がある（10_API設計 6.7節）。該当報告は地図とは別のテキストリストに「位置情報：未確定」として表示する（5.4/5.5/5.11節）。`locations`マスタへのフォールバックは、そもそも報告自体が存在しない**未報告者**（`status=unreported`）にのみ許容する — 「本来いるはずの場所」を示す参考情報として意味を持つため。 |

## 2.1 デザイントークン

全11画面で使い回す値をここで確定する。個別画面で新しい色・余白を発明しない。**基本セットはEC Site / PMS（各`docs/06_画面設計.md` 2.1節）と同一のTailwindデフォルトスケールを踏襲**し（ポートフォリオ内の一貫性・二重管理回避）、本システム固有の安否・災害・地図系トークンのみ追加する。

### 基本トークン（EC Site / PMSと共通）

| 区分 | トークン | Tailwindクラス例 | 用途 |
| --- | --- | --- | --- |
| カラー | Primary | `blue-600` / hover `blue-700` | 主要CTA（ログイン、災害作成、保存） |
| カラー | Secondary | `gray-600` | 補助ボタン（キャンセル、戻る） |
| カラー | Background | `gray-50` | ページ背景 |
| カラー | Surface | `white` | カード・モーダル・パネル背景 |
| カラー | Text-primary | `gray-900` | 見出し・本文 |
| カラー | Text-secondary | `gray-500` | キャプション・補足・Empty文言 |
| カラー | Border | `gray-200` | カード枠・区切り線・スケルトン枠 |
| カラー | Success | `green-600` | 完了メッセージ（報告送信完了等） |
| カラー | Warning | `amber-500` | 注意喚起 |
| カラー | Error | `red-600` | E001〜E010のエラーメッセージ |
| タイポ | H1 / H2 / H3 / Body / Caption | `text-2xl font-bold` / `text-xl font-semibold` / `text-lg font-semibold` / `text-base` / `text-sm text-gray-500` | EC Site/PMSと同一の5段階 |
| スペーシング | 基準単位 | Tailwind標準スケール（4px刻み） | `gap-2`/`p-4`/`p-6`/`gap-8`を基本 |
| ブレークポイント | md / lg | `768px` / `1024px` | **UI-002のMobile First基準**。base=モバイル。md以上でダッシュボードを2カラム化、lg以上でAdminサイドバー常時表示 |
| コンポーネント状態 | Button / Input / Card | EC Site 2.1節と同一（loading=スピナー+非活性、error=`border-red-600`+直下赤字、Card hover=`shadow-md`） | 二重送信防止・E003表示・クリック可能示唆 |

| 安否 | Safety-needs-help | `red-600` | `status=needs_help`の表示。**最も目立つ色を割り当てる**（初動対応の優先対象） |
| 安否 | Safety-unreported | `gray-400` | 未報告（派生状態、BR-RPT-001）の表示 |

# 5. 画面詳細

## 5.0 状態バリエーション 共通ルール

EC Site / PMS 06_画面設計 5.0節と同一の書式を初版から全画面へ適用する。

| 共通ルール | 内容 |
| --- | --- |
| Loading（取得系） | スケルトン（`Border`色の枠 + `Background`色のバー）で件数分プレースホルダー表示 |
| Loading（送信系） | 送信ボタンはButtonトークンのloading状態（二重送信防止）。応答までフォーム非活性 |
| Empty | 0件は正常応答。「なぜ0件か + 次に取るべき行動」を`Text-secondary`で案内 |
| Error（通信失敗） | 「取得に失敗しました。」+ 再試行ボタン |
| Error（業務エラー） | E001〜E010は7章のメッセージを`Error`色で表示。E003は該当Inputのerror状態 + 直下赤字 |
| 地図パネル（本システム固有） | Loading=パネル内スピナー / Error=「地図を表示できません」+ 再試行（**集計数値・フォームは通常表示を継続**、UI-005） |

## 5.1 SCR-001 ログイン画面

### 入力項目
- メールアドレス（Required / Email形式 / Max 255）
- パスワード（Required / 8〜20文字）

### 操作
- 「ログインする」→ 成功時: ロール別ランディングへ（一般社員SCR-002・部門管理者SCR-004・AdminSCR-005）。`redirect`クエリがあれば元のURLへ復帰（G-01 — 通知経由のSCR-003直リンク復帰を含む）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 「ログインする」押下後はButton loading状態 |
| Empty | 対象外 |
| Error | E001/E003をフォーム上部に表示。E001はどちらの項目が誤りかを明示しない |

## 5.2 SCR-002 ホーム（進行中の災害・自分の報告状況）

一般社員のランディング。TOUIの起点（UI-001）。

### 表示項目
- Alert Banner（3.2節 — 未報告の進行中災害があるとき最上部）
- 進行中の災害カード: 種別、発生日時、Disaster-activeバッジ、**自分の報告状況**（Safety-safe / Safety-needs-help / Safety-unreportedトークンのバッジ）
- 各カードのReport-CTA: 未報告なら「安否を報告する」、報告済みなら「報告を更新する」（→SCR-003）
- 過去の災害（収束済み）リンク → SCR-011

### 操作
- Report-CTA → SCR-003。災害カード本体 → SCR-011

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 災害カードをスケルトン表示 |
| Empty | 「現在、進行中の災害はありません。」（`Text-secondary`。平常であることを示す正常状態） |
| Error | 取得失敗は再試行 |

---

## 5.3 SCR-003 安全報告フォーム

本システムのTOUI核心画面（UI-003、NFR-021）。

### 入力項目
- 安否ステータス（**Required・唯一の必須項目**）: 「安全です」（Safety-safe色）/「支援が必要です」（Safety-needs-help色）の大型2択ボタン（Report-CTAトークン）
- コメント（Optional / Max 500）
- 位置情報（Optional）: 「現在地を取得」（GPS）または住所入力（Max 200 → Geocoding API変換）。取得結果を小型地図パネルでプレビュー表示

### 操作
- ステータス選択 → 「送信する」→ 完了トースト（Success）→ SCR-002へ戻る
- 再報告時（BR-RPT-002）: 前回の報告内容を初期値表示し、同じフォームで上書き送信（UC-009）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 送信中はButton loading。**ジオコーディング処理中でも送信ボタンは非活性にしない**（位置情報なし送信を常に許可 — NFR-007、G-07） |
| Empty | 対象外 |
| Error | E006（収束済み・対象範囲外）→「この災害への報告はできません」+ SCR-011へ誘導。ジオコーディング失敗は「位置情報を取得できませんでした（報告はそのまま送信できます）」と`Warning`色で表示し、エラー扱いにしない |

## 5.4 SCR-004 部署ダッシュボード（部門管理者）

### 表示項目
- 進行中の災害セレクタ（複数進行中の場合の切替）
- 自部署の集計サマリ: 報告済み（Safety-safe）/ 要支援（Safety-needs-help）/ 未報告（Safety-unreported）の人数（◯/◯人形式、NFR-002: 3秒以内）
- **要支援者一覧**（最優先表示 — 氏名、コメント、報告日時、位置情報有無）
- 未報告者一覧（氏名のみ。電話等のシステム外フォローの判断材料 — BF-004）
- 地図パネル（Map-panel）: 自部署の**緯度経度を持つ**報告をMap-pin-safe/needs-helpで色分けピン表示 + 未報告者の通常勤務地をSafety-unreported色（グレー）でピン表示 + 凡例（UI-007）
- **位置情報未確定リスト**（v1.1、UI-007）: 報告済み（safe/needs_help）だが緯度経度を持たない従業員を、地図とは別にテキストで表示。氏名・ステータス・コメント・「位置情報：未確定」の注記。要支援者一覧内の該当者にも同じ注記を添える
- 自分自身の報告状況 + Report-CTA（部門管理者も一従業員として報告する — 02 8章）

### 操作
- 手動更新ボタン + 一定間隔の自動再取得（ポーリング。間隔は12_詳細設計書で確定）
- 災害セレクタ切替、SCR-011への遷移

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 集計サマリ・一覧をスケルトン表示。地図パネルは5.0節の地図ルール |
| Empty | 進行中災害なし→「現在、進行中の災害はありません」。位置情報付き報告0件→地図パネルに「位置情報のある報告はまだありません」 |
| Error | 集計取得失敗は再試行。**地図パネルのみの障害は集計表示を止めない**（UI-005、UC-011 3-a）。他部署IDのAPI直撃はE007（G-04） |

## 5.5 SCR-005 全社ダッシュボード（Admin）

### 表示項目
- 進行中の災害セレクタ
- 全社集計サマリ（報告済み/要支援/未報告、Safety-*トークン）
- **部署別内訳テーブル**（部署名、◯/◯人、要支援数 — Adminの部署別把握はこの内訳で行う、05 5章 注1）。行タップで当該部署の詳細（要支援者・未報告者一覧）を展開
- 地図パネル: 全社の**緯度経度を持つ**報告をピン表示 + 未報告者はグレーピン（SCR-004と同一ルール、UI-007）
- **位置情報未確定リスト**（v1.1、UI-007）: SCR-004と同一仕様、全社分
- 「災害管理へ」ショートカット（→SCR-006）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | サマリ・内訳テーブルをスケルトン表示 |
| Empty | 進行中災害なし→「現在、進行中の災害はありません」+ 災害管理へのリンク |
| Error | 一般社員・部門管理者のURL直撃はE002（G-03）。地図パネル障害はUI-005に従う |

## 5.6 SCR-006 災害管理一覧・作成・編集（Admin）

### 表示項目
- 災害一覧（種別、発生日時、対象範囲、Disaster-active/resolvedバッジ、報告集計サマリへのリンク）
- 作成・編集フォーム（モーダル）: 種別（Required / Max 50）、発生日時（Required / 日時形式）、対象範囲（Required / 「全社」または部署の複数選択 — BR-DIS-001）

### 操作
- 「新規作成」→ 作成確認ダイアログに**通知の配信対象人数を表示**（「◯名へ通知が送信されます」— 誤配信の抑止）→ 作成と同時に同報通知発火（UC-005）
- 「収束」→ 確認ダイアログ（「新規の安否報告を受け付けなくなります」）→ `resolved`へ。「進行中に戻す」で取り消し可（REQ-007）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 一覧行スケルトン。作成実行中はButton loading（同報通知作成を含むため、NFR-001の範囲で他画面より長くなり得る旨をスピナー文言「通知を配信しています…」で示す） |
| Empty | 「災害イベントはまだ登録されていません」+ 作成ボタン |
| Error | E003は該当Input直下。一般社員・部門管理者のURL直撃はE002 |

## 5.7 SCR-007 部署管理（Admin）

### 表示項目・入力項目
- 部署一覧（部署名、所属人数、状態）。作成・編集フォーム: 部署名（Required / Max 100）
- 無効化トグル（確認ダイアログ付き）

### 状態バリエーション

| --- | --- |

# 7. エラーメッセージ

| エラーコード | 画面表示メッセージ |
| --- | --- |
| E001 | メールアドレスまたはパスワードが正しくありません。 |
| E002 | この操作を実行する権限がありません。 |
| E003 | 入力内容に誤りがあります。 |
| E006 | この操作は現在の状態では実行できません。 |
| E007 | 対象のデータが見つかりません。 |
| E010 | ログインが必要です。 |

**補足1**: エラーコードの意味はProject 01〜03と共通の語彙（02_要件定義書 16章）。E004/E005/E008/E009/E011は本システムでは使用しない。

**補足2**: 画面固有の文脈に合わせ、上表の標準文言を言い換えて表示してよい（例: SCR-003の収束済み災害への報告はE006を「この災害は収束済みのため、新しい報告はできません。」と表示）。ただしエラーコード自体は変更しない（試験仕様書はコードで判定する）。

**補足3（本システム固有）**: ジオコーディング失敗はE001〜E010のいずれにも該当しない（報告自体は成功するため）。`Warning`色の補足表示のみとし、エラートーストを出さない（5.3節、NFR-007）。

# 9. まとめ

全11画面をMobile First（UI-002）+ 報告系TOUI/管理系OOUIのハイブリッド（UI-001）で設計した。デザイントークンはEC Site/PMSの基本セットを踏襲しつつ、本システム固有の安否3状態（safe/needs_help/unreported）・災害状態・地図ピンのトークンを追加で確定した — 特にSafety-needs-helpに最も強い警告色を割り当て、ダッシュボード・地図・バッジの全てで同一色を使うことで「要支援者がどこにいるか」が画面のどこでも同じ色で追えるようにした。状態バリエーションは全画面初版から適用し、本システム固有の第4状態として「地図パネル障害の隔離表示」（UI-005）を共通ルール化した — 外部API（Google Maps）の障害が安否報告・集計という本質機能のUIを道連れにしないことを、画面仕様のレベルで保証する。v1.1では設計監査により、位置情報のない報告を地図にフォールバック表示することがBCPシステムでは誤誘導リスクになるという業務上重大な論点を発見し、UI-007として位置情報未確定リストを新設した（10_API設計と連動修正）。
