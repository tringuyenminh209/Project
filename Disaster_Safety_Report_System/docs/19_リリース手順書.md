# 1. 本書の目的

本書はDisaster Safety Report SystemをAWS本番環境（EC2 + RDS）へリリースする手順を定義する。Project 03（PMS）と同様に**SPA（Vue 3）とAPI（Laravel）のビルド・デプロイが2系統に分かれる**点に加え、本プロジェクト固有として**Google Maps APIキーの環境別発行・リファラ制限の確認**、および**S3を作成しないことの確認**（13_インフラ設計 INF-POL-004）が手順に含まれる。

# 2. リリース方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| REL-POL-001 | 試験Pass前提 | `15/16/17_試験仕様書.md`が全てPassし、18_UATがサインオフ済みであることをリリースの前提とする。 |
| REL-POL-002 | FE/BE分離ビルド | フロントエンドはCI上でビルドし、本番EC2にはビルド成果物（静的ファイル）のみを配置する（Nodeランタイムを本番に置かない — 13_インフラ設計 INF-POL-005）。 |
| REL-POL-003 | 環境ごとの再ビルド必須 | `VITE_*`環境変数（APIベースURL・Google Maps APIキー）はビルド時に焼き込まれるため、環境（Local/Production）を跨いだビルド成果物の使い回しを禁止する（13_インフラ設計 14.2節）。 |
| REL-POL-004 | 未使用インフラの不作成確認 | S3バケット・IAMロールを誤って作成していないことをリリース前に確認する（13_インフラ設計 INF-POL-004 — 使わないリソースは攻撃面を増やすだけであるため）。 |

# 4. リリース前チェックリスト

```
□ 15/16/17_試験仕様書.md のテストが全てPass
□ 18_UAT.md のシナリオが全てPass、Critical不具合0件
□ .env.production に DB_*/SANCTUM関連が正しく設定されている（13_インフラ設計 14.1節）
□ .env.production の CORS_ALLOWED_ORIGINS が本番SPA配信オリジンのみになっている（14_セキュリティ設計 5.3節）
□ frontend/.env.production の VITE_API_BASE_URL / VITE_GOOGLE_MAPS_API_KEY が本番値になっている（Local用の値が紛れ込んでいないか要確認）
□ 本番用Google Maps APIキーのHTTPリファラ制限が本番ドメインに設定されている（13_インフラ設計 INF-POL-006）
□ Google Cloud Consoleの利用量アラートが設定されている（00_開発計画書 11章リスク対策）
□ AWS S3バケットを作成していないことを確認する（13_インフラ設計 INF-POL-004。誤って作成された未使用バケットが残っていないか）
□ RDSの自動バックアップが有効
□ セキュリティヘッダ（CSPにmaps.googleapis.comを含む）がNginx設定に反映されている（14_セキュリティ設計 5.2節）
□ GitHub ActionsのCI（backend/frontend 2ジョブ）が直近コミットでPass
□ composer audit / npm audit がHigh以上0件
```

# 5. リリース手順

```
1. mainブランチの最新コミットを確認する
2. GitHub Actionsのfrontendジョブでビルド成果物（frontend/dist）を生成する（REL-POL-002）
3. Production EC2にSSH接続する
4. git pull で最新コード（backend）を取得する
5. ビルド成果物（dist/）をEC2の配信ディレクトリへ配置する（7章参照）
6. docker compose build（backend/schedulerイメージに変更がある場合）
7. docker compose up -d
8. php artisan migrate --force（6章参照）
9. php artisan config:cache / route:cache
10. 8章のGoogle Maps APIキー設定確認を実施する
11. 9章のSmoke Testを実施する
```

# 6. Migration手順

```
1. RDSの手動スナップショットを取得する（念のためのバックアップ）
2. Development環境で同一Migrationが成功することを事前確認済みであることを再確認する
3. php artisan migrate --force を実行する
4. php artisan migrate:status で全Migrationが実行済みであることを確認する
```

7テーブルすべてに外部キー制約があるため、Migration順序は09_テーブル定義 3章のテーブル一覧順（依存関係の浅い順: companies/departments/locations→employees→disasters→safety_reports/notifications）を厳守する。

# 7. フロントエンドビルド・配置手順

Project 01（Blade、サーバーサイドレンダリング）には存在しない、本プロジェクト固有の手順である。

```
1. CI（frontendジョブ）で本番用環境変数（.env.production）を用いて vite build を実行する
   → frontend/dist/ に静的ファイル一式が生成される
2. dist/ をEC2の配信ディレクトリ（nginxコンテナがマウントするパス）へ配置する
3. Nginx設定（13_インフラ設計 8章）に try_files $uri /index.html が有効であることを確認する
   → 無効な場合、SPAのディープリンク（例: 通知経由の /disasters/10/report への直接アクセス）が404になる
4. ブラウザのキャッシュ影響を避けるため、ビルド成果物のファイル名にハッシュが含まれていること（Viteのデフォルト挙動）を確認する
```

**注意（REL-POL-003の実務化）**: Local開発用に一度ビルドした`dist/`を誤って本番へ配置しないこと。`VITE_API_BASE_URL`がLocalの値のまま焼き込まれた成果物を配置すると、SPAは本番APIへ接続できず画面が真っ白または全API失敗になる。同様に`VITE_GOOGLE_MAPS_API_KEY`がLocal用（リファラを`localhost`に限定したキー）のまま焼き込まれると、本番ドメインからの呼び出しがGoogle側でリファラ不一致により拒否され、地図パネルのみエラー表示になる（UI-005の隔離設計により報告機能自体は影響を受けないが、意図しない障害のため事前確認を優先する）。

# 8. Google Maps APIキー設定確認手順

Project 01〜03には存在しない、外部APIをフロントエンドから直接呼び出す本プロジェクト固有の手順である。

```
1. Google Cloud Consoleで本番用APIキーが発行されていることを確認する（Local用キーとの取り違え防止）
2. 当該キーのHTTPリファラ制限に本番ドメインが登録されていることを確認する（他ドメインは拒否される設定になっているか）
3. Maps JavaScript API / Geocoding API の両方が有効化されていることを確認する
4. 利用量アラート（月$200クレジットに対する閾値、例: 80%到達時）が設定されていることを確認する
5. 失敗した場合: リリースをブロックしない（REL-POL-004とは独立の判断 — 地図機能はUI-005により隔離されているため）。
   ただし本番稼働開始後、速やかに9章のSmoke Testで地図パネルの表示を確認し、
   問題があれば20_運用保守手順書の手順で追って対応する
```

# 9. リリース後確認（Smoke Test）

```
□ /api/health が200を返す（DB接続含む）
□ SPAのトップ（/login）が表示される
□ テストアカウント（Admin/部門管理者/一般社員）でログインできる
□ Adminが災害イベントを作成でき、対象従業員へ通知が作成される
□ 一般社員が通知経由でSCR-003へ直リンク遷移し、安全報告を送信できる
□ 部門管理者のダッシュボードに報告内容が反映される
□ 地図パネルが表示される（またはAPIキー未設定時はエラー表示のみに留まり、他機能に影響しない）
□ 通知一覧が表示される
□ Admin管理画面（部署/従業員/災害管理・全社ダッシュボード）にログインできる
□ ディープリンク（/disasters/{id}/report）への直接アクセスが正しく表示される（try_filesの実地確認）
```

# 10. ロールバック手順

```
1. docker compose down
2. 直前の安定版タグへ git checkout（backend）
3. 直前の安定版のビルド成果物（dist/）へ配信ディレクトリを戻す
4. docker compose up -d
5. Migrationをロールバックしたテーブルがある場合、RDSスナップショットから復元を検討する
   （安全報告・災害の記録データを含むため、Migrationのロールバックは原則
    新規カラムの追加のみに限定し、破壊的変更は避ける方針とする — Project 01〜03と同一方針）
6. Google Maps APIキーはロールバック対象外（バックエンドのMigrationとは独立して管理されるため、
   通常は変更不要）
```

# 13. まとめ

本書のリリース手順で最も注意すべきはREL-POL-002/003（FE/BE分離ビルドと環境ごとの再ビルド必須）であり、Blade構成のProject 01にはなかった「ビルド成果物の環境取り違え」という失敗モードに対応する。Google Maps APIキーの設定確認（8章）は本プロジェクト固有の手順だが、地図機能がUI-005により隔離されているため、単独の設定ミスがリリース全体をブロックしない設計になっている点はPMSのReverb隔離設計（REL-POL-004相当）と同じ思想である。S3を作成しないことの確認（REL-POL-004）は、13_インフラ設計の判断が実際のインフラ構築時にも一貫して守られていることのチェックポイントである。
