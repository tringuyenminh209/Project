■ 1. 本書の目的

Disaster Safety Report Systemの各画面について、表示項目、入力項目、操作、バリデーション、エラー、状態バリエーションを定義する。02_要件定義書 03_ユースケース 05_画面遷移図と整合し、10_API設計（作成後に8章へAPI対応を追記）および試験仕様書の基準とする。

■ 2. 画面設計方針
方針ID方針内容
UI-001報告系=TOUI / 管理系=OOUIのハイブリッド一般社員・部門管理者向けの報告系画面（SCR-002/003）は「安否を報告する」というタスクを最短で完了させるタスク指向UI（TOUI — ATM・申請フォーム型）とする。Admin向けの管理系画面（SCR-005〜008）は災害・部署・従業員という対象物を選んでから操作するオブジェクト指向UI（OOUI）とする。緊急時の報告者と平時の管理者では最適なUI原則が異なるため、1方式に統一しない。
UI-002Mobile First安全報告は緊急時に社員個人のスマートフォンから行われることを想定し、モバイル（375px基準）を最優先で最適化する（NFR-020。Project 02と同方針、Project 03のPC Firstとは異なる）。ダッシュボード・Admin管理画面はlg（1024px）以上で情報密度を上げるが、モバイルでも全機能が操作可能であること — 災害時に部門管理者・AdminがPCの前にいる保証はない。
UI-003報告フォームの最小化SCR-003は必須入力を安否ステータス1項目のみとし、大ボタン2択（安全 / 要支援）+任意項目（コメント・位置情報）の構成とする（NFR-021: 数タップで送信完了）。
UI-004権限による表示制御はUIの配慮、防御はAPIAdmin専用メニューは一般社員・部門管理者に非表示にするが、権限の実体は常にAPI側の判定（BR-PRM）。UIの出し分けをセキュリティと混同しない。
UI-005外部API障害の隔離表示Google Maps関連（地図パネル・ジオコーディング）の障害は該当パネルのみのエラー表示に留め、集計数値・報告送信という本質機能のUIは通常表示を継続する（NFR-007）。
UI-006Traceability各画面はSCR・UC・REQ・エラーIDと対応させる。
UI-007位置情報のない報告を地図に描画しない（v1.1、設計監査で追加）安否報告に緯度経度が伴わない場合（ジオコーディング/GPS失敗、NFR-007で受理自体は継続）、その報告をlocationsマスタの通常勤務地へフォールバックして地図にピン表示することはしない。needs_help報告が実際とは異なる場所（オフィス）に表示されると、BCP担当者・救助側の初動判断を誤らせる危険がある（10_API設計 6.7節）。該当報告は地図とは別のテキストリストに「位置情報：未確定」として表示する（5.4/5.5/5.11節）。locationsマスタへのフォールバックは、そもそも報告自体が存在しない未報告者（status=unreported）にのみ許容する — 「本来いるはずの場所」を示す参考情報として意味を持つため。

5.0 状態バリエーション 共通ルール

EC Site / PMS 06_画面設計 5.0節と同一の書式を初版から全画面へ適用する。

共通ルール内容
Loading（取得系）スケルトン（Border色の枠 + Background色のバー）で件数分プレースホルダー表示
Loading（送信系）送信ボタンはButtonトークンのloading状態（二重送信防止）。応答までフォーム非活性

5.3 SCR-003 安全報告フォーム

本システムのTOUI核心画面（UI-003、NFR-021）。

入力項目

  • 安否ステータス（Required・唯一の必須項目）: 「安全です」（Safety-safe色）/「支援が必要です」（Safety-needs-help色）の大型2択ボタン（Report-CTAトークン）
  • コメント（Optional / Max 500）
  • 位置情報（Optional）: 「現在地を取得」（GPS）または住所入力（Max 200 → Geocoding API変換）。取得結果を小型地図パネルでプレビュー表示

操作

  • ステータス選択 → 「送信する」→ 完了トースト（Success）→ SCR-002へ戻る
  • 再報告時（BR-RPT-002）: 前回の報告内容を初期値表示し、同じフォームで上書き送信（UC-009）

5.4 SCR-004 部署ダッシュボード（部門管理者）

表示項目

  • 進行中の災害セレクタ（複数進行中の場合の切替）
  • 自部署の集計サマリ: 報告済み（Safety-safe）/ 要支援（Safety-needs-help）/ 未報告（Safety-unreported）の人数（◯/◯人形式、NFR-002: 3秒以内）
  • 要支援者一覧（最優先表示 — 氏名、コメント、報告日時、位置情報有無）
  • 未報告者一覧（氏名のみ。電話等のシステム外フォローの判断材料 — BF-004）
  • 地図パネル（Map-panel）: 自部署の緯度経度を持つ報告をMap-pin-safe/needs-helpで色分けピン表示 + 未報告者の通常勤務地をSafety-unreported色（グレー）でピン表示 + 凡例（UI-007）

  • 手動更新ボタン + 一定間隔の自動再取得（ポーリング。間隔は12_詳細設計書で確定）
  • 災害セレクタ切替、SCR-011への遷移

■ 9. まとめ

全11画面をMobile First（UI-002）+ 報告系TOUI/管理系OOUIのハイブリッド（UI-001）で設計した。デザイントークンはEC Site/PMSの基本セットを踏襲しつつ、本システム固有の安否3状態（safe/needs_help/unreported）・災害状態・地図ピンのトークンを追加で確定した — 特にSafety-needs-helpに最も強い警告色を割り当て、ダッシュボード・地図・バッジの全てで同一色を使うことで「要支援者がどこにいるか」が画面のどこでも同じ色で追えるようにした。状態バリエーションは全画面初版から適用し、本システム固有の第4状態として「地図パネル障害の隔離表示」（UI-005）を共通ルール化した — 外部API（Google Maps）の障害が安否報告・集計という本質機能のUIを道連れにしないことを、画面仕様のレベルで保証する。v1.1では設計監査により、位置情報のない報告を地図にフォールバック表示することがBCPシステムでは誤誘導リスクになるという業務上重大な論点を発見し、UI-007として位置情報未確定リストを新設した（10_API設計と連動修正）。
