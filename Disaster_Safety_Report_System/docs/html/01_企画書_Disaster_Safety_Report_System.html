■ 1. 企画背景

Portfolio Roadmap（SE/BrSE 2026-2027）における10プロジェクト構想のうち、Project 04 として位置づけられる案件。Project 01（HR & Attendance System）で組織階層（部署・従業員）とCRUD・認証を、Project 02（EC Site）でトランザクション整合性を、Project 03（Project Management System）でSPA分離・通知fan-out・並行処理制御（Aggregate Root Locking）を習得した。次の段階として、GIS（地図・位置情報）と大量同報通知という、これまでの3プロジェクトで一度も扱っていない設計課題に取り組む。

防災・安否確認（BCP: Business Continuity Plan）は、地震・台風等の自然災害が頻発する日本において、一定規模以上の企業では法令・取引先要件（大企業の調達要件にBCP整備を含むケースがある）として整備が求められる領域である。BrSEとして日本企業と協業する際、この業務ドメインへの理解は直接的な会話の解像度向上につながる。Project 03の企画書（PMS-001 14章）でも「Project 04（災害安否確認の通知）」を次の布石として明記しており、本プロジェクトはその実現にあたる。

■ 2. 現状の課題
  • Project 01〜03はいずれも位置情報（緯度経度・住所）を扱った経験がなく、地図表示・ジオコーディング（住所→座標変換）・外部地図APIとの連携（APIキー管理、クライアント側直接呼び出し）を経験していない
  • Project 03の通知は「担当者1名への個別通知」（task_assigned等）のみであり、「災害発生時に対象の全従業員（数百〜数千人規模）へ一斉配信する」という同報（Broadcast）パターンを設計・実装した経験がない。件数増加時の性能劣化・Queue化の要否判断も未経験
  • 「部署別・全社の集計値をリアルタイムに把握する」という、書き込みより読み取り集計が支配的な画面（ダッシュボード）を設計した経験がない
  • Project 03（PMS）で「通知の重複防止フラグ設計」「並行処理時のAggregate Root Locking」を習得済みだが、これらの知見を新しいドメイン（安否報告の再報告、災害イベントの同時更新）へ転用できるかがまだ実証されていない
■ 3. 企画目的

上記の課題を解消するため、以下を満たす防災安全報告システムを自主開発する。

  1. 従業員が災害通知を受け取り、低い操作コストで自分の安否状況（安全 / 要支援）を報告できる（「未確認」は未報告の従業員をダッシュボードが表示する派生状態であり、報告時の選択肢ではない、02_要件定義書 BR-RPT-001）
  2. 部門管理者が自部署の報告状況をリアルタイムに把握し、未報告者への現場対応（電話等）に活用できる
  3. Admin（BCP担当）が災害イベントを作成・管理し、全社の状況を一望できる
  4. 報告内容には任意で位置情報を添えられ、地図上で視覚的に確認できる
  5. 大量の対象者へ同時に通知を配信しても、システムの応答性能が劣化しない設計とする
■ 4. 対象ユーザー

Admin（BCP担当・全社）、部門管理者（部署単位）、一般社員（個人）の3層。単一企業内システムであり、マルチテナントSaaSではない（確定事項、00_開発計画書 3章・9章）。権限マトリクスは 02_要件定義書 で確定する。

■ 6. 解決方針
課題解決方針
GIS未経験（地図・ジオコーディング）Google Maps API採用（00_開発計画書 6章のADR）。ジオコーディング失敗時は位置情報なしで報告を受理するフォールバックを設計し、地図機能の障害が本質機能（安否報告）を止めないようにする
大量同報通知未経験12_詳細設計書で対象人数の閾値を検証し、必要であればLaravel Queueによる非同期化を導入する。同期ループのままで許容できる範囲かをまず見積もる
リアルタイム集計ダッシュボード未経験集計クエリ（部署別・全社の報告状況カウント）を09_テーブル定義/12_詳細設計書で設計する。Project 03のようなWebSocketリアルタイム配信は本スコープでは採用せず、ポーリングまたは画面リロードで十分とする（緊急時の集計値が数秒遅延しても実害が小さいため — bonusでの高度化は13_今後の拡張予定に記載）
Project 03知見の新ドメインへの転用（重複防止フラグ・並行ロック）安否報告の再報告可否、災害イベントの同時更新（複数Adminが同時編集する場合）について、Project 03で得た「識別子ベースか状態ベースか」「Aggregate Root Locking」の視点を02_要件定義書・12_詳細設計書で明示的に検討する
外部APIキー管理未経験Google Maps APIキーにHTTPリファラ制限をかけ、.env経由で注入する（Project 02のStripeキー管理と同じ原則）。14_セキュリティ設計で方式を確定する
■ 7. システム概要

企業内の防災・安否確認システム。災害発生時にAdminが災害イベントを作成すると、対象従業員へ通知が一斉配信される。従業員は通知を受けて自分の安否状況を報告し、部門管理者・Adminはそれぞれの範囲でリアルタイムに近い集計状況を確認できる。位置情報を伴う報告は地図上に可視化される。詳細構成は 00_開発計画書 4章の構成図を参照。

■ 8. 主要機能

認証・組織マスタ管理・災害管理・安全報告・通知（同報配信）・ダッシュボード・地図表示の7系統。詳細は 00_開発計画書 5章および 07_機能一覧を参照。

■ 9. 開発範囲
9.1 開発対象
  • ログイン・ログアウト（Sanctum トークン認証）
  • 組織マスタ管理（Admin）: 部署・従業員CRUD
  • 災害管理（Admin）: 災害イベントの作成・進行中/収束切替・編集
  • 安全報告（一般社員）: 安否状況の報告（コメント・位置情報は任意）
  • 通知（同報配信）: 災害作成時の一斉通知、未報告者への催促バッチ
  • ダッシュボード: 部署別・全社の報告状況集計
  • 地図表示: 従業員位置情報・報告位置のGoogle Maps表示
9.2 開発対象外
  • マルチテナント／複数企業対応
  • 外部災害検知サービス（気象庁 緊急地震速報API等）との自動連携
  • 従業員↔Admin間の双方向チャット
  • オフライン地図キャッシュ・バックグラウンド位置同期（Bonus扱い、README.md確定済み）
  • ネイティブモバイルアプリ・多言語UI（日本語のみ、内部文書はベトナム語併記）

00_開発計画書 2.1/2.2節と同一。

■ 16. まとめ

本企画は、Project 01〜03で確立した「Laravelレイヤー構成 + 業務ルール駆動の設計プロセス」を、初のGIS統合と大量同報通知という新しい設計課題へ拡張する挑戦である。核心は「災害発生という緊急性の高い状況下で、大量の従業員へ確実かつ高速に情報を届け、その応答をリアルタイムに集計する」設計であり、この一連の流れを性能試験まで含めて検証することを、本プロジェクト最大の成果物とする。
