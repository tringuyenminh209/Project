■ 1. 本書の目的

本書はDisaster Safety Report Systemの単体試験（PHPUnit / Feature + Unit）を定義する。02_要件定義書 19章の受入条件である「8章の権限マトリクスの全セルが単体試験で網羅され、全件Passする」を8章で担保する。PMS（Project 03）の重点が「認可の網羅（80セル）と並び順・通知の整合」だったのに対し、本プロジェクトの重点は「認可の網羅（33セル、フラット3層ロールゆえPMSより小規模）」と「地図ピン・位置情報の正確性」である — 12_詳細設計書v1.1で発見・修正した地図ピンの誤誘導リスクは、そのまま本書の重点試験項目になる。

■ 2. 試験方針
方針ID方針内容
UT-POL-001Requirement BasedREQ / FUNC / API / BRに対応した試験を作成する。
UT-POL-002Normal / Abnormal / Boundary正常系・異常系に加え、文字数・件数の境界値（最小未満/最小/最大/最大超）を確認する。
UT-POL-003Matrix Exhaustive02_要件定義書 8章の権限マトリクス11操作×3ロール=33セル全部を試験する。〇セルの成功確認だけでなく×セル（E002）を必ず含める — 認可バグは拒否側でしか見つからない（14_セキュリティ設計 7.2節）。
UT-POL-004Database Assertion安全報告・通知・災害の状態はDB保存結果を直接確認する（レスポンスだけで判定しない）。
UT-POL-005Upsert Concurrency安全報告の初回送信が同時に2件発生した場合でも、atomic upsertにより最終的に1行のみ存在することを確認する（12_詳細設計書 5.6節v1.1）。
UT-POL-006Location Accuracy（本システム固有・重点）地図ピン（map_pins）は緯度経度を実際に持つ報告と未報告者の勤務地のみで構成され、位置情報を持たない報告済みデータが1件でもmap_pinsに混入していないことを明示的にassertする（06_画面設計 UI-007 / 10_API設計 6.7節v1.1で修正した重大リスクの再発防止）。
UT-POL-007外部API非依存バックエンドはGoogle Maps/Geocoding APIを一切呼び出さないため（10_API設計 6.5節）、試験での外部APIモック化は不要。フロントエンドのGeocoding連携試験は17_システム試験仕様書に委ねる。

■ 17. まとめ

本書の重心は2点である。①8章 — 権限マトリクス33セルの全数試験（フラット3層ロールのためPMSの80セルより小規模、02_要件定義書 3章の学習ポイントがそのまま試験規模に反映されている）。②14章 — 地図ピンの位置情報正確性試験（UT-POL-006）。これは12_詳細設計書v1.1で発見・修正した「位置情報のないneeds_help報告を通常勤務地へフォールバック表示する」重大リスクの再発防止を目的とした、本システム固有の最重要試験カテゴリであり、UT-DASH-005/006は実装完了後に真っ先にPassを確認すべき試験としてマークする。次点はUT-RPT-007（atomic upsertの並行処理試験）とUT-DIS-005/006・UT-BAT-005（設計監査で修正した性能課題の再発防止試験）である。
