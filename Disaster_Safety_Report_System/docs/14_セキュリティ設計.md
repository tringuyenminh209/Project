# 2. 脅威モデルと設計方針

## 2.1 攻撃面の特徴

11_基本設計書 10章のとおり、本システムのバックエンドは外部API（Google Maps/Geocoding含む）を一切呼び出さない。また従業員アカウントはAdminのみが作成でき（REQ-005）、**公開の会員登録機能が存在しない** — PMS/EC_Siteと異なり、登録フローに起因するアカウント列挙リスクはそもそも発生しない。したがって攻撃面は「認証済みユーザーによる越権（縦方向・部署スコープ方向）」と「位置情報という機微データの取扱い」の2点に集中する。

| 脅威 | 具体例 | 対策章 |
| --- | --- | --- |
| 縦方向の越権 | 一般社員がAdmin専用操作（部署/従業員/災害の作成・編集）を直接APIコール | 7章 |
| 部署スコープの越権 | 部門管理者が他部署のダッシュボード・安全報告情報を取得しようとする | 7章 |
| 位置情報の不適切な開示 | 一般社員が同僚の被災時位置情報を閲覧できてしまう | 6章 |
| トークン窃取 | XSSによるlocalStorage読み出し、盗聴 | 4.2節 / 5章 |
| アカウント推測 | ログイン総当たり（Admin作成アカウントのみが対象。公開登録なし） | 4.3節 / 4.6節 |

## 2.2 設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| SEC-POL-001 | 認可はサーバーが唯一の判定者 | フロントの表示制御（G-01〜07、05_画面遷移図）は**利便性のためであり防御ではない**。全APIがサーバー側で2段判定（3章）を通る。 |
| SEC-POL-002 | 部署スコープは「入力経路の不存在」で守る | 部門管理者が他部署へアクセスする経路は、判定ロジックではなくAPI設計自体（部署IDをパラメータとして受け取らない、API-POL-004）で塞ぐ。PMSの「存在秘匿（E007）」とは異なるアプローチだが、目的（部署スコープの越権防止）は同じ。 |
| SEC-POL-003 | シークレットの非露出 | DBパスワードをレスポンス・ビルド成果物・Gitに含めない（13_インフラ設計 14章）。Google Maps APIキーは性質上「秘匿」できないため、リファラ制限・利用量アラートという別の防御に委ねる（13_インフラ設計 INF-POL-006）。 |
| SEC-POL-004 | 判定の一元化 | 認可判定はPolicyまたはMiddlewareに一元化する（NFR-011）。ControllerやVueに権限ロジックを分散させない（12_詳細設計書 8章）。 |
| SEC-POL-005 | 位置情報は必要最小限にしか露出しない | 緯度経度・住所という機微データは、業務上その情報を必要とするロール（部門管理者・Admin、自部署/全社スコープ内）にのみ露出する。ログ・監査記録にも生の位置情報は含めない（10章）。 |

# 3. 認証・認可の全体像

全APIリクエストは以下の順で検査される（10_API設計 4章の共通前処理と同一。本書はセキュリティ観点の裏付けを与える）。

| 段 | 検査 | 失敗時 | 実装 |
| --- | --- | --- | --- |
| 1 | 認証: 有効なSanctumトークンか | E010 (401) | `auth:sanctum`ミドルウェア |
| 2 | ロール: 操作に必要なロール（admin/manager/staff）を満たすか | E002 (403) | `role:admin`/`role:manager`ミドルウェア（大半のAPI） |
| 3 | 所有関係: 安全報告・通知の対象が自分（または自部署）のものか | E002 (403) / E007 (404) | SafetyReportPolicy / NotificationPolicy（12_詳細設計書 8章） |

権限の正は02_要件定義書 8章の権限マトリクスであり、本書はこれを実装レイヤーに割り付ける文書間の結節点である。PMSの4段（認証→メンバーシップ→ロール→状態）と異なり3段で完結するのは、本システムにメンバーシップという中間層が存在しない（フラット3層ロール、BR-PRM-001）ためである。

# 4. 認証設計

## 4.1 認証方式

Laravel Sanctumの**Personal Access Token（Bearerトークン）方式**を採用する（NFR-010）。SPAは`Authorization: Bearer {token}`ヘッダを全APIリクエストに付与する。

| 観点 | Bearerトークン（採用） | SPA Cookieモード（不採用） |
| --- | --- | --- |
| トークンライフサイクル | 発行・8時間期限・ログアウト失効を明示的に実装 — Project 03（PMS）から続く「SPA向けトークン認証」の学習目標 | セッションに隠蔽され学習効果が薄い |
| CSRF | Cookieを認証に使わないため**CSRF攻撃自体が成立しない**（8.4節） | CSRFトークン往復の実装が必要 |
| XSS時のトークン窃取 | 弱い（→4.2節の緩和策で補う） | 強い（httpOnlyで読めない） |

XSS耐性ではCookieモードが勝るが、その差分は4.2節の緩和策で受け止め、Project 01〜03との一貫性を優先する。

## 4.2 トークン保管方式の確定 ★論点①

**localStorageに保管する**（12_詳細設計書 9.2節 authストアの前提を正式確定）。

**リスクの直視**: localStorageはJavaScriptから読めるため、XSSが1箇所でも成立するとトークンが窃取される。これを受け入れる代わりに、以下の緩和策を**セットで必須**とする。

| # | 緩和策 | 内容 |
| --- | --- | --- |
| 1 | XSSの根本遮断 | Vueのテキスト補間のみ使用。**`v-html`の使用を原則禁止**（安全報告のコメント欄が主な自由記述入力、8.3節） |
| 2 | CSP | `script-src`を原則`'self'`とするContent-Security-Policyヘッダ（5.2節）。ただしGoogle Maps SDKの読み込みのため`maps.googleapis.com`を明示的に信頼リストへ追加する（本システム固有の緩和 — 5.2節で詳述） |
| 3 | 有効期限8時間 | 窃取されても当日限りで失効（NFR-010） |
| 4 | ログアウトで即失効 | ログアウトAPIがサーバー側のトークンレコードを削除 — クライアント側の破棄だけに頼らない |
| 5 | HTTPS全面化 | 経路上の盗聴を遮断（NFR-016、5.1節） |

タブを閉じても再ログイン不要というUX（05_画面遷移図 G-01の前提。通知タップ→ログイン→SCR-003復帰、NFR-021）はlocalStorageだから成立する。sessionStorage（タブごと消失）は不採用。

## 4.3 ログイン失敗時の応答設計

06_画面設計 7章「E001はどちらの項目が誤りかを明示しない」の設計根拠を定義する。

| ルール | 内容 |
| --- | --- |
| メッセージ統一 | メール不存在・パスワード不一致・アカウント無効の3ケースすべてで同一のE001「メールアドレスまたはパスワードが正しくありません。」を返す（10_API設計 6.1節） |
| 理由 | ケースを区別すると「このメールは登録済みか」を外部から判定できてしまう（アカウント列挙攻撃）。本システムは公開登録がなくAdmin作成アカウントのみのため、対象は限定されるが原則は同様に適用する |
| レスポンス時間 | メール不存在時もパスワードハッシュ照合相当の処理時間を確保する（`Hash::check`をダミー実行）— タイミング差による列挙を防ぐ |

**PMS/EC_Siteとの違い**: 公開の会員登録APIが存在しないため、「登録フローでのメール列挙は許容トレードオフ」という論点自体が本システムには発生しない — 4.6節のレートリミットが対象とするのはログインのみである。

## 4.4 トークンライフサイクル

| イベント | 挙動 |
| --- | --- |
| ログイン成功 | 既存トークンは失効させず新規発行（複数デバイス許容）。有効期限8時間 |
| 期限切れ / 失効後のAPIコール | E010 (401) → SPAはauthストアを破棄しSCR-001へ（G-01、12_詳細設計書 9.3節のAPI Client共通処理） |
| ログアウト | 使用中トークンのみサーバー側で削除（他デバイスのトークンは維持） |
| パスワード変更（API-004） | 現在のパスワード確認のうえ変更（不一致はE003）。**成功時は使用中トークン以外をすべて失効**（窃取済みトークンを持つ攻撃者を締め出す標準動作） |
| Admin無効化（API-012、REQ-005 / BR-ORG-003） | **既発行トークンを全件即時失効させる**（PMSの同種の確定と同一方針）。無効化は「今すぐアクセスを止める」操作であるべきで、有効期限内のトークンが生き残ると意図と矛盾する。実装は`employees.status`更新と同一トランザクションで対象従業員の`personal_access_tokens`を削除する（12_詳細設計書 EmployeeService相当）。再有効化しても旧トークンは復活せず、再ログインが必要 |

## 4.5 パスワードポリシー

| 項目 | 内容 |
| --- | --- |
| 保存 | bcryptハッシュ（Laravelデフォルト）。平文・可逆暗号は禁止 |
| 文字数 | 8〜20文字（02_要件定義書 12章） |
| ログへの非出力 | パスワードはいかなるログにも出力しない（10章） |

## 4.6 レートリミット

| 対象 | 制限 | 実装 |
| --- | --- | --- |
| ログイン | 同一IP+メールで5回/分。超過時は429 | Laravel `throttle`ミドルウェア |
| 一般API | 60回/分（暴走クライアント対策の粗い上限） | `throttle:api` |

公開登録がないため、PMS/EC_Siteにあった「会員登録の連打対策」は本システムには存在しない。

# 5. 通信セキュリティ

## 5.1 HTTPS

全通信をHTTPSとする（NFR-016）。HTTP(80)はHTTPSへリダイレクトのみ。

## 5.2 セキュリティヘッダ

Nginxで付与する（13_インフラ設計 8章）。

| ヘッダ | 値 | 目的 |
| --- | --- | --- |
| Content-Security-Policy | `default-src 'self'; script-src 'self' https://maps.googleapis.com; connect-src 'self' https://maps.googleapis.com https://maps.google.com; img-src 'self' data: https://maps.gstatic.com https://maps.googleapis.com; style-src 'self' 'unsafe-inline'` | XSS緩和の中核（4.2節の前提） |
| X-Content-Type-Options | `nosniff` | MIMEスニッフィング防止 |
| X-Frame-Options | `DENY` | クリックジャッキング防止 |
| Strict-Transport-Security | `max-age=31536000` | HTTPS強制 |
| Referrer-Policy | `strict-origin-when-cross-origin` | URL情報の漏洩抑制 |

**PMS/EC_Siteとの違い（重要なトレードオフ）**: PMSは外部フロントエンド依存が一切なく`script-src 'self'`のみで完結できたが、本システムはGoogle Maps JavaScript SDKをブラウザから読み込むため、`script-src`に`maps.googleapis.com`を追加する必要がある。これはCSPが信頼するスクリプト提供元を1つ増やすことを意味し、XSS防御力をわずかに弱める。地図機能自体が本プロジェクトの学習目標（00_開発計画書 1章）であるため、このトレードオフは許容する — ただし信頼先はGoogleの正規ドメインに限定し、ワイルドカードや`unsafe-inline`をscript-srcに追加することはしない。

`style-src 'unsafe-inline'`はVueコンポーネントの動的style（地図パネルのレイアウト調整等）のために許容する。

## 5.3 CORS

Nginxが同一オリジンでSPAとAPIを配信する（13_インフラ設計 8章）ため、本番ではクロスオリジンリクエストは本来発生しない。それでも`config/cors.php`で許可オリジンをSPA配信オリジン**のみ**に明示設定する（NFR-013）— ローカル開発（Vite dev server :5173 → API）ではクロスオリジンになるため、環境変数`CORS_ALLOWED_ORIGINS`で環境ごとに切り替える（13_インフラ設計 14.1節）。`*`（全許可）は全環境で禁止。

# 6. 位置情報プライバシー設計 ★論点②（本システム固有）

02_要件定義書 15章が持ち越していた「位置情報という個人のプライバシーに関わるデータの取扱い方針（アクセスできる者の範囲、保持期間）」を確定する。

## 6.1 アクセスできる者の範囲

02_要件定義書 8章の権限マトリクスがそのまま正である。位置情報固有の観点として改めて明記する:

| ロール | 自分の位置情報 | 他人の位置情報 |
| --- | --- | --- |
| 一般社員 | 入力・自分の報告確認のみ（自分の位置情報を一覧・地図で見返す専用画面はない） | 閲覧不可 |
| 部門管理者 | 同上（自分も一従業員として報告） | 自部署の従業員のみ（SafetyReportPolicy::viewList、BR-PRM-002） |
| Admin | 報告対象外（BR-PRM-003） | 全社（SafetyReportPolicy::viewList） |

実装は12_詳細設計書 8章のSafetyReportPolicyに一元化する — 部署スコープを超えた位置情報が誤ってレスポンスに含まれることを、Service層の抽出条件（5.5節）とPolicyの二重チェックで防ぐ。

## 6.2 保存期間・削除方針の確定

**確定**: 位置情報（`safety_reports.latitude`/`longitude`/`address_text`）に対する個別の削除機能・保持期限は設けない。`safety_reports`自体が物理削除されない設計（09_テーブル定義 TBL-POL-003）であるため、位置情報もその報告の生存期間に付随する。

**理由**: 安全報告はBCP（事業継続計画）における災害対応記録であり、後日の対応検証・監査のために「いつ・誰が・どこで」報告したかを保持する業務上の必要性がある（02_要件定義書 11.2節データ保持方針、`disasters`が収束後も保持される理由と同根）。個人のプライバシーへの配慮と業務記録としての保持要求はトレードオフ関係にある。

**明記すべき限界（本ポートフォリオのスコープ）**: 実運用へ移行する場合、日本の個人情報保護法（APPI）に基づき、位置情報のような要配慮性の高いデータの保持期間・利用目的の通知・本人からの削除請求への対応方針を法務確認のうえ別途定める必要がある。本書はその法務レビューを代替するものではなく、学習用ポートフォリオとしての現実的なデフォルト（「削除機能を持たない」という09_テーブル定義の設計方針をそのまま継承する）を明記するに留める（13章「今後の拡張予定」に将来課題として記録）。

## 6.3 位置情報の伝送経路

ジオコーディング（住所→緯度経度変換）はブラウザからGoogle Geocoding APIへ直接送信される（00_開発計画書 4章）。これは、報告者が入力した住所文字列がGoogle（第三者）へ送信されることを意味する — Google Maps APIを採用した時点で内在するトレードオフであり、00_開発計画書 6章のADR相当の記述に既に含意されている。バックエンド・DBへの伝送は5.1節のHTTPS全面化で保護する。

## 6.4 地図表示の最小化（06_画面設計 UI-007との関連）

06_画面設計 v1.1で確定したUI-007（緯度経度のない報告を地図にフォールバック表示しない）は、地図の正確性だけでなくプライバシーの観点でも意味を持つ: 実際の座標を持たない従業員について「通常の勤務地にいるはず」という推測を地図上に断定的に表示しないことは、その人の実際の所在に関する誤った情報を作り出さないという点で、位置情報の取扱いにおける最小限主義（SEC-POL-005）とも整合する。

# 7. 認可設計（部署スコープ対策） ★論点③

本プロジェクトの学習の核心の1つ（02_要件定義書 19章の受入条件: 権限マトリクス全セルの試験網羅）。実装の正は12_詳細設計書 8章 — 本書は攻撃シナリオ側から検証する。

## 7.1 攻撃シナリオと遮断ポイント

| 攻撃シナリオ | 遮断ポイント | 応答 |
| --- | --- | --- |
| 一般社員がAdmin専用API（部署/従業員/災害の作成・編集）を直接コール | `role:admin`ミドルウェア | E002 (403) |
| 部門管理者が他部署のダッシュボード情報を得ようとする | そもそもAPIが部署IDをパラメータとして受け取らない設計（API-POL-004）— 攻撃の入力経路自体が存在しない | 常に自部署の結果のみ返る |
| 一般社員が他人の安全報告詳細を取得しようとする | SafetyReportPolicy::viewOwn（自分以外は拒否） | E002 (403) |
| 部門管理者が他部署の安全報告一覧を取得しようとする | SafetyReportPolicy::viewList + Service層の抽出条件が自部署に固定（6.1節） | 自部署の結果のみ返る（他部署のデータは応答に一切現れない） |
| 認証済みユーザーが他人の通知IDを既読化しようとする | NotificationPolicy::view/read | E007 (404、BR-NTF-005) |
| Adminが安全報告を提出しようとする | SafetyReportPolicy::upsertOwn（BR-PRM-003） | E002 (403) |

## 7.2 検証の網羅（試験への引き渡し）

02_要件定義書 8章の権限マトリクス（Admin/部門管理者/一般社員 × 11操作）の**全セル**を15_単体試験仕様書の認可テストに1:1で落とす。○のセル（成功）だけでなく、**×のセル（E002が返ること）・秘のセル（自部署以外が結果に現れないこと）の検証を含めて**02_要件定義書 19章の受入条件を満たす。

# 8. 入力検証・インジェクション対策

| # | 脅威 | 対策 |
| --- | --- | --- |
| 8.1 | SQLインジェクション | 全DBアクセスをEloquent/クエリビルダ経由（プレースホルダ自動適用）。生SQLの文字列連結は禁止 |

# 13. まとめ

本書で持ち越し論点2件（および13_インフラ設計と合わせて4件）を確定した。①トークンはlocalStorage保管とし、その弱点（XSS窃取）は「CSP + `v-html`禁止 + 8時間期限 + サーバー側失効」のセット運用で受け止める。②位置情報の取扱いは、アクセス範囲を8章権限マトリクスのまま維持し、保持期間は`safety_reports`の物理削除なし方針（09_テーブル定義）に従う — ただし実運用移行時には個人情報保護法に基づく法務確認が別途必要であることを明記した。攻撃面が「認証済みユーザーの部署スコープ越権」と「位置情報という機微データの取扱い」に集中する本システムでは、7章の部署スコープ対策（判定ロジックではなく入力経路の不存在）と6章の位置情報プライバシー設計が、PMS/EC_Siteにはなかった本システム固有の防御の核心である。
