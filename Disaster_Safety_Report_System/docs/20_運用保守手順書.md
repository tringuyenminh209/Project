# 1. 本書の目的

本書はDisaster Safety Report Systemの日常運用・障害対応の手順を定義する。PMS（Project 03）が権限（IDOR）・カンバン整合・S3孤児・Reverbを重点としたのに対し、本プロジェクトは**部署スコープの整合・安全報告の同時送信整合性・Google Maps API利用量・催促バッチの稼働**を重点的に定義する。設計フェーズを通じて本書へ委ねられていた項目（13_インフラ設計の利用量監視、14_セキュリティ設計のAudit調査、19_リリース手順書のロールバック後始末）はすべて本書に集約する。

```
□ /api/health 死活確認

```

# 6. 権限・部署スコープ関連の障害対応（重点）

## 6.1 部署スコープ越境の疑い（他部署のデータが見えた/操作できたとの報告）

```
1. 報告された画面・APIエンドポイントを特定する
2. 当該APIが `role:manager` ミドルウェア配下にあり、かつ部署IDをパラメータとして
   受け取らない設計（API-POL-004）になっているか確認する
   → 新規APIで部署IDをパラメータとして受け取る実装になっている場合、
     設計方針からの逸脱がバグの直接原因である可能性が高い
3. auditログで実際に発行されたリクエストのemployee_id・所属department_idを確認する
4. 15_単体試験仕様書 8章の該当セル（UT-PRM-xx）を再実行し、Fail箇所を特定する
5. 修正後、8章の権限マトリクス全セル（33セル）を再実行してから緊急リリースする
   （部分修正が他のセルを壊していないことの確認 — 権限まわりの修正は必ず全数再試験、
   部分試験で済ませない）
```

## 6.2 Admin無効化後もログインできてしまう（14_セキュリティ設計 4.4節の実装漏れ疑い）

```
1. 該当従業員のpersonal_access_tokensが無効化操作時に削除されているか確認する
2. 削除されていない場合、EmployeeService（Admin無効化処理）がトークン削除を欠落させている
   実装バグとして扱う（14_セキュリティ設計 4.4節で確定済みの仕様 — 「無効化=即時失効」からの逸脱）
3. 応急対応として該当従業員のトークンを手動でDBから削除する
4. 16_結合試験仕様書 IT-SCN-004を再実行してから修正をリリースする
```

## 6.3 Adminが安全報告を提出できてしまう（BR-PRM-003違反）

```
1. SafetyReportPolicy::upsertOwnがAdminロールを正しく拒否しているか確認する
2. safety_reportsにAdminのemployee_idを持つ行が誤って作成されていないか確認する
   （発生していた場合はデータ不整合として手動で削除を検討し、原因コードを最優先で修正する）
3. 15_単体試験仕様書 UT-PRM-X03を再実行してから修正をリリースする
```

# 7. 安全報告・通知関連の障害対応（重点）

## 7.1 同一災害・同一従業員の安全報告が重複して存在する

```
1. safety_reportsを (disaster_id, employee_id) でGROUP BYし、件数2以上の組み合わせを抽出する
   （UNIQUE制約とatomic upsertが機能していれば理論上発生しない — 12_詳細設計書 5.6節）
2. 発生していた場合、最新のreported_atの行を正として残し、古い行を削除する
   （安全報告は「最新の自己申告」を保持すればよい設計のため、BR-RPT-002）
3. 原因がUNIQUE制約自体の欠落（Migration適用漏れ等）であれば、09_テーブル定義の制約を
   再適用してから15_単体試験仕様書 UT-RPT-007を再実行する
```

## 7.2 通知が来ない・地図に誤って表示される

```
1. 「来ない」場合:
   a. NotificationService::notify()の呼び出し自体がアプリログに到達しているか確認する
      （12_詳細設計書 5.8節）
   b. 対象従業員が対象範囲（target_scope）の判定条件から漏れていないか確認する
      （BR-DIS-001の抽出ロジック、5.2-4節）
2. 「地図に誤って表示される」場合（最優先対応、OPS-POL-003）:
   a. 該当報告のlatitude/longitudeがNULLであるにもかかわらずmap_pinsに含まれていないか、
      DashboardServiceのレスポンスを直接確認する
   b. 該当していた場合、10_API設計 6.7節・12_詳細設計書 5.4節の分離ロジック（map_pins/
      unlocated_reports）の実装が仕様から逸脱している重大バグとして最優先で修正する
   c. 修正後、15_単体試験仕様書 UT-DASH-005/006・16_結合試験仕様書 IT-MAP-005を
      再実行してからリリースする（本システムで最も再発を防ぐべき不具合）
```

## 7.3 催促通知が来ない・過剰に来る

```
1. 「来ない」場合: 対象従業員が本当に未報告か、対象範囲の判定条件（BR-DIS-001）から
   漏れていないかを確認する
2. 「過剰に来る」場合: notifications の (employee_id, disaster_id, type, created_at) を
   時系列で確認し、4時間の時間窓（BR-NTF-004）が正しく機能しているか確認する
   （SendReportRemindersのバッチ間隔設定ミス、または時間窓クエリの実装ミスを疑う）
```

# 8. Google Maps/Geocoding API関連の運用

00_開発計画書 11章・13_インフラ設計 INF-POL-006が本節へ委ねた運用手順。

```
1. 利用量アラート到達時、Google Cloud Consoleで直近の呼び出し件数の内訳（Maps JavaScript API /
   Geocoding API）を確認する
2. 急激な増加が見られる場合、意図しない大量呼び出し（例: フロントエンドのバグによる
   Geocodingの多重発火）がないかフロントエンドのネットワークログで確認する
3. リファラ制限外からの利用試行がないか（不正利用の兆候）を確認する
4. APIキー自体は「秘匿」できない前提（14_セキュリティ設計 INF-POL-006）のため、
   不正利用が疑われる場合はキーのローテーション（新規発行 + リファラ再設定 + 旧キー無効化）を
   実施し、19_リリース手順書の手順でフロントエンドを再ビルド・再デプロイする
5. 障害時（Google側の障害等）: NFR-007により報告受理・集計表示は継続するため、
   復旧を急ぐ必要はない。地図パネルのエラー表示（UI-005）が正しく機能しているかのみ確認する
```

# 12. 変更管理・エスカレーション

個人開発プロジェクトのため、変更管理・承認フローは簡略化する。ただし**権限・部署スコープに関わる変更**（Policy・Middleware・Service層の抽出条件）は、必ず15_単体試験仕様書 8章の権限マトリクス33セル**全数**を再実行してからリリースする（6.1節と同一方針 — 部分試験での見切りリリースを禁止する）。同様に**地図ピン・位置情報の表示ロジックに関わる変更**は、必ずUT-DASH-005/006・IT-MAP-005を再実行してからリリースする（7.2節と同一方針）。

# 14. まとめ

本書はPMS（権限・カンバン整合・S3孤児・bonus隔離を重点）と対照的に、**部署スコープの整合・安全報告の同時送信整合性・地図の位置情報正確性・Google Maps API利用量**の4点を運用の重点とした。7.2節の障害対応は本システムで最も再発を防ぐべき不具合（位置情報のない報告の誤表示）を筆頭に据えており、応急対応で終わらせず必ず該当する単体/結合試験（UT-DASH-005/006・IT-MAP-005）を再実行してからリリースする方針を徹底する。これにより21文書構成（DSR-00〜20）の設計フェーズを完了する。
