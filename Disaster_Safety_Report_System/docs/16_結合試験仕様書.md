# 1. 本書の目的

単体試験（15_単体試験仕様書）はSQLite in-memoryで業務ルールを高速に検証した。バックエンドが外部APIを一切呼び出さない本システムでは、単体試験がfakeで代替した境界は「実MySQL」のみに限定される — PMSのようなS3・Reverbのfake置換は存在しない。代わりに本書は**フロントエンドが実際にGoogle Maps/Geocoding APIを呼び出す境界**を重点的に扱う。特にIT-MAP-005（位置情報未確定リストの実描画確認）は、12_詳細設計書v1.1で修正した重大リスクをAPI応答レベルだけでなく実際のブラウザ画面で再確認する、本書で最も重要な試験である。

| IT-POL-005 | **地図描画の実ブラウザ確認（本システム最重要）** | 15章のUT-DASH-005/006はAPI応答のJSON構造レベルの検証だった。本書ではそれに加え、**実際にブラウザで地図を開き、位置情報のない報告が目視でも地図上に一切現れず、位置情報未確定リストに正しく表示されること**を確認する — API応答が正しくてもフロントエンドの描画ロジックにバグがあれば同じ誤誘導リスクが再発するため、両レベルでの確認を必須とする。 |

# 6. API ↔ DB（実MySQL）結合試験

15章でSQLite上の論理を確認済み — ここではMySQL固有の物理挙動を確認する。

| Test ID | シナリオ | 確認内容 |
| --- | --- | --- |
| IT-DB-001 | atomic upsertの実動作 | 2クライアントから同一`(disaster_id, employee_id)`への初回報告を同時送信→MySQLの`INSERT ... ON DUPLICATE KEY UPDATE`で処理され、`safety_reports`に重複行が生成されない（15 UT-RPT-007のMySQL実物再確認） |
| IT-DB-002 | JSON列の実挙動 | `target_scope=specific`で災害を作成→`disasters.target_department_ids`が実際にJSON型で保存され、Eloquentの配列castで正しく読み出せる | 
| IT-DB-003 | 部署実在性チェックの実DB問い合わせ | 存在しない部署IDを含むリクエスト→DisasterServiceが実際にMySQLへ問い合わせてE003を返す（Fakeモックではなく実クエリ） |
| IT-DB-004 | bulk insertの実動作 | 対象従業員50名規模で災害を作成→実MySQLへのINSERT系クエリが定数回で完結すること（クエリログで確認、15 UT-DIS-006のMySQL実物再確認） |
| IT-DB-005 | トランザクション境界 | 災害作成処理の途中でDB接続を強制切断→`disasters`・`notifications`のどちらも残らない（11_基本設計書 14.2節の境界表） |
| IT-DB-006 | ロック不要設計の確認 | 安全報告の同時送信試験（IT-DB-001）実行中、MySQLの`SHOW ENGINE INNODB STATUS`でデッドロック・長時間ロック待ちが発生していないことを確認（12_詳細設計書 DD-POL-006の実証 — PMSのFOR UPDATE直列化試験に相当する項目自体が本システムには存在しない） |

# 13. まとめ

本書は「単体で正しいものが、実物の境界をまたいでも正しいか」を確認する。本システムはPMSと異なりS3・WebSocketの実物結合試験を持たないため、実質的な重心は2つ — ①実MySQLでのatomic upsert・JSON列・bulk insertの実挙動（6章）、②フロントエンドが実際にGoogle Maps/Geocoding APIと通信する境界（7章）である。特にIT-MAP-005は、12_詳細設計書v1.1で発見・修正した「位置情報のない報告の誤った地図表示」というリスクを、API応答のJSON構造だけでなく実際にブラウザで目視確認する試験であり、本システムの試験計画全体の中で最も優先度が高い。IT-ENV-001（`try_files`ディープリンク）は本番ビルド配信でしか検証できないため、dev serverの動作確認で代用しないこと。
