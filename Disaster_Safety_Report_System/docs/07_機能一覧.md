# 1. 本書の目的

本書は、Disaster Safety Report Systemの全機能をFUNC-IDで一意に採番し、要件（REQ）・ユースケース（UC）・業務フロー（BF）・画面（SCR）との対応を確定する。以降の10_API設計（API-ID⇔FUNC-ID対応）および試験仕様書の基準とする。

**FUNC-018について**: 特定の画面・単一ユースケースに紐づかない横断的機能である。認証ガード（INC-001）・Adminロール確認（INC-002）・部署スコープ確認（INC-003）として全APIに適用され、実装上はLaravel Policy + Middlewareに集約される（12_詳細設計書）。Project 03のFUNC-032（2層ロールモデル）と異なり、本システムはフラットな3層ロール1本で判定が完結する（02_要件定義書 3章・BR-PRM-001）。

**FUNC-013について**: 通知作成はユーザーから直接呼ばれる機能ではなく、災害イベント作成（FUNC-006）の副作用としてSystemが同一トランザクションで実行する。対象人数が多いため性能要件（NFR-001）が独立して定義される点が、Project 03のFUNC-027（通知作成）との最大の違いである。

# 9. まとめ

全19機能を7分類に整理した。Project 03（PMS）が33機能・9分類だったのに対し本システムが少ないのは、DSRが「単一企業・フラット3層ロール」というシンプルな権限モデルを採用した結果であり（02_要件定義書 3章の確定判断）、機能数の少なさそのものが設計判断の直接的な帰結である。一方でFUNC-006/013（災害作成+同報通知）とFUNC-009/010（安全報告・再報告）は、他プロジェクトにない新規性（GIS・大量同報通知）を担う中核機能であり、12_詳細設計書・17_システム試験仕様書で最も設計密度が高くなる部分として引き継ぐ。
