本書は08_ER図の論理設計を、MySQL 8上で実行可能な物理テーブル定義に落とし込む。本書の型・制約定義がそのままLaravel Migrationの実装根拠となる。

# 2. テーブル設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| TBL-POL-001 | 全テーブルにタイムスタンプ | `created_at`（+更新がありうるテーブルは`updated_at`）を必須とする。例外1件: `notifications`は状態変化（既読化）を`read_at`が記録するため`updated_at`を持たない（PMSと同一方針）。 |
| TBL-POL-002 | 外部キーは物理制約で強制 | Laravel側のリレーション定義だけでなく、DBのFOREIGN KEY制約でも整合性を強制する（Project 01〜03と同一方針）。 |
| TBL-POL-003 | 物理削除を伴う操作を持たない | 部署・従業員は無効化（status）のみ、災害は収束切替（status）のみ、安全報告はUPDATEのみ、通知は削除機能なし。Project 03（PMS）はタスク削除時の連動物理削除（BR-TSK-007）を持っていたが、本システムには相当する操作が1つもない — 本システム全体の特徴として明記する（7章参照）。 |
| TBL-POL-004 | 対象範囲はJSON列 + アプリ層検証 | `disasters.target_department_ids`はJSON型とし、内包する部署IDの実在性チェックはDBのFK制約では表現できないためアプリ層のバリデーションに委譲する。理由とトレードオフは11章に記録する。 |
| TBL-POL-005 | DB制約とアプリ層の役割分担の明示 | UNIQUE・CHECK・FKで守れるものはDBで守る。DBで表現できない2件（`target_department_ids`内の部署ID整合性、`companies`の行数1件制限）は理由と代替保証をセットで11章に記録する（PMSのTBL-POL-005と同じ哲学の再適用）。 |

# 5. 外部キー制約

| 子テーブル | 子カラム | 親テーブル | 親カラム | ON DELETE |
| --- | --- | --- | --- | --- |
| employees | department_id | departments | id | RESTRICT |
| employees | location_id | locations | id | SET NULL |
| disasters | created_by | employees | id | RESTRICT |
| safety_reports | disaster_id | disasters | id | RESTRICT |
| safety_reports | employee_id | employees | id | RESTRICT |
| notifications | employee_id | employees | id | RESTRICT |
| notifications | disaster_id | disasters | id | RESTRICT |

**方針**: 本システムは物理削除を伴う操作を持たない（TBL-POL-003）ため、上記ON DELETEは実運用上ほぼ発動しない。それでも全FKに安全側のデフォルト（RESTRICT。`location_id`のみ拠点未設定を許容する設計と整合させSET NULL）を明示するのは、将来的に物理削除機能が追加された場合の防波堤とするためである（08_ER図 7章）。

# 8. CHECK制約

| テーブル | 制約 | 目的 |
| --- | --- | --- |
| safety_reports | `(latitude IS NULL AND longitude IS NULL) OR (latitude IS NOT NULL AND longitude IS NOT NULL)` | 緯度・経度の片方だけが登録される不整合状態の物理的防止 |
| notifications | `(is_read = FALSE AND read_at IS NULL) OR (is_read = TRUE AND read_at IS NOT NULL)` | 「既読なのに日時がない/未読なのに日時がある」矛盾状態の物理的防止（PMSと同一パターン） |

ステータス系の許容値はENUM型自体が制約として機能するため、他のCHECKは設けない（disasters/safety_reportsのステータス遷移はBR-DIS-002/BR-RPT-001に限定されるが、いずれもアプリ層のトランジション制御で十分と判断 — Project 02の`orders.status`のような複雑な多段階遷移がないため）。

# 11. 設計判断（設計判断メモ）

00_開発計画書 9章・08_ER図が残していた論点への最終回答を記録する。

1. **`disasters.target_department_ids`をJSON列にした理由（junction tableを追加しない）**:
   - **検討した代替案**: `disaster_target_departments`（disaster_id, department_id）という中間テーブルを追加する案。正規化の観点ではこちらが正しい。
   - **採用しなかった理由**: 00_開発計画書 9章で「Roadmap記載の7テーブルをそのまま採用する」ことが確定事項として明記されており、中間テーブルを追加すると8テーブルになりこの前提を崩す。また対象部署の集合は災害作成時に1回確定すれば以後ほぼ更新されない（BR-DIS-001）ため、正規化のメリット（更新時の整合性）が小さい。
   - **トレードオフ**: JSON列内の部署IDが実在する`departments.id`であることをDBのFK制約では保証できない。存在しない部署IDが紛れ込むと、対象範囲の判定（BR-PRM-002の部署スコープ確認）が静かに不正確になるリスクがある。
   - **代替保証**: 災害作成・編集のService層で、`target_department_ids`内の全IDが`departments`テーブルに実在し、かつ`status='active'`であることをアプリ層で検証してから保存する（10_API設計・12_詳細設計書で実装根拠を明記する）。単体試験（15_単体試験仕様書）で「存在しない部署IDを含むリクエストはE003で拒否される」ことを検証項目に含める。
2. **`companies`の行数1件制限はDBで表現できない（TBL-POL-005）**:
   - CHECK制約・UNIQUE制約のいずれも「テーブル全体の行数」は表現できないため、DB層での強制は不可能。
   - **代替保証**: Seederで初期データを1行投入した後、`companies`への新規作成API（POST相当）自体を設けない設計とする（10_API設計では更新APIのみを定義する）。これによりアプリケーションの入口レベルで2行目以降の作成経路自体が存在しなくなる。
3. **`departments`へのBR-ORG-003（無効化・物理削除しない）の適用拡大**:
   - 02_要件定義書のBR-ORG-003は文言上`employees`（退職者等）を主眼に書かれているが、REQ-004「部署の作成・編集・無効化」を満たすには`departments`にも同一の無効化パターンが必要である。08_ER図・本書で`departments.status`ENUMとして明示することで、この拡大適用を物理設計レベルで確定させた。
4. **物理削除を伴う操作が1つも存在しないことの確認（TBL-POL-003）**:
   - Project 02（EC_Site）は会計証憑（`payments`等）を永続保持する一方で商品・カテゴリの無効化を持ち、Project 03（PMS）はタスク削除に伴う連動物理削除（コメント・ファイル）を持っていた。本システムは7テーブルいずれも「無効化」「ステータス切替」「UPDATE」のいずれかで状態変化が完結し、物理削除に相当する操作が存在しない。この単純さは学習上の弱点ではなく、BCP（事業継続計画）ドメインが本質的に「記録は残し続ける」性質を持つことの反映である（02_要件定義書 11.2データ保持方針）。
5. **`safety_reports`の位置情報を`locations`のFK参照にしなかった理由の物理層での再確認**:
   - 02_要件定義書 BR-RPT-003・08_ER図 ER-004で確定済みの判断を、本書では列レベルで具体化した。`safety_reports.latitude`/`longitude`/`address_text`は`locations`テーブルの複製ではなく、報告のたびに個別に発生する値であるため、外部キーではなく独立したカラムとして持つ。

7テーブルの物理設計を確定した。本書の核心は11章の設計判断 — 特に①`target_department_ids`のJSON列化（ロードマップの7テーブル方針を維持するためDB制約の一部を意図的にアプリ層へ委譲した判断）と、②本システムには物理削除を伴う操作が1つも存在しないという確認（BCPドメイン特有の「記録は残し続ける」性質の反映）である。実装時は`safety_reports`のUNIQUE複合キーと`notifications`のCHECK制約を必ずMigrationに含め、15_単体試験仕様書でDB制約レベルの検証（直接INSERT試験）およびtarget_department_idsの部署ID実在性検証（アプリ層）を行うこと。
