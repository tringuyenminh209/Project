Chương này hoàn thiện phần "quản trị" của vòng đời đó: bảng sổ cái inventory_logs (TBL-008), điều chỉnh tồn kho thủ công của Admin (API-027) theo pseudocode 5.7節, và batch tự huỷ đơn pending quá hạn 30 phút (5.6節, BR-INV-006) — mảnh ghép khiến "giữ chỗ" không trở thành "giam hàng vĩnh viễn". ⚠️ Điều kiện vào chương

Đã hoàn thành Chương 04 (cần bảng orders/order_items và luồng đặt hàng chạy được để tạo đơn pending làm dữ liệu test). Batch chạy tự động qua container scheduler đã dựng sẵn ở Chương 01 (php artisan schedule:work, 13_インフラ設計 7.2節) — nếu lúc đó bạn bỏ qua container này, quay lại bổ sung trước.

💡 Vì sao tách "trạng thái" (inventories) và "lịch sử" (inventory_logs)?

inventories chỉ trả lời "bây giờ còn bao nhiêu" — 2 con số, ghi đè liên tục. inventory_logs trả lời "vì sao ra con số đó" — mỗi biến động đã chốt là 1 dòng bất biến kèm balance_after (số dư sau biến động). Khi kiểm kê lệch hoặc điều tra sự cố, cộng dồn quantity_change từ log phải khớp với balance_after cuối cùng — nếu lệch tức có chỗ sửa kho ngoài luồng. Đây là REL-011/012, và 08_ER図.md 13章 gọi cặp này là 1 trong 2 điểm cốt lõi của toàn bộ thiết kế.

💡 Vì sao batch phải tách transaction THEO TỪNG ĐƠN? (12_詳細設計書 10章)

Nếu gói 100 đơn hết hạn vào 1 transaction: đơn thứ 73 lỗi (dữ liệu bẩn, deadlock...) → rollback cả 100, và phút sau batch lại chết y chỗ đó, mãi mãi — hàng của 99 đơn kia bị giam vô thời hạn. Tách mỗi đơn 1 transaction + try/catch: đơn hỏng chỉ ảnh hưởng chính nó, được ghi log lỗi để xử lý tay (20_運用保守手順書), 99 đơn còn lại giải phóng bình thường. Nguyên tắc chung cho mọi batch xử lý hàng loạt: đơn vị thất bại = đơn vị nghiệp vụ nhỏ nhất.

💡 Race nguy hiểm nhất chương: batch huỷ ĐÚNG LÚC Webhook báo thanh toán thành công

Đơn pending ở phút thứ 30: batch vừa SELECT xong danh sách cần huỷ thì Webhook nhận payment_intent.succeeded và đổi đơn sang paid. Nếu batch cứ thế UPDATE cancelled → khách đã trả tiền nhưng đơn bị huỷ và hàng bị trả về kho. Cách chặn: trong transaction của từng đơn, lockForUpdate() dòng order rồi kiểm tra lại status === 'pending' và điều kiện quá hạn — nếu đã đổi thì bỏ qua. Danh sách ID lấy ở bước 1 chỉ là "ứng viên", sự thật chỉ được xác nhận dưới khoá. Pattern re-check-under-lock này gặp lại ở mọi hệ thống có batch + event song song.

💡 Quy ước dấu của quantity_change và cột created_by

TBL-008: quantity_change mang dấu — dương khi hàng quay về khả dụng (nhập kho restock/adjustment tăng, giải phóng release), âm khi xuất (purchase_deduct, Chương 07 sẽ ghi −N). created_by chỉ có giá trị khi con người thao tác (Admin điều chỉnh); mọi biến động do System (batch, Webhook) để NULL — nhìn cột này là biết ngay biến động nào cần hỏi "ai làm, tại sao" (REL-005, TBL-POL-006).

app/Services/InventoryService.phpphp
<?php
// app/Services/InventoryService.php
// 12_詳細設計書.md 5.7節: InventoryService::adjust（BR-INV-008）
// ※5.7の疑似コードでは「1.ロック→3.トランザクション開始」の順に書かれているが、
//   SELECT ... FOR UPDATEはトランザクション内でのみ有効なため、実装では全体をtransactionで包む

namespace App\Services;

use App\Exceptions\ResourceNotFoundException;
use App\Models\Inventory;
use App\Models\InventoryLog;
use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Log;
use Illuminate\Validation\ValidationException;

class InventoryService
{
    // API-027 在庫調整
    public function adjust(int $variantId, array $data, int $adminUserId): Inventory
    {
        return DB::transaction(function () use ($variantId, $data, $adminUserId) {
            // 5.7-1: variant_idでFOR UPDATE取得（UNIQUE制約により行を一意特定、BR-INV-007）
            $inventory = Inventory::where('variant_id', $variantId)
                ->lockForUpdate()
                ->first();
            if (!$inventory) {
                throw new ResourceNotFoundException('対象の在庫が見つかりません。');
            }

            $balanceAfter = $inventory->quantity_available + $data['quantity_change'];

            // 5.7-2: マイナスになる調整はE003で拒否（UT-INV-003。CHECK制約はあくまで最終防衛線）
            if ($balanceAfter < 0) {
                throw ValidationException::withMessages([
                    'quantity_change' => '調整後の在庫がマイナスになるため実行できません。',
                ]);
            }

            // 5.7-4
            $inventory->update(['quantity_available' => $balanceAfter]);

            // 5.7-5: BR-INV-008 — change_type=adjustment、実行者(created_by)を記録
            InventoryLog::create([
                'variant_id' => $variantId,
                'change_type' => 'adjustment',
                'quantity_change' => $data['quantity_change'],
                'balance_after' => $balanceAfter,
                'reference_type' => null,
                'reference_id' => null,
                'created_by' => $adminUserId,
            ]);

            // reasonの記録先: TBL-008にreason列は存在しないため、アプリケーションログへ記録する
            // （12_詳細設計書 13章のログ設計に従う。REQ-027の操作ログ要件もこれで満たす）
            Log::info('在庫手動調整', [
                'variant_id' => $variantId,
                'quantity_change' => $data['quantity_change'],
                'balance_after' => $balanceAfter,
                'reason' => $data['reason'],
                'admin_user_id' => $adminUserId,
            ]);

            return $inventory->refresh();
        });
    }
}

Batch — CancelExpiredPendingOrders（5.6節）

Đọc kỹ comment ở cancelOne(): re-check trạng thái dưới khoá là linh hồn của batch này (box-why số 3). Thứ tự khoá inventory trong mỗi đơn vẫn là variant_id tăng dần — batch chạy song song với các OrderService::place đang giữ chỗ, cùng một quy ước khoá thì không deadlock.

app/Console/Commands/CancelExpiredPendingOrders.phpphp
<?php
// app/Console/Commands/CancelExpiredPendingOrders.php
// 12_詳細設計書.md 5.6節 / BR-INV-005〜006 / FUNC-030
// 毎分実行され、在庫確保期限（30分）を過ぎたpending注文を自動キャンセルし在庫・クーポンを解放する

namespace App\Console\Commands;

use App\Models\Coupon;
use App\Models\Inventory;
use App\Models\InventoryLog;
use App\Models\Order;
use Illuminate\Console\Command;
use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Log;

class CancelExpiredPendingOrders extends Command
{
    protected $signature = 'orders:cancel-expired';

    protected $description = 'BR-INV-006: 在庫確保期限切れのpending注文を自動キャンセルする';

    public function handle(): int
    {
        // 5.6-1: 対象IDのみ抽出。実データの検証・更新は注文単位のトランザクション内で行う
        $expiredIds = Order::pending()
            ->where('created_at', '<', now()->subMinutes(30))
            ->pluck('id');

        foreach ($expiredIds as $orderId) {
            try {
                $this->cancelOne($orderId);
            } catch (\Throwable $e) {
                // 12_詳細設計書 10章: 注文単位でトランザクションを分割し、
                // 1件の失敗が他の注文の解放処理に波及しないようにする
                Log::error('CancelExpiredPendingOrders: 解放失敗', [
                    'order_id' => $orderId,
                    'error' => $e->getMessage(),
                ]);
            }
        }

        return self::SUCCESS;
    }

    private function cancelOne(int $orderId): void
    {
        DB::transaction(function () use ($orderId) {
            // 行ロックの上でpendingを「再確認」する — ここが本バッチの最重要ポイント。
            // 5.6-1の抽出とこの瞬間の間に、WebhookがStripe決済成功を受けてpaidへ更新している
            // 可能性がある（BR-PAY-001との競合）。再確認せずにcancelledへ倒すと
            // 「支払い済みなのにキャンセル・在庫解放」という最悪の不整合が生まれる
            $order = Order::whereKey($orderId)->lockForUpdate()->first();
            if (!$order
                || $order->status !== 'pending'
                || $order->created_at->gt(now()->subMinutes(30))) {
                return; // 対象外に変わっていたら何もしない
            }

            // 5.6-2a
            $order->update(['status' => 'cancelled']);

            // 5.6-2b: 在庫解放（BR-INV-005）。ロック順はOrderService::placeと同じvariant_id昇順
            //（5.2節-4と同じ理由 — 並行する注文確定処理とのデッドロック防止）
            $items = $order->items()->orderBy('variant_id')->get();
            foreach ($items as $item) {
                $inventory = Inventory::where('variant_id', $item->variant_id)
                    ->lockForUpdate()
                    ->first();
                if (!$inventory) {
                    continue;
                }

                $balanceAfter = $inventory->quantity_available + $item->quantity;
                $inventory->update([
                    'quantity_available' => $balanceAfter,
                    'quantity_reserved' => $inventory->quantity_reserved - $item->quantity,
                ]);

                InventoryLog::create([
                    'variant_id' => $item->variant_id,
                    'change_type' => 'release',
                    'quantity_change' => $item->quantity, // +N: 利用可能在庫へ戻す（TBL-008 符号規約）
                    'balance_after' => $balanceAfter,
                    'reference_type' => 'orders',
                    'reference_id' => $order->id,
                    'created_by' => null, // System起因はNULL（TBL-008 / REL-005）
                ]);
            }

            // 5.6-2c: クーポン利用の解放（BR-CPN-003）。
            // クーポン適用ロジック自体はChương 06だが、解放側を先に備えておく（coupon_id未使用ならスキップ）
            if ($order->coupon_id) {
                Coupon::whereKey($order->coupon_id)->decrement('used_count');
            }
        });
    }
}
