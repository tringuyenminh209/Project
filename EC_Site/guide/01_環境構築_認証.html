Chương 01 — Môi trường phát triển & Xác thực (Auth)

Chương này dựng nền móng đầu tiên của EC Site: Docker Compose cho local dev (nginx + app + db + scheduler), khung project Laravel 12, bảng users (TBL-001), response envelope dùng chung cho toàn hệ thống, và luồng xác thực cốt lõi — đăng ký (API-001) và đăng nhập (API-002) — bằng Laravel Sanctum. Mọi chương sau (giỏ hàng, đơn hàng, thanh toán...) đều cần user đã đăng nhập, nên đây là điều kiện bắt buộc trước tiên.

💡 Vì sao dùng Sanctum token, không dùng session?

10_API設計.md API-POL-003 quy định API là Stateless, xác thực bằng Sanctum token thay vì session, để đáp ứng NFR-009 (Web server có thể mở rộng nhiều instance mà không cần sticky session). Token đăng ký/đăng nhập có hạn 24 giờ (API設計 4.1節) — dài hơn Project 01 (8 giờ) vì đối tượng là Customer, ưu tiên trải nghiệm mua sắm liền mạch hơn.

💡 Vì sao cột mật khẩu tên là password_hash chứ không phải password?

09_テーブル定義.md TBL-001 đặt tên cột là password_hash (rõ nghĩa hơn quy ước mặc định password của Laravel). Vì Eloquent's Authenticatable mặc định đọc cột password khi xác thực, model User phải override getAuthPassword() để trỏ đúng cột — nếu bỏ qua bước này, Hash::check() nội bộ của Laravel (Auth::attempt) sẽ luôn thất bại. Vì lý do đó, guide này tự kiểm tra mật khẩu bằng Hash::check() trực tiếp trong Service thay vì dùng Auth::attempt(), để không phụ thuộc ngầm vào quy ước cột của framework.

💡 Vì sao E001 dùng chung cho "sai email", "sai mật khẩu" VÀ "tài khoản bị khoá"?

02_要件定義書.md BR-USR-001: tài khoản status=inactive không thể đăng nhập. Thay vì trả một mã lỗi riêng cho trường hợp này, AuthService::login gộp cả 3 nguyên nhân (không tồn tại / sai mật khẩu / bị khoá) vào cùng một E001 với cùng một thông điệp — tránh lộ thông tin cho kẻ tấn công dò email nào đã tồn tại trong hệ thống (enumeration attack), đúng tinh thần NFR-012/NFR-013.

Migration bảng users（TBL-001）

Bám sát đúng từng cột trong 09_テーブル定義.md 4.1節 — đặc biệt password_hash (không phải password), và 2 cột ENUM có index riêng phục vụ lọc theo role/status. Không tạo file migration mới: Laravel đã có sẵn file mặc định 0001_01_01_000001_create_users_table.php tạo bảng users — tạo file mới sẽ nổ "table already exists". Thay toàn bộ nội dung file mặc định bằng bản dưới (giữ nguyên 2 bảng phụ password_reset_tokens/sessions — bảng sessions đang được SESSION_DRIVER=database mặc định của .env sử dụng).

database/migrations/0001_01_01_000001_create_users_table.php（Laravel標準ファイルをこの内容で置き換える）php

        'phone',
        'role',
        'status',

    {

    {
        return $query->where('status', 'active');
    }

Response envelope & exception base dùng chung toàn hệ thống

10_API設計.md 3.3節 quy định mọi API phải trả về đúng 1 trong 2 hình dạng: {success:true, message, data} hoặc {success:false, error:{code,message,details}}. Thay vì lặp lại cấu trúc này ở từng Controller, ta định nghĩa 1 exception nền (ApiException) để mọi exception nghiệp vụ (E004 thiếu tồn kho, E005 coupon, E006 sai trạng thái đơn hàng... ở các chương sau) chỉ cần kế thừa và khai báo mã lỗi.

app/Exceptions/ApiException.phpphp
<?php
// app/Exceptions/ApiException.php
// 10_API設計.md 3.3節（共通Response形式）+ 12_詳細設計書.md DD-POL-003
// 「業務エラーはカスタムExceptionとして定義し、render()でJSON+HTTPステータスを返す」の基底クラス

namespace App\Exceptions;

use Exception;
use Illuminate\Http\JsonResponse;

abstract class ApiException extends Exception
{
    protected string $errorCode;
    protected int $httpStatus;
    protected array $details = [];

    public function render(): JsonResponse
    {
        return response()->json([
            'success' => false,
            'error' => [
                'code' => $this->errorCode,
                'message' => $this->getMessage(),
                'details' => $this->details,
            ],
        ], $this->httpStatus);
    }
}
app/Exceptions/AuthenticationFailedException.phpphp
<?php
// app/Exceptions/AuthenticationFailedException.php
// 02_要件定義書.md 16章 E001 / 10_API設計.md 6.2節(API-002)・7章
// BR-USR-001（無効化会員はログインできない）もこの例外に統一する
// -> メール不一致・パスワード不一致・status=inactive のいずれも「ログイン失敗」として同じE001を返す
//    （攻撃者にメール存在有無を推測させないため、理由ごとにメッセージを変えない）

namespace App\Exceptions;

class AuthenticationFailedException extends ApiException
{
    protected string $errorCode = 'E001';
    protected int $httpStatus = 401;

    public function __construct()
    {
        parent::__construct('メールアドレスまたはパスワードが正しくありません。');
    }
}

Framework tự ném ValidationException (khi FormRequest fail) và AuthenticationException (khi middleware auth:sanctum không thấy token hợp lệ) — 2 exception này không kế thừa ApiException nên cần đăng ký riêng trong bootstrap/app.php để cùng trả đúng envelope (E003/E010):

bootstrap/app.php（抜粋）php
<?php
// bootstrap/app.php（抜粋）
// 10_API設計.md 3.3節の共通Response形式を、フレームワークが投げる例外（Validation/認証エラー等）にも適用する。
// ApiException系（業務エラー）はrender()を持つため個別処理は不要。ここではフレームワーク標準の例外のみ変換する。

use Illuminate\Auth\AuthenticationException;
use Illuminate\Foundation\Application;
use Illuminate\Foundation\Configuration\Exceptions;
use Illuminate\Foundation\Configuration\Middleware;
use Illuminate\Validation\ValidationException;

return Application::configure(basePath: dirname(__DIR__))
    ->withRouting(
        api: __DIR__.'/../routes/api.php',
        commands: __DIR__.'/../routes/console.php',
        health: '/api/health',
    )
    ->withMiddleware(function (Middleware $middleware) {
        //
    })
    ->withExceptions(function (Exceptions $exceptions) {
        // E003: バリデーションエラー（422）
        $exceptions->render(function (ValidationException $e) {
            return response()->json([
                'success' => false,
                'error' => [
                    'code' => 'E003',
                    'message' => '入力内容に誤りがあります。',
                    'details' => $e->errors(),
                ],
            ], 422);
        });

        // E010: 未認証（auth:sanctumミドルウェアがトークン無し/失効時にこれを投げる）
        $exceptions->render(function (AuthenticationException $e) {
            return response()->json([
                'success' => false,
                'error' => [
                    'code' => 'E010',
                    'message' => '認証が必要です。',
                    'details' => [],
                ],
            ], 401);
        });
    })->create();

AuthService & AuthController
app/Services/AuthService.phpphp
<?php
// app/Services/AuthService.php
// 12_詳細設計書.md 3章（Fat Service, Thin Controller）
// register/login のみ実装。logout/me/updatePassword は本チャプターの課題（下の表を参照）。

namespace App\Services;

use App\Exceptions\AuthenticationFailedException;
use App\Models\User;
use Illuminate\Support\Facades\Hash;

class AuthService
{
    // API-001 会員登録
    public function register(array $data): array
    {
        $user = User::create([
            'name' => $data['name'],
            'email' => $data['email'],
            'password_hash' => Hash::make($data['password']),
            'role' => 'customer',
            'status' => 'active',
        ]);

        // 10_API設計.md 6.1節: 「登録成功時はログインと同様に即座にアクセストークンを発行する」
        $token = $user->createToken('api-token')->plainTextToken;

        return $this->buildAuthResponse($user, $token);
    }

    // API-002 ログイン
    public function login(array $data): array
    {
        $user = User::where('email', $data['email'])->first();

        // BR-USR-001: status=inactive の会員はログイン不可。
        // メール不存在・パスワード不一致・inactiveのいずれも同じE001に丸める（列挙攻撃対策）。
        if (!$user || $user->status !== 'active' || !Hash::check($data['password'], $user->password_hash)) {
            throw new AuthenticationFailedException();
        }

        $token = $user->createToken('api-token')->plainTextToken;

        return $this->buildAuthResponse($user, $token);
    }

    private function buildAuthResponse(User $user, string $token): array
    {
        return [
            'access_token' => $token,
            'user' => [
                'id' => $user->id,
                'name' => $user->name,
                'email' => $user->email,
                'role' => $user->role,
            ],
        ];
    }
}
app/Http/Controllers/Auth/AuthController.phpphp
<?php
// app/Http/Controllers/Auth/AuthController.php
// 12_詳細設計書.md 4章: Auth\AuthController — register, login, logout, me, updatePassword (API-001〜005)
// このチャプターでは register / login のみ実装する。残り3メソッドは課題（下の表を参照）。

namespace App\Http\Controllers\Auth;

use App\Http\Controllers\Controller;
use App\Http\Requests\Auth\LoginRequest;
use App\Http\Requests\Auth\RegisterRequest;
use App\Services\AuthService;
use Illuminate\Http\JsonResponse;

class AuthController extends Controller
{
    public function __construct(private AuthService $authService)
    {
    }

    public function register(RegisterRequest $request): JsonResponse
    {
        $result = $this->authService->register($request->validated());

        return response()->json([
            'success' => true,
            'message' => '会員登録が完了しました。',
            'data' => $result,
        ], 201);
    }

    public function login(LoginRequest $request): JsonResponse
    {
        $result = $this->authService->login($request->validated());

        return response()->json([
            'success' => true,
            'message' => 'ログインしました。',
            'data' => $result,
        ]);
    }
}
