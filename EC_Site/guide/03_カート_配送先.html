Chương 03 — Giỏ hàng & Địa chỉ giao hàng

Chương này dựng 3 bảng addresses / carts / cart_items (TBL-002/009/010) và luồng giỏ hàng: xem giỏ (API-009) và thêm vào giỏ (API-010) theo đúng pseudocode CartService::addItem (12_詳細設計書 5.1節). Đây là chương "đệm" quan trọng trước chương Đặt hàng: mọi khái niệm ở đây — active cart, kiểm tra tồn kho không trừ kho, cộng dồn số lượng — đều là tiền đề trực tiếp cho OrderService::place (5.2節).

⚠️ Điều kiện vào chương

Đã hoàn thành Chương 02 (cần bảng product_variants + inventories và dữ liệu mẫu có tồn kho — nếu chưa làm 課題 seeder, hãy tạo tay 1 sản phẩm + variant qua API-025 rồi chỉnh tồn kho trực tiếp trong DB, vì API-027 điều chỉnh tồn kho thuộc chương sau).

💡 Vì sao "1 user 1 active cart" không ép bằng UNIQUE constraint?

09_テーブル定義.md 11章-2: MySQL không hỗ trợ partial unique index (UNIQUE có điều kiện WHERE status='active'), nên không thể biểu diễn ràng buộc "mỗi user chỉ có 1 cart active" thuần bằng DB — UNIQUE trên (user_id, status) sẽ chặn luôn việc có nhiều cart converted trong lịch sử. Có thể giả lập bằng generated column + UNIQUE, nhưng thiết kế đánh giá là phức tạp không tương xứng, và chọn bảo đảm ở tầng Service: mọi thao tác lấy-hoặc-tạo active cart phải chạy trong transaction với SELECT ... FOR UPDATE. Nhưng chú ý cái bẫy: FOR UPDATE không khoá được dòng chưa tồn tại — nếu user chưa có cart nào, 2 request song song đều thấy "chưa có" rồi cùng tạo 2 cart. Vì vậy code mẫu khoá thêm dòng users của chính user (luôn tồn tại) làm mỏ neo trực tiếp hoá mọi thao tác cart của một user — xem comment trong addItem.

💡 Vì sao thêm vào giỏ KHÔNG trừ tồn kho? (BR-INV-002)

Đây là quyết định thiết kế trung tâm của 02_要件定義書.md 9.3節: trừ kho lúc thêm giỏ → người "chỉ xem" chiếm giữ hàng, mất cơ hội bán; không trừ gì đến tận lúc thanh toán → dễ hết hàng giữa chừng khi đang thanh toán. EC Site chọn phương án 2 giai đoạn: giỏ hàng không đụng tồn kho (chương này), đặt hàng mới giữ chỗ (reserve, chương sau), thanh toán mới chốt. Vì vậy E004 ném ra ở addItem chỉ mang tính cảnh báo hiển thị — kể cả thêm giỏ thành công cũng không có gì bảo đảm còn hàng lúc đặt; kiểm tra "thật" nằm ở OrderService::place với khóa bi quan (BR-INV-007).

💡 Vì sao giá trong giỏ không phải giá cuối cùng?

Response của API-009 trả price / line_subtotal tính từ product_variants.price hiện tại — thuần để hiển thị. Nếu Admin đổi giá trong lúc hàng nằm trong giỏ, lần mở giỏ sau sẽ thấy giá mới (đúng chủ đích). Giá chỉ bị "đóng băng" tại thời điểm xác nhận đơn, khi snapshot vào order_items.unit_price (BR-ORD-002). Hiểu rõ ranh giới này thì chương Đặt hàng sẽ rất tự nhiên.

app/Services/CartService.phpphp
<?php
// app/Services/CartService.php
// 12_詳細設計書.md 5.1節: CartService::addItem（BR-INV-002）
// show/addItemのみ実装。updateItem/removeItem（API-011/012）は課題（下の表を参照）。

namespace App\Services;

use App\Exceptions\InsufficientStockException;
use App\Exceptions\ResourceNotFoundException;
use App\Models\Cart;
use App\Models\CartItem;
use App\Models\ProductVariant;
use App\Models\User;
use Illuminate\Support\Facades\DB;

class CartService
{
    // API-009 カート取得
    public function show(int $userId): array
    {
        $cart = Cart::active()
            ->where('user_id', $userId)
            ->with(['items.variant.product', 'items.variant.inventory'])
            ->first();

        // 10_API設計.md 6.9節: activeカートが無い場合はエラーにせず空配列を返す
        if (!$cart) {
            return ['cart_id' => null, 'items' => [], 'subtotal' => 0];
        }

        $items = $cart->items->map(fn ($item) => [
            'cart_item_id' => $item->id,
            'variant_id' => $item->variant_id,
            'product_name' => $item->variant->product->name,
            // 表示名は order_items.variant_label（TBL-012、例: "M / Red"）と同じ組み立て方
            'variant_label' => implode(' / ', array_filter([$item->variant->size, $item->variant->color])),
            'sku' => $item->variant->sku,
            'price' => $item->variant->price,
            'quantity' => $item->quantity,
            'line_subtotal' => $item->variant->price * $item->quantity,
            'quantity_available' => $item->variant->inventory?->quantity_available ?? 0,
        ]);

        return [
            'cart_id' => $cart->id,
            'items' => $items->values()->all(),
            // カート画面の小計はあくまで現在価格による表示用プレビュー。
            // 確定金額は注文確定時にスナップショットして計算する（BR-ORD-002、次章）
            'subtotal' => $items->sum('line_subtotal'),
        ];
    }

    // API-010 カート追加（12_詳細設計書.md 5.1節の疑似コードに1:1対応）
    public function addItem(int $userId, array $data): CartItem
    {
        return DB::transaction(function () use ($userId, $data) {
            // 直列化アンカー: SELECT ... FOR UPDATEは「まだ存在しない行」をロックできない。
            // カート未作成の会員が2並列でaddItemすると、両方ともactiveカートを見つけられず
            // 2つ作ってしまう（幻の行問題）。必ず存在するusersの自分の行をロックすることで、
            // 同一会員のカート作成を1本ずつに直列化する（09_テーブル定義 11章-2の保証を実装で担う）
            User::whereKey($userId)->lockForUpdate()->first();

            // 5.1-1: activeカートを取得、無ければ新規作成。
            // 09_テーブル定義.md 11章-2: 「1会員1 activeカート」はMySQLの部分UNIQUE制約非対応のため
            // DB制約で表現できず、トランザクション + 上記ロックで保証する
            $cart = Cart::active()
                ->where('user_id', $userId)
                ->lockForUpdate()
                ->first();

            if (!$cart) {
                $cart = Cart::create(['user_id' => $userId, 'status' => 'active']);
            }

            // 5.1-2: variant + inventory を取得（inactiveや在庫行欠損はE007）
            $variant = ProductVariant::with('inventory')
                ->where('status', 'active')
                ->find($data['variant_id']);

            if (!$variant || !$variant->inventory) {
                throw new ResourceNotFoundException('対象の商品が見つかりません。');
            }

            // 5.1-3 / BR-INV-002: この時点では在庫を一切変更しない。
            // このE004は「表示上の警告」目的のチェックのみ（確保は注文確定時、BR-INV-003）
            if ($data['quantity'] > $variant->inventory->quantity_available) {
                throw new InsufficientStockException();
            }

            // 5.1-4: UNIQUE(cart_id, variant_id) — 既存行があれば数量を加算、無ければ新規作成
            $item = CartItem::where('cart_id', $cart->id)
                ->where('variant_id', $variant->id)
                ->first();

            if ($item) {
                $item->update(['quantity' => $item->quantity + $data['quantity']]);
            } else {
                $item = CartItem::create([
                    'cart_id' => $cart->id,
                    'variant_id' => $variant->id,
                    'quantity' => $data['quantity'],
                ]);
            }

            return $item;
        });
    }
}

// app/Http/Controllers/CartController.php
// 12_詳細設計書.md 4章: CartController — show, addItem, updateItem, removeItem (API-009〜012)

namespace App\Http\Controllers;

use Illuminate\Http\JsonResponse;

class CartController extends Controller
{

    }

    // GET /cart (API-009)

    {
        return response()->json([
            'success' => true,

            'data' => [
                'cart_item_id' => $item->id,
                'variant_id' => $item->variant_id,
                'quantity' => $item->quantity,
            ],
        ], 201);
    }
}
