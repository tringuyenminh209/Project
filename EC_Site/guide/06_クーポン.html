Chương 06 — Coupon

Chương này lấp 2 mảnh đã gác lại ở Chương 04: bước 5.2-6 (kiểm tra coupon) và 5.2-11 (đếm lượt dùng) của OrderService::place. Nội dung gồm: CouponService::validate (5.3節, BR-CPN-001) + calculateDiscount (5.4節, BR-CPN-002), API preview API-014, và tích hợp vào luồng đặt hàng. Điểm học chính: coupon dùng lại y nguyên tư duy tồn kho — "tạm giữ khi chốt đơn → hoàn lại khi huỷ" (BR-CPN-003), và cũng có race condition cần khoá y như tồn kho.

⚠️ Điều kiện vào chương

Đã hoàn thành Chương 04 (bảng coupons đã tạo trong migration, model Coupon đã có) và Chương 05 (batch huỷ đơn — phần hoàn lượt dùng used_count -= 1 đã được cài sẵn ở đó, chương này sẽ kiểm chứng nó chạy thật).

💡 Coupon là "tồn kho thứ hai" — cùng pattern tạm giữ → chốt/hoàn (BR-CPN-003)

usage_limit là tài nguyên hữu hạn giống quantity_available: chốt đơn (pending) thì used_count += 1 ngay — không đợi thanh toán xong; đơn huỷ/hết hạn 30 phút thì batch hoàn used_count -= 1 (đã viết ở Chương 05). Nếu đợi đến lúc thanh toán mới đếm, 2 khách cùng cầm 1 mã "còn 1 lượt" sẽ đều qua cửa kiểm tra. Và cũng vì là tài nguyên tranh chấp, lúc chốt đơn phải lockForUpdate dòng coupon rồi mới kiểm tra — thứ tự khoá cố định "inventories (variant_id tăng dần) → coupons" để mọi transaction xếp hàng cùng chiều, không deadlock (cùng lý do với 5.2-4).

💡 Vì sao mọi lý do từ chối đều trả chung E005?

5.3節 quy định 4 cửa kiểm tra (tồn tại/active → thời hạn → tiền tối thiểu → lượt dùng) nhưng tất cả đều ném cùng một E005 với cùng một message. Nếu message phân biệt "mã không tồn tại" với "mã hết hạn", kẻ dò mã hàng loạt sẽ biết mã nào có thật để tập trung thử — cùng triết lý với E001 không nói rõ sai email hay sai mật khẩu (Chương 01).

💡 Vì sao API-014 tin subtotal do client gửi lên?

Request của API-014 chứa subtotal từ client — nghe có vẻ dễ gian lận, nhưng API này chỉ là preview hiển thị (6.14節: "used_countはまだ更新しない"). Con số có thẩm quyền được tính lại hoàn toàn ở server trong OrderService::place từ giá variant thật (client không gửi được giỏ hàng lên — 6.15節 改ざん防止), và coupon được validate lại lần nữa dưới khoá tại đó. Gian lận preview chỉ lừa được... chính màn hình của kẻ gian lận.

💡 Vì sao giảm giá tính trên subtotal (chưa thuế) và phải chặn trần?

BR-CPN-002: giảm giá áp lên subtotal (tax-exclusive) — thuế vẫn tính đủ trên giá gốc từng dòng (BR-TAX-004), không giảm nghĩa vụ thuế. percent cắt lẻ bằng floor (nhất quán với thuế). Và 5.4-3 chặn min(discount, subtotal): coupon cố định 500円 áp cho đơn 300円 mà không chặn thì grand_total âm — mà cột này là INT UNSIGNED, INSERT sẽ nổ ngay tầng DB.

CouponService — validate（5.3節）& calculateDiscount（5.4節）

4 cửa kiểm tra của validate viết tuần tự đúng thứ tự 5.3節 (đọc pseudocode trước). Tham số $lockForUpdate phân đôi 2 ngữ cảnh dùng: preview (API-014, không khoá) và chốt đơn (khoá — box-why số 1).

app/Services/CouponService.phpphp
<?php
// app/Services/CouponService.php
// 12_詳細設計書.md 5.3節（validate） / 5.4節（calculateDiscount）

namespace App\Services;

use App\Exceptions\CouponNotApplicableException;
use App\Models\Coupon;

class CouponService
{
    // 5.3節: BR-CPN-001。不成立理由が何であれ同一のE005を返す
    public function validate(string $code, int $subtotal, bool $lockForUpdate = false): Coupon
    {
        $query = Coupon::where('code', $code);
        if ($lockForUpdate) {
            // 注文確定時のみロックする。usage_limit残り1枚を並行注文が奪い合うと
            // validate通過→両方used_count+1で上限超過が起きるため、行ロックの上で判定する
            //（在庫と同じ発想。ロック順は「inventories(variant_id昇順)→coupons」の固定順とし、
            //  全注文が同じ順序で取得することでデッドロックを防ぐ — 5.2節-4と同じ理屈）
            $query->lockForUpdate();
        }
        $coupon = $query->first();

        // 5.3-1: 存在しない / status=inactive
        if (!$coupon || $coupon->status !== 'active') {
            throw new CouponNotApplicableException();
        }

        // 5.3-2: 有効期間外（valid_from〜valid_to、両端含む）
        $today = today();
        if ($today->lt($coupon->valid_from) || $today->gt($coupon->valid_to)) {
            throw new CouponNotApplicableException();
        }

        // 5.3-3: 最低購入金額未満（判定は税抜subtotalに対して行う、BR-CPN-002）
        if ($subtotal < $coupon->min_purchase_amount) {
            throw new CouponNotApplicableException();
        }

        // 5.3-4: 利用上限到達（usage_limit=NULLは無制限、TBL-015）
        if (!is_null($coupon->usage_limit) && $coupon->used_count >= $coupon->usage_limit) {
            throw new CouponNotApplicableException();
        }

        return $coupon;
    }

    // 5.4節: BR-CPN-002
    public function calculateDiscount(int $subtotal, Coupon $coupon): int
    {
        // 5.4-1/2: fixed=固定額 / percent=floor(subtotal × value / 100)（円未満切り捨て）
        $discount = $coupon->discount_type === 'fixed'
            ? $coupon->discount_value
            : intdiv($subtotal * $coupon->discount_value, 100); // 整数演算（Chương 04の税計算と同じ理由でfloatを避ける）

        // 5.4-3: 割引はsubtotalを超えない — grand_total（UNSIGNED列）の負数化を防ぐ
        return min($discount, $subtotal);
    }
}

<?php
// app/Http/Controllers/CouponController.php
// 12_詳細設計書.md 4章: CouponController — validateCode (API-014)

namespace App\Http\Controllers;

use App\Http\Requests\Coupon\ValidateCouponRequest;
use App\Services\CouponService;
use Illuminate\Http\JsonResponse;

class CouponController extends Controller
{
    public function __construct(private CouponService $couponService)
    {
    }

    // POST /coupons/validate (API-014) — 注文確定前のプレビュー専用。
    // used_countはここでは一切更新しない（実際の消費はAPI-015、6.14節）
    public function validateCode(ValidateCouponRequest $request): JsonResponse
    {
        $data = $request->validated();

        $coupon = $this->couponService->validate($data['code'], $data['subtotal']);
        $discountAmount = $this->couponService->calculateDiscount($data['subtotal'], $coupon);

        // Response shapeは10_API設計.md 6.14節に厳密対応
        return response()->json([
            'success' => true,
            'message' => 'OK',
            'data' => [
                'coupon_id' => $coupon->id,
                'discount_type' => $coupon->discount_type,
                'discount_value' => $coupon->discount_value,
                'discount_amount' => $discountAmount,
            ],
        ]);
    }
}

✅ チェックポイント（UT-CPN-001〜005対応）
  • ☐ POST /coupons/validate với SUMMER10 + subtotal=8500 → discount_amount=850, và used_count không đổi (UT-CPN-001, preview thuần)
  • ☐ Mã hết hạn (EXPIRED) → 409 E005 (UT-CPN-002); mã đạt trần lượt dùng (MAXED) → 409 E005 (UT-CPN-003); mã không tồn tại → cũng E005 cùng message (không lộ thông tin)
  • ☐ Đặt hàng kèm coupon_code=SUMMER10 → orders.coupon_id được set, discount_total=850, grand_total = subtotal + tax_total − 850, used_count+1 (UT-CPN-004). Đơn 8500円 tái hiện đúng ví dụ response 6.15節: tax 850 − discount 850 → grand đúng bằng subtotal (trùng hợp có chủ đích trong doc)
  • ☐ Đặt hàng với coupon không thoả điều kiện → 409 E005, rollback toàn phần: không có orders/order_items, tồn kho nguyên, used_count nguyên
  • ☐ Đơn pending có coupon, lùi created_at 31 phút → chạy orders:cancel-expired → used_count −1 (UT-CPN-005 — kiểm chứng code hoàn trả viết sẵn ở Chương 05)
  • ☐ WELCOME500 (fixed 500円) với đơn subtotal 300円 → discount_total=300 (bị chặn trần), grand_total không âm, INSERT không nổ UNSIGNED
  • ☐ Percent với số lẻ: subtotal 8505円 × 10% = 850.5 → floor → 850
