Chương 04 — Đặt hàng & Snapshot (chương trung tâm)

Đây là chương quan trọng nhất của cả project. OrderService::place (API-015) là nơi gần như toàn bộ business rules giao nhau: BR-ORD-001~003 (trạng thái + snapshot), BR-TAX-001~004 (thuế), BR-INV-001~003 (giữ chỗ tồn kho), BR-INV-007 (khoá bi quan). Trước khi đọc tiếp, hãy mở 12_詳細設計書.md 5.2節 và đọc kỹ 13 bước pseudocode — code mẫu dưới đây đánh số comment 5.2-1~5.2-13 tương ứng từng bước.

⚠️ Điều kiện vào chương & phạm vi gác lại

Cần hoàn thành Chương 03 (giỏ hàng hoạt động, có 課題 API-013 địa chỉ — bắt buộc, vì đặt hàng cần address_id). Hai mảnh của 5.2節 được chủ ý gác lại: coupon (bước 5.2-6/11 — Chương 06, hiện discount_total=0) và Stripe PaymentIntent + bảng payments (bước 5.2-13 — Chương 07, hiện response chưa có stripe_client_secret). Đơn hàng tạo ra sẽ nằm ở pending vĩnh viễn cho tới khi có Webhook (Chương 07) — đó là trạng thái đúng, không phải bug.

💡 Vì sao phải snapshot thay vì JOIN? (BR-ORD-002/003)

Đơn hàng là chứng từ kế toán: khách đã đồng ý mua "áo T-shirt, size M, giá 2,980円, thuế 10%" tại một thời điểm. Nếu order_items chỉ giữ variant_id rồi JOIN lấy giá hiện tại, thì Admin đổi giá một cái là toàn bộ đơn hàng quá khứ "tự đổi giá" theo — sai cả nghiệp vụ lẫn pháp lý. Vì vậy 08_ER図 10章 chấp nhận phá chuẩn hoá có chủ đích: copy cứng product_name / variant_label / sku / unit_price / tax_rate vào order_items, copy địa chỉ vào orders.shipping_* (nên orders không có FK tới addresses). Cột order_items.variant_id vẫn còn nhưng chỉ để phân tích (REL-014) — cấm dùng để tính tiền.

💡 Vì sao khoá tồn kho phải theo thứ tự variant_id tăng dần? (5.2-4)

Hai đơn hàng song song cùng chứa variant 1 và 5: nếu đơn A khoá theo thứ tự (1→5) còn đơn B khoá (5→1), A giữ khoá 1 chờ khoá 5, B giữ khoá 5 chờ khoá 1 → deadlock, MySQL sẽ kill một transaction. Cố định thứ tự khoá tăng dần theo variant_id khiến mọi transaction xếp hàng cùng chiều — ai đến trước khoá trước, không bao giờ chờ vòng tròn. Đây là kỹ thuật kinh điển khi khoá nhiều dòng, và là lý do UNIQUE trên inventories.variant_id quan trọng (khoá đúng 1 dòng xác định, BR-INV-007).

💡 Vì sao thuế làm tròn theo TỪNG DÒNG chứ không theo tổng đơn? (BR-TAX-004)

line_tax = floor(line_subtotal × tax_rate) tính và cắt lẻ trên từng dòng, rồi tax_total = Σ line_tax. Nếu tính floor(subtotal × rate) trên tổng, kết quả có thể lệch 1~2円 so với tổng các dòng — và khi in hoá đơn từng dòng sẽ không khớp tổng. Chọn quy tắc nào cũng phải nhất quán; spec chọn theo dòng (khớp cách 国税庁 chấp nhận). Chú ý đơn có cả hàng 10% lẫn hàng 8% (軽減税率): mỗi dòng mang tax_rate riêng — chính vì thế tax_rate nằm ở order_items chứ không phải orders.

💡 Vì sao giữ chỗ (reserve) nhưng CHƯA ghi inventory_logs? (5.2-10)

Ở bước đặt hàng, tồn kho chỉ chuyển trạng thái available → reserved — chưa phải tiêu thụ thật (đơn có thể bị huỷ/hết hạn 30 phút, BR-INV-006, khi đó trả ngược lại). inventory_logs là sổ ghi biến động đã chốt: chỉ khi Webhook xác nhận thanh toán thành công mới ghi change_type=purchase_deduct (BR-INV-004, Chương 07). Nếu ghi log ngay lúc reserve, mỗi đơn huỷ sẽ tạo cặp log rác deduct/release cho giao dịch chưa từng thành hình.

Migration（TBL-015 → TBL-011 → TBL-012）

Thứ tự: coupons trước (vì orders.coupon_id cần FK đích), rồi orders, rồi order_items. Ba điểm dễ sai: (1) order_items chỉ có created_at (snapshot bất biến, ngoại lệ TBL-POL-004); (2) tax_rate DECIMAL(4,2) là cột DECIMAL duy nhất toàn hệ thống (tỷ lệ, không phải tiền — TBL-POL-001); (3) composite index (user_id, status) và index created_at trên orders phục vụ màn hình lịch sử/quản lý và báo cáo tháng (09 6章).

<?php
// app/Services/OrderService.php
// 12_詳細設計書.md 5.2節: OrderService::place — 本システムの中核処理。
// 疑似コードのステップ番号（5.2-1〜5.2-13）をコメントに明記しているので必ず突き合わせて読むこと。
// クーポン（5.2-6/11）はChương 06、Stripe PaymentIntent + payments（5.2-13）はChương 07で追加する。

namespace App\Services;

use App\Exceptions\InsufficientStockException;
use App\Exceptions\ResourceNotFoundException;
use App\Models\Address;
use App\Models\Cart;
use App\Models\Inventory;
use App\Models\Order;
use App\Models\OrderItem;
use App\Models\ProductVariant;
use Illuminate\Support\Facades\DB;
use Illuminate\Support\Str;
use Illuminate\Validation\ValidationException;

class OrderService
{
    public function place(int $userId, array $data): Order
    {
        // 5.2-1: activeカートを取得、空ならE003（UT-ORD-003: 422）
        $cart = Cart::active()->where('user_id', $userId)->with('items')->first();
        if (!$cart || $cart->items->isEmpty()) {
            throw ValidationException::withMessages([
                'cart' => 'カートに商品がありません。',
            ]);
        }

        // 5.2-2: address_idが本人の住所であることを確認（他人の住所ならE007、UT-ORD-004）
        $address = Address::where('id', $data['address_id'])
            ->where('user_id', $userId)
            ->first();
        if (!$address) {
            throw new ResourceNotFoundException('対象の配送先住所が見つかりません。');
        }

        // 5.2-3: トランザクション開始（DD-POL-002: 1業務=1トランザクション）
        return DB::transaction(function () use ($userId, $cart, $address) {

            // 二重送信ガード（re-check under lock）: カート行をロックしてactiveのまま
            // であることを再確認する。同一会員が「注文確定」を2連打すると、上の5.2-1を
            // 両方のリクエストが通過し、同じカートから注文が2つ作られてしまう。
            // 先勝ちの1本が5.2-12でconvertedに変えるので、後続はここで空振りして中断する
            $cart = Cart::active()->whereKey($cart->id)->lockForUpdate()->first();
            if (!$cart) {
                throw ValidationException::withMessages([
                    'cart' => 'この注文は既に処理されています。',
                ]);
            }

            // 5.2-4: デッドロック防止のためvariant_id昇順でロックする。
            // 注文Aが(1,5)、注文Bが(5,1)の順にロックすると相互待ちになるため、順序を固定する
            //（明細もロック済みカートから読み直す — ロック前に読んだ値を信用しない）
            $cartItems = $cart->items()->orderBy('variant_id')->get();

            $lines = [];
            $subtotal = 0;
            $taxTotal = 0;

            foreach ($cartItems as $cartItem) {
                $variant = ProductVariant::with('product')
                    ->where('status', 'active')
                    ->find($cartItem->variant_id);
                if (!$variant) {
                    // カート投入後にAdminが無効化したケース。E007でロールバック
                    throw new ResourceNotFoundException('対象の商品が見つかりません。');
                }

                // SELECT ... FOR UPDATE（BR-INV-007。UNIQUE(variant_id)により行を一意特定できる）
                $inventory = Inventory::where('variant_id', $variant->id)
                    ->lockForUpdate()
                    ->first();

                // 5.2-5: 在庫不足ならE004、トランザクション全体をロールバック（UT-ORD-002）
                if (!$inventory || $cartItem->quantity > $inventory->quantity_available) {
                    throw new InsufficientStockException();
                }

                // 5.2-7: 金額計算。税率はproducts.tax_categoryから換算（BR-TAX-001）
                $taxPercent = $variant->product->tax_category === 'reduced' ? 8 : 10;
                $taxRate = $taxPercent / 100; // order_items.tax_rateスナップショット用（0.08 / 0.10）
                $lineSubtotal = $variant->price * $cartItem->quantity;
                // BR-TAX-004: 円未満切り捨て。floatの掛け算（floor($x * 0.1)等）は表現誤差で
                // 1円ズレる恐れがあるため、整数演算（intdiv）で行う — JPYが整数であることを貫く
                $lineTax = intdiv($lineSubtotal * $taxPercent, 100);
                $subtotal += $lineSubtotal;
                $taxTotal += $lineTax;

                $lines[] = [
                    'variant' => $variant,
                    'inventory' => $inventory,
                    'quantity' => $cartItem->quantity,
                    'tax_rate' => $taxRate,
                    'line_subtotal' => $lineSubtotal,
                    'line_tax' => $lineTax,
                ];
            }

            // 5.2-7続き: 送料は本スコープでは一律無料。割引はChương 06まで常に0
            $shippingFee = 0;
            $discountTotal = 0;
            $grandTotal = $subtotal + $taxTotal + $shippingFee - $discountTotal;

            // 5.2-8: ordersをstatus=pending, placed_at=now()で作成し、住所をスナップショット（BR-ORD-003）。
            // placed_atはpending作成時点で確定する（12_詳細設計書 v1.2。決済完了時刻はpayments.paid_atが持つ）
            $order = Order::create([
                // 一時値。'PENDING'のような固定文字列だと、並行する注文どうしが
                // order_numberのUNIQUE制約で衝突してINSERT自体が失敗する
                'order_number' => 'TMP-' . Str::uuid(),
                'user_id' => $userId,
                'status' => 'pending',
                'subtotal' => $subtotal,
                'tax_total' => $taxTotal,
                'shipping_fee' => $shippingFee,
                'discount_total' => $discountTotal,
                'grand_total' => $grandTotal,
                'coupon_id' => null,
                'shipping_postal_code' => $address->postal_code,
                'shipping_prefecture' => $address->prefecture,
                'shipping_city' => $address->city,
                'shipping_address_line' => $address->address_line,
                'shipping_recipient_name' => $address->recipient_name,
                'shipping_phone' => $address->phone,
                'placed_at' => now(),
            ]);

            // 注文番号の採番: 10_API設計.md 6.15節の例「ORD-20260713-0501」に合わせる（日付+id 4桁）
            $order->update([
                'order_number' => sprintf('ORD-%s-%04d', now()->format('Ymd'), $order->id),
            ]);

            // 5.2-9: order_itemsへスナップショット（BR-ORD-002, BR-TAX-002）。
            // この瞬間に商品名・単価・税率が「凍結」される — UT-ORD-005が検証する核心部分
            foreach ($lines as $line) {
                OrderItem::create([
                    'order_id' => $order->id,
                    'variant_id' => $line['variant']->id,
                    'product_name' => $line['variant']->product->name,
                    'variant_label' => implode(' / ', array_filter([
                        $line['variant']->size, $line['variant']->color,
                    ])),
                    'sku' => $line['variant']->sku,
                    'unit_price' => $line['variant']->price,
                    'tax_rate' => $line['tax_rate'],
                    'quantity' => $line['quantity'],
                    'line_subtotal' => $line['line_subtotal'],
                    'line_tax' => $line['line_tax'],
                    'line_total' => $line['line_subtotal'] + $line['line_tax'],
                ]);
            }

            // 5.2-10: 在庫確保（BR-INV-003）。available→reservedへの振り替えのみ。
            // inventory_logsはまだ書かない — 確定消費の記録はWebhook受信時（BR-INV-004、Chương 07）
            foreach ($lines as $line) {
                $line['inventory']->update([
                    'quantity_available' => $line['inventory']->quantity_available - $line['quantity'],
                    'quantity_reserved' => $line['inventory']->quantity_reserved + $line['quantity'],
                ]);
            }

            // 5.2-11: クーポンused_count += 1 — Chương 06で追加
            // 5.2-12: カートをconvertedへ（次回のカート追加時に新しいactiveカートが自動作成される、5.1節）
            $cart->update(['status' => 'converted']);

            // 5.2-13: Stripe PaymentIntent作成 + payments(status=pending)作成 — Chương 07で追加

            return $order->load('items');
        });
    }
}
