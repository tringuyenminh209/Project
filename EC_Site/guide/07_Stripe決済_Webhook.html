Chương 07 — Thanh toán Stripe (PaymentIntent + Webhook)

Chương nặng nhất sau Chương 04, và là nơi đơn hàng lần đầu chạy hết vòng đời: pending → paid. Nội dung: bảng payments (TBL-013), tạo PaymentIntent ở bước 5.2-13 (mảnh cuối cùng còn gác của OrderService::place), và Webhook API-017 theo pseudocode 5.5節 — xác thực chữ ký, idempotency, chốt tồn kho (BR-INV-004). Cặp API-015 + API-017 chính là "2 API trung tâm" mà 10_API設計 11章 tuyên bố cả tài liệu xoay quanh.

⚠️ Điều kiện vào chương

Cần tài khoản Stripe (miễn phí) + Stripe CLI để forward Webhook về localhost. Điền STRIPE_KEY / STRIPE_SECRET (test mode, tiền tố pk_test_/sk_test_) vào backend/.env — khung đã chừa sẵn từ Chương 01. STRIPE_WEBHOOK_SECRET sẽ lấy từ lệnh stripe listen ở bước 2. Tuyệt đối không commit các key này (INF-POL-003).

💡 Vì sao Webhook quyết định trạng thái đơn, không phải callback của frontend? (BR-PAY-001)

Sau khi khách bấm thanh toán, frontend cũng biết kết quả — nhưng frontend là môi trường của người dùng: có thể tắt tab giữa chừng (mất callback), hoặc bị chỉnh sửa để gửi "thành công" giả. Nếu server tin callback đó thì kẻ gian có đơn paid miễn phí. Webhook là kênh server-to-server từ Stripe, có chữ ký Stripe-Signature chống mạo danh (NFR-017): tiền thật đã chạy hay chưa, chỉ nguồn này được phép nói. Frontend callback chỉ dùng để... chuyển màn hình sang SCR-008 cho mượt.

💡 Idempotency — vì sao check status trước khi xử lý? (BR-PAY-003)

Stripe cam kết gửi lại event khi không chắc bạn đã nhận (network flaky, timeout) — nghĩa là cùng một payment_intent.succeeded có thể đến 2-3 lần. Không có chốt chặn thì mỗi lần đến là một lần trừ quantity_reserved → kho âm. Chốt chặn 2 lớp: UNIQUE trên stripe_payment_intent_id (định danh đúng 1 dòng payments) + kiểm tra status ∈ {captured, failed} thì bỏ qua ngay (5.5-3). UT-PAY-003 gọi đây là 核心試験 thứ hai của project.

💡 Vì sao lỗi nội bộ trong Webhook vẫn trả 200? (thiết kế "ngược đời" có chủ đích)

Trực giác bảo: lỗi thì trả 500 để Stripe gửi lại. Nhưng nếu lỗi là bug vĩnh viễn (không phải lỗi thoáng qua), Stripe sẽ nã lại event đó theo backoff tối đa 3 ngày — mỗi lần đều nổ đúng chỗ cũ, log ngập, không tự lành. 12_詳細設計書 11章 chọn: bắt Throwable, ghi log đầy đủ event_id, trả 200, và giao phần "chữa" cho quy trình thủ công trong 20_運用保守手順書. Chỉ duy nhất lỗi chữ ký (E009) trả 400 — vì đó là request không đáng tin, không phải lỗi của ta. Đánh đổi: mất khả năng tự retry của Stripe → đổi lấy hệ thống không tự tra tấn mình.

💡 Vì sao gọi Stripe API BÊN TRONG transaction của place()?

Nguyên tắc chung là không gọi external API trong DB transaction (giữ lock lâu). Nhưng 12_詳細設計書 9.1節 vẽ rõ thứ tự: PaymentIntent tạo trước COMMIT. Lý do: nếu tạo PaymentIntent thất bại mà đơn đã commit, ta có "đơn pending không có phương tiện thanh toán" — khách kẹt, kho bị giam 30 phút vô nghĩa. Đặt trong transaction thì thất bại → rollback sạch cả đơn. Đánh đổi giữ lock thêm vài trăm ms được chặn trên bởi NFR-003 (Stripe call timeout 5 giây). Đây là ví dụ tốt về "nguyên tắc chung" thua "quyết định thiết kế có căn cứ".

<?php
// app/Services/PaymentService.php
// 12_詳細設計書.md 3章: PaymentService — refund（5.8.1節、Chương 09）。
// PaymentIntent作成はOrderService::place(5.2-13)とAPI-016(再試行)の両方が使うため、
// 共通メソッドcreateIntentForOrderとしてここに置く（同一ロジックの二重実装禁止）

namespace App\Services;

use App\Exceptions\PaymentFailedException;
use App\Models\Order;
use App\Models\Payment;
use Illuminate\Support\Facades\Log;
use Stripe\Exception\ApiErrorException;
use Stripe\StripeClient;

class PaymentService
{
    private StripeClient $stripe;

    public function __construct()
    {
        $this->stripe = new StripeClient(config('services.stripe.secret'));
    }

    // PaymentIntentを作成し、payments行(status=pending)を追加する
    public function createIntentForOrder(Order $order): array
    {
        try {
            $intent = $this->stripe->paymentIntents->create([
                // JPYは小数を持たない通貨のため、金額を「そのままの円」で渡す（BR-TAX-003。
                // USD等はセント単位に変換が必要だが、JPYは変換不要 — Stripeのzero-decimal currency）
                'amount' => $order->grand_total,
                'currency' => 'jpy',
                // capture_methodは既定のautomatic（02_要件定義書 10.2節: authorizedを永続化しない）
                'automatic_payment_methods' => ['enabled' => true],
                'metadata' => [
                    'order_id' => $order->id,
                    'order_number' => $order->order_number,
                ],
            ]);
        } catch (ApiErrorException $e) {
            // PaymentIntent作成自体の失敗はE008（12_詳細設計書 11章）。
            // 呼び出し元（place）のトランザクション内で投げられるため、注文ごとロールバックされる
            Log::channel('single')->error('PaymentIntent作成失敗', [
                'order_id' => $order->id,
                'error' => $e->getMessage(),
            ]);
            throw new PaymentFailedException();
        }

        Payment::create([
            'order_id' => $order->id,
            'payment_method' => 'credit_card',
            'status' => 'pending',
            'amount' => $order->grand_total,
            'stripe_payment_intent_id' => $intent->id, // BR-PAY-003 冪等性キー
        ]);

        return [
            'payment_intent_id' => $intent->id,
            'client_secret' => $intent->client_secret, // フロントのStripe Elementsへ渡す（SCR-007）
        ];
    }
}

<?php
// app/Http/Controllers/WebhookController.php
// 10_API設計.md 6.17節(API-017) / 12_詳細設計書.md 8章:
// 「署名検証はWebhookController内でStripe SDKのWebhook::constructEventを直接使用」
// （Middleware化せず、Stripeシークレットの取り回しをController内に閉じる）

namespace App\Http\Controllers;

use App\Exceptions\WebhookSignatureInvalidException;
use App\Services\PaymentWebhookService;
use Illuminate\Http\Request;
use Illuminate\Http\Response;
use Illuminate\Support\Facades\Log;
use Stripe\Exception\SignatureVerificationException;
use Stripe\Webhook;

class WebhookController extends Controller
{
    // POST /webhooks/stripe (API-017)
    public function handleStripe(Request $request, PaymentWebhookService $service): Response
    {
        // 5.5-1: 署名検証。不正なら400（E009）で即終了し、DBには一切触れない（UT-PAY-004）。
        // 生のリクエストボディが必要（JSONデコード後だと署名が一致しない — ハマりどころ）
        try {
            $event = Webhook::constructEvent(
                $request->getContent(),
                $request->header('Stripe-Signature', ''),
                config('services.stripe.webhook_secret')
            );
        } catch (SignatureVerificationException | \UnexpectedValueException $e) {
            throw new WebhookSignatureInvalidException();
        }

        try {
            $service->handleStripeEvent($event);
        } catch (\Throwable $e) {
            // 10_API設計 9章 / 12_詳細設計書 11章: 内部エラーでも200を返す設計。
            // 500を返すとStripeが最大3日間再送し続け、恒久バグの場合は同一イベントが
            // 際限なく叩かれるため、ログに記録して手動リカバリ（20_運用保守手順書）へ委ねる
            Log::error('Webhook処理内部エラー', [
                'event_id' => $event->id,
                'type' => $event->type,
                'error' => $e->getMessage(),
            ]);
        }

        // 200 OK（ボディは空でよい、6.17節）。Webhook処理は5秒以内に応答する（9章）
        return response()->noContent(200);
    }
}
app/Services/PaymentWebhookService.phpphp
<?php
// app/Services/PaymentWebhookService.php
// 12_詳細設計書.md 5.5節: PaymentWebhookService::handleStripeEvent
// （BR-PAY-001〜003 / BR-INV-004〜005）。署名検証(5.5-1)はWebhookController側（12 8章）。

namespace App\Services;

use App\Models\Inventory;
use App\Models\InventoryLog;
use App\Models\Order;
use App\Models\Payment;
use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Log;
use Stripe\Event;

class PaymentWebhookService
{
    public function handleStripeEvent(Event $event): void
    {
        $intent = $event->data->object; // PaymentIntentオブジェクト

        // 5.5-2: stripe_payment_intent_idでpayments行を特定。
        // 見つからなければログのみ記録して正常終了（自システム管轄外のイベント）
        $payment = Payment::where('stripe_payment_intent_id', $intent->id)->first();
        if (!$payment) {
            Log::info('Webhook: 対象外のPaymentIntent', ['pi' => $intent->id, 'type' => $event->type]);
            return;
        }

        // 5.5-3: 冪等性チェック（BR-PAY-003）。同一イベントは複数回配信され得る。
        // 既にcaptured/failedの確定状態なら何もせず終了 — これが「二重の在庫減算」を防ぐ核心
        if (in_array($payment->status, ['captured', 'failed'], true)) {
            Log::info('Webhook: 冪等性スキップ', ['pi' => $intent->id, 'status' => $payment->status]);
            return;
        }

        if ($event->type === 'payment_intent.succeeded') {
            $this->handleSucceeded($payment);
        } elseif ($event->type === 'payment_intent.payment_failed') {
            // 5.5-5: BR-PAY-002 — 決済失敗は「注文キャンセル」と区別する。
            // orders.statusはpendingのまま維持し再決済を許可。在庫・クーポンも解放しない
            //（解放はChương 05のバッチが期限切れ判定で行う）
            $payment->update(['status' => 'failed']);
        }
    }

    // 5.5-4: 決済成功 → 注文確定・在庫引き当て確定
    private function handleSucceeded(Payment $payment): void
    {
        DB::transaction(function () use ($payment) {
            // バッチ（CancelExpiredPendingOrders）との競合に備え、注文行をロックして状態を確認する。
            // Chương 05では「バッチ側がロック下で再確認」した — こちらはその鏡像
            $order = Order::whereKey($payment->order_id)->lockForUpdate()->first();

            // 5.5-4b
            $payment->update(['status' => 'captured', 'paid_at' => now()]);

            if (!$order || $order->status !== 'pending') {
                // 期限切れバッチが一足先にcancelledへ倒した後に決済成功が届いたケース。
                // 「入金済みだがキャンセル済み」の状態であり、自動では解決しない —
                // ログに記録し、返金の手動対応へ回す（20_運用保守手順書）
                Log::error('Webhook: 決済成功したが注文がpendingではない（要手動返金）', [
                    'order_id' => $payment->order_id,
                    'status' => $order?->status,
                ]);
                return;
            }

            // 5.5-4c: pending → paid。placed_atは注文確定時に設定済みのため変更しない（12 v1.2）
            $order->update(['status' => 'paid']);

            // 5.5-4d: 在庫引き当て確定（BR-INV-004）。
            // quantity_availableは注文確定時（BR-INV-003）に既に減算済み — ここではreservedのみ減らし、
            // inventory_logsに「確定消費」を記録する。ロック順はvariant_id昇順（5.2-4と同一規約）
            foreach ($order->items()->orderBy('variant_id')->get() as $item) {
                $inventory = Inventory::where('variant_id', $item->variant_id)
                    ->lockForUpdate()
                    ->first();
                if (!$inventory) {
                    continue;
                }

                $inventory->update([
                    'quantity_reserved' => $inventory->quantity_reserved - $item->quantity,
                ]);

                InventoryLog::create([
                    'variant_id' => $item->variant_id,
                    'change_type' => 'purchase_deduct',
                    'quantity_change' => -$item->quantity, // −N: 出庫（TBL-008 符号規約）
                    // availableはこの操作では変化しない（確保時に減算済み）ため現在値をそのまま記録
                    'balance_after' => $inventory->quantity_available,
                    'reference_type' => 'orders',
                    'reference_id' => $order->id,
                    'created_by' => null, // System起因
                ]);
            }
        });
    }
}
