# 1. 本書の目的

本書は08_ER図の論理設計を、MySQL 8上で実行可能な物理テーブル定義に落とし込む。本書の型・制約定義がそのまま`ddl.sql`（Migration）の実装根拠となる。

---

# 2. テーブル設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| TBL-POL-001 | 金額はINTEGER | JPYは小数を持たないため、金額列はすべて`INT UNSIGNED`（またはINT）とし、DECIMALは使用しない（BR-TAX-003）。例外は`tax_rate`（比率であり金額ではない）のみDECIMAL(4,2)を使用する。 |
| TBL-POL-002 | スナップショットは非正規化を許容 | `order_items`と`orders`の一部列は、正規化を崩してでも過去データの不変性を優先する（BR-ORD-002/003）。 |
| TBL-POL-003 | 論理削除優先 | `products` `categories` `product_variants` `coupons`は`status`列による無効化を基本とし、物理削除しない。 |
| TBL-POL-004 | 全テーブルにタイムスタンプ | `created_at`（+更新がありうるテーブルは`updated_at`）を必須とする。例外3件、いずれも「更新されない行」であるため`updated_at`を省略: `inventories`（`updated_at`のみ、`created_at`相当は初回作成時と同時刻のため省略）、`inventory_logs`（`created_at`のみ、追記専用ログのため不変）、`order_items`（`created_at`のみ、注文確定時のスナップショットで以後不変・TBL-POL-002参照）。 |
| TBL-POL-005 | 外部キーは物理制約で強制 | Laravel側のリレーション定義だけでなく、DBのFOREIGN KEY制約でも整合性を強制する。 |
| TBL-POL-006 | 外部キー命名の例外 | 原則`[参照先単数形]_id`だが、`variant_id`（`product_variants`参照）と`created_by`（`users`参照、監査目的）の2パターンは例外として許容する。詳細は10章「命名規則の例外」参照。 |

# 5. 外部キー制約

| 子テーブル | 子カラム | 親テーブル | 親カラム | ON DELETE |
| --- | --- | --- | --- | --- |
| addresses | user_id | users | id | CASCADE |
| categories | parent_category_id | categories | id | SET NULL |
| products | category_id | categories | id | RESTRICT |
| product_variants | product_id | products | id | RESTRICT |
| product_images | product_id | products | id | CASCADE |
| inventories | variant_id | product_variants | id | RESTRICT |
| inventory_logs | variant_id | product_variants | id | RESTRICT |
| inventory_logs | created_by | users | id | SET NULL |
| carts | user_id | users | id | CASCADE |
| cart_items | cart_id | carts | id | CASCADE |
| cart_items | variant_id | product_variants | id | RESTRICT |
| orders | user_id | users | id | RESTRICT |
| orders | coupon_id | coupons | id | SET NULL |
| order_items | order_id | orders | id | CASCADE |
| order_items | variant_id | product_variants | id | RESTRICT |
| payments | order_id | orders | id | CASCADE |
| shipments | order_id | orders | id | CASCADE |
| reviews | product_id | products | id | RESTRICT |
| reviews | user_id | users | id | RESTRICT |
| reviews | order_item_id | order_items | id | RESTRICT |

**方針**: 注文・決済・レビューなど「発生した事実」に関わるFKは`RESTRICT`（親を消せなくする）とし、履歴の消失を防ぐ。カート・住所など「利用者の一時的な入れ物」は`CASCADE`とし、会員退会時に連動して整理される。

# 11. 設計判断（設計判断メモ）

Phase 2 DB設計特訓（`plan/02_PHASE2_DB_DESIGN.md`）の要求に基づき、実装前に判断した主要な設計決定を記録する。

1. **在庫の減算タイミング**: カート追加時ではなく注文確定時に確保（reserve）し、決済確定時に引き当て確定（commit）する2段階方式を採用した。理由: カート追加時点で減算すると「見ているだけ」のユーザーが在庫を占有してしまい機会損失が大きい。逆に決済完了まで一切減らさないと、決済中の売り切れリスクが高い。実務上最も一般的な折衷案である「確保→確定」を採用した（9.3節BR-INV参照）。

2. **1カート=1 active制約をDBで強制しない**: MySQLは条件付きUNIQUE制約（WHERE句付きINDEX）を標準サポートしないため、「同一user_idでstatus=activeのカートは1件のみ」をDB制約として表現できない。生成カラム＋UNIQUE制約で擬似的に実現する方法もあるが、複雑性に見合わないと判断し、Service層でトランザクション内`SELECT ... FOR UPDATE`により保証する方針とした。

3. **payments を1注文1行ではなく1注文N行にした**: 決済失敗後の再試行（BR-PAY-002）を素直に表現するため。1行に上書きする設計も可能だが、失敗履歴が消えてしまい障害調査がしづらくなるため履歴型を選んだ。
