■ 1. 本書の目的

実際のブラウザ操作に近い形で、Login〜購入〜配送〜レビューまでの一連の業務フローが期待通り動作することをシステム全体として確認する。

■ 2. 試験方針
方針ID方針内容
ST-POL-001E2E実際の画面操作の連鎖で確認する。
ST-POL-002Role BasedGuest / Customer / Adminそれぞれの視点でシナリオを作成する。
ST-POL-003Recovery決済失敗・在庫切れ等の異常系からの復旧動線を確認する。
■ 3. 試験環境

Docker Compose環境（Local）またはAWS Development環境。Stripe Sandbox、S3（またはMinIO代替）を使用する。

■ 4. E2Eシナリオ試験
ST-SCN-001 Guest→購入完了
1. Guestとして商品一覧を閲覧する
2. 商品詳細でバリエーションを選択しカートに追加を試みる → ログイン画面へ誘導される
3. 会員登録する
4. 再度カートに追加する
5. カート画面で数量を確認し注文手続きへ進む
6. 配送先を新規登録し、クーポンコードを適用する
7. 注文を確定する
8. Stripeテストカード（成功用）で決済する
9. 注文完了画面が表示され、注文番号が発行される
10. 注文履歴で該当注文がpaidステータスで確認できる
期待結果: 全ステップが例外なく完了し、在庫が正しく減算されている
ST-SCN-002 Admin出荷〜Customer受け取り確認
1. Adminでログインし、ST-SCN-001の注文を確認する
2. ステータスをshippedに変更し、配送業者・追跡番号を登録する
3. ステータスをdeliveredに変更する
4. Customerで再ログインし、注文詳細で配送状況を確認する
5. 対象商品のレビューを投稿する
期待結果: 配送状況が正しく反映され、レビュー投稿が成功する
ST-SCN-003 決済失敗からの復旧
1. Customerが注文確定まで進む
2. Stripeテストカード（拒否用）で決済を試みる → 失敗メッセージが表示される
3. 注文はpendingのまま在庫が確保され続けていることを確認する
4. 再度Stripeテストカード（成功用）で決済する
期待結果: 再決済が成功し、orders.statusがpaidになる
ST-SCN-004 在庫切れによる購入不可
1. Adminで対象商品の在庫を0に調整する
2. Customerが同一商品をカートに追加しようとする
期待結果: E004が表示され、カートに追加できない
