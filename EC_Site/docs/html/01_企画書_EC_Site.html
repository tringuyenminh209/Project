■ 1. 企画背景

Portfolio Roadmap（SE/BrSE 2026-2027）における10プロジェクト構想のうち、Project 02 として位置づけられる案件。Project 01（HR & Attendance System）で CRUD・認証・認可の基礎を固めた後、実務で最も需要の高いドメインの一つである EC（電子商取引）システムに取り組み、「お金」と「在庫」という Junior 止まりのエンジニアが避けて通りがちな設計課題に正面から取り組む。

日本のIT企業（特にBrSEとして働く場合）では、EC・決済連携案件の引き合いが多く、税率・注文状態・在庫という業務ルールを正しく仕様化し日本側と合意形成できる人材が求められる。

■ 2. 現状の課題
  • Project 01 では単一テーブルの状態遷移（休暇申請の Pending→Approved/Rejected）のみを扱っており、複数テーブルにまたがるトランザクション設計の経験が不足している
  • 「今の価格」と「過去に買った時の価格」を区別する設計（スナップショット）を経験していない
  • 在庫のような有限リソースを複数ユーザーが同時に奪い合う状況（排他制御）を経験していない
  • 外部決済サービス（Stripe等）との連携パターン（Webhook、非同期ステータス更新）を経験していない
■ 3. 企画目的

上記の課題を解消するため、以下を満たす EC サイトを自主開発する。

  1. 会員が商品を検索・購入し、決済を完了できる
  2. 在庫が二重販売されない（同時アクセス下でも整合性を保つ）
  3. 注文確定時点の価格・税率が保存され、後から商品価格を変更しても過去の注文に影響しない
  4. 管理者が商品・在庫・注文・クーポンを一元管理できる
■ 4. 対象ユーザー

Guest（未ログイン閲覧者）、Customer（会員）、Admin（管理者）の3種別。詳細は 00_開発計画書 3章、02_要件定義書 3章を参照。

■ 6. 解決方針
課題解決方針
過去の注文価格が変動するorder_items に商品名・単価・税率をスナップショットとして保存し、products/product_variants への参照は分析目的のFKに留める
在庫の二重販売注文時に quantity_reserved へ振り替える2段階制御 + 悲観的ロック（SELECT ... FOR UPDATE）
決済との整合性Stripe PaymentIntent + Webhook。DB更新は Webhook 受信をトリガーとし、フロントの決済成功表示のみで確定させない
税率の混在（軽減税率8%/標準10%）products.tax_category で管理し、order_items.tax_rate に確定値をスナップショット
■ 7. システム概要

単一事業者運営（マーケットプレイスではない）の B2C EC サイト。会員登録した消費者が商品を検索・カートに入れ・注文・決済し、管理者が商品/在庫/注文/クーポンを管理する。詳細構成は 00_開発計画書 4章の構成図を参照。

■ 8. 主要機能

会員機能・商品閲覧・カート・注文・決済（Stripe）・在庫管理・クーポン・レビュー・管理者機能の9系統。詳細は 07_機能一覧 を参照。

■ 9. 開発範囲
9.1 開発対象
  • 会員登録・ログイン・ログアウト（認証）
  • 商品一覧・詳細閲覧（カテゴリ、バリエーション対応）
  • カート機能（追加・数量変更・削除）
  • 注文（配送先入力・クーポン適用・注文確定）
  • 決済（Stripe Sandbox 連携、決済ステータス管理）
  • 在庫管理（在庫確保・確定引き当て・在庫ログ）
  • 注文管理（管理者：ステータス変更、出荷登録）
  • 商品管理（管理者：商品・カテゴリ・バリエーション・画像登録）
  • クーポン管理（管理者：発行・利用条件設定）
  • レビュー投稿（購入者のみ、星評価＋コメント）
  • 管理者向け売上・在庫レポート
9.2 開発対象外
  • マーケットプレイス機能（複数出品者・出品者向けダッシュボード）
  • 定期購入（サブスクリプション）
  • 多通貨対応（JPYのみ）
  • 実際の配送業者API連携（配送ステータスは手動更新でシミュレート）
  • ポイント／会員ランク制度
  • 多言語UI（日本語のみ、コメント・内部文書はベトナム語併記）

00_開発計画書 2.1/2.2節と同一。

■ 15. リスクと対策
リスク種別リスク内容影響度発生確率対策
技術リスクStripe Webhookの署名検証・非同期処理に不慣れ中中Stripe公式ドキュメントでPoC実施後に本実装
設計リスク在庫の排他制御（同時注文）の考慮漏れ高中トランザクション + 悲観的ロック（SELECT ... FOR UPDATE）を採用
スケジュールリスクProject 01との並行進行で時間確保が困難中高設計書は先行作成済み、実装はProject 01完了後に着手
データ整合性リスク商品価格変更が過去注文に波及する高低（設計で防止）order_itemsへのスナップショット必須ルール化

詳細な担当・進捗管理は00_開発計画書11章を参照。

■ 16. まとめ

本企画は、Project 01 で得た Laravel 実装力を土台に、「お金」「在庫」「時間経過するデータ」という新しい設計軸を追加する EC サイト構築である。特に価格スナップショットと在庫の2段階制御は、要件定義書・詳細設計書において明確な設計判断として記録し、実装時に迷わず参照できる状態を整える。
