■ 1. 開発目的

本プロジェクトは、Portfolio Roadmap（SE/BrSE 2026-2027）における Project 02 として位置づけられる、単一事業者運営の汎用ECサイト（B2C）である。

Project 01（HR & Attendance System）が CRUD・認証・認可という「静的なマスタ管理」を主軸としたのに対し、本プロジェクトは以下の点で難易度を一段階引き上げることを目的とする。

  • お金が動く処理（注文・決済・返金）の正確な設計と実装
  • 在庫という有限リソースの排他制御（二重販売の防止）
  • 時間とともに変化するデータ（商品価格・税率）を過去の注文にどう固定するか（スナップショット設計）
  • 外部サービス連携（Stripe決済、AWS S3画像保存）

これらは実務における EC / SaaS 課金系システムで頻出する設計判断であり、Junior → Mid レベルへのステップアップに直結する。

■ 2. 開発範囲
2.1 開発対象（In Scope）
  • 会員登録・ログイン・ログアウト（認証）
  • 商品一覧・詳細閲覧（カテゴリ、バリエーション対応）
  • カート機能（追加・数量変更・削除）
  • 注文（配送先入力・クーポン適用・注文確定）
  • 決済（Stripe Sandbox 連携、決済ステータス管理）
  • 在庫管理（在庫確保・確定引き当て・在庫ログ）
  • 注文管理（管理者：ステータス変更、出荷登録）
  • 商品管理（管理者：商品・カテゴリ・バリエーション・画像登録）
  • クーポン管理（管理者：発行・利用条件設定）
  • レビュー投稿（購入者のみ、星評価＋コメント）
  • 管理者向け売上・在庫レポート
2.2 開発対象外（Out Scope）
  • マーケットプレイス機能（複数出品者・出品者向けダッシュボード）
  • 定期購入（サブスクリプション）
  • 多通貨対応（JPYのみ）
  • 実際の配送業者API連携（配送ステータスは手動更新でシミュレート）
  • ポイント／会員ランク制度
  • 多言語UI（日本語のみ、コメント・内部文書はベトナム語併記）
■ 3. 対象ユーザー
ユーザー種別説明
Guest（未ログイン）商品閲覧のみ可能。カート・注文には会員登録が必須
Customer（会員）商品購入、カート、注文履歴、レビュー投稿を行う一般利用者
Admin（管理者）商品・在庫・注文・クーポンを管理する事業者側スタッフ

Project 01 のような Manager 層（承認者）は本システムには存在しない。EC サイトの承認フローは「注文の状態遷移（pending→paid→shipped→delivered）」自体が担うため、別ロールの承認者を置く設計にはしない（4章の業務ルールで詳述）。

■ 5. 主な機能
5.1 会員機能

登録・ログイン・ログアウト・パスワード変更。Sanctum によるトークン認証（Project 01 と同じ方式を踏襲し、既習の知識を再利用する）。

5.2 商品閲覧機能

カテゴリ別一覧、キーワード検索、商品詳細（バリエーション選択で価格・在庫が動的に変わる）。

5.3 カート機能

会員ごとに1つの active カートを持つ。バリエーション単位で数量管理。

5.4 注文機能

カート→配送先入力→クーポン適用→注文確定。確定時点の商品名・価格・税率を order_items にスナップショットする。

5.5 決済機能

Stripe Sandbox の PaymentIntent を利用。Webhook で決済結果を受信し orders.status を更新する。

5.6 在庫管理機能

「注文時に確保 → 決済確定で引き当て確定 → 決済失敗/キャンセルで解放」という2段階の在庫制御（詳細は 02_要件定義書 9章）。

5.7 クーポン機能

金額固定 / 割合の2種類。有効期間・利用上限・最低購入金額の条件判定。

5.8 レビュー機能

購入済み（order_items に該当レコードがある）商品のみレビュー可能（なりすまし・過剰投稿の防止）。

5.9 管理者機能

商品/カテゴリ/バリエーション/画像 CRUD、在庫調整、注文ステータス変更、クーポン発行、売上レポート。

■ 9. データベース概要
テーブル一覧（16テーブル）

users, addresses, categories, products, product_variants, product_images, inventories, inventory_logs, carts, cart_items, orders, order_items, payments, shipments, coupons, reviews

Portfolio Roadmap 記載の11テーブル案（inventory_logs を含む）と、Phase 2 DB設計特訓の15テーブル案（addresses / product_variants / product_images / inventories / shipments を含む）を統合した構成。詳細は 08_ER図 および 09_テーブル定義 を参照。

ER概要

中心エンティティは orders。users → carts → orders という購買導線と、products → product_variants → inventories という在庫導線の2系統が order_items で交差する構造。詳細は 08_ER図.md 参照。

■ 10. テスト計画
単体試験

Service層のビジネスロジック（税額計算、在庫確保/引き当て、クーポン割引計算、注文合計計算）を中心に PHPUnit で実施。

結合試験

API ↔ DB、Stripe Webhook ↔ 注文ステータス更新、の連携を確認。

システム試験

Guest閲覧→会員登録→カート→注文→決済→管理者出荷登録→会員が配送状況確認、という一連のシナリオを通しで実施。

テスト項目（概要）
  • 在庫0商品のカート追加拒否
  • 同時注文による在庫マイナス化の防止（排他制御）
  • クーポン利用上限到達後の拒否
  • 決済失敗時の在庫解放
  • 注文確定後の商品価格変更が過去注文に影響しないこと（スナップショット検証）

詳細は 15/16/17 の各試験仕様書を参照。

■ 14. まとめ

本プロジェクトは Project 01 で習得した Laravel の基本アーキテクチャ（Controller/Service/Model分離、FormRequest、Exception設計、RBAC）を土台としながら、「お金」「在庫」「時間経過によるデータ変化」という新しい難易度軸を追加する。特に価格スナップショットと在庫の2段階制御は、実務のEC/SaaS開発で必ず問われる設計判断であり、本プロジェクトの中核的な学習目標である。
