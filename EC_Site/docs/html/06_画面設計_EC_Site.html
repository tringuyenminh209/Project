■ 1. 本書の目的

EC Siteの各画面について、表示項目、入力項目、操作、バリデーション、エラー、画面遷移を定義する。02_要件定義書 03_ユースケース 05_画面遷移図 と整合し、10_API設計 15_単体試験仕様書 の基準とする。

■ 5. 画面詳細
5.0 状態バリエーション 共通ルール

v1.2でSCR-003/004に試験適用した書式を全20画面へ展開する。各画面の「状態バリエーション」表は以下の共通ルールを前提とし、画面固有の差分のみを記載する（2.1節のトークン名で表記を統一）。

共通ルール内容
Loading（取得系）一覧・詳細の初期表示はスケルトン（Border色の枠 + Background色のバー）で件数分プレースホルダー表示し、API応答後に差し替える
Loading（送信系）送信ボタンは2.1節Buttonトークンのloading状態（スピナー + 非活性）にし、二重送信を防止する。応答までフォーム入力も非活性にする
Empty0件は正常応答として扱い、Error表示と混同しない。「なぜ0件か + 次に取るべき行動」をText-secondaryで案内する
Error（通信失敗）「取得に失敗しました。」+ 再試行ボタン（SCR-003のError書式と同一）
Error（業務エラー）E001〜E011は7章のメッセージをError色で表示する。入力項目起因（E003）は該当Inputのerror状態 + 直下赤字（2.1節Inputトークン）
5.1 SCR-001 ログイン画面

表示項目

  • サイトロゴ、会員登録画面（SCR-002）への導線リンク

入力項目

  • メールアドレス（Required / Email形式 / Max 255）
  • パスワード（Required / 8〜20文字）

操作

  • 「ログインする」→ API-002を呼び出し、成功時はCustomerはSCR-003（商品一覧）、AdminはSCR-013（管理者ダッシュボード）へ遷移。Progressive Login経由の場合は元の操作画面（例: SCR-004）に戻る。
  • 「新規会員登録はこちら」→ SCR-002へ遷移

エラー

  • メールアドレス/パスワード不一致 → E001（アカウントの有無を推測されないよう「どちらが誤りか」は明示しない、14_セキュリティ設計4.3節と一致）
  • 入力形式不正（未入力・Email形式違反等）→ E003

画面遷移

  • Progressive Login（UI-002）によりSCR-004（商品詳細）でカート追加を試みた未ログインGuestは本画面へ遷移し、ログイン成功後は元の操作を継続する。

状態バリエーション

状態表示
Loading「ログインする」押下後はButton loading状態（共通ルール）。API-002応答まで再送信不可
Empty対象外（一覧を持たない画面）
ErrorE001/E003をフォーム上部に表示。E001はどちらの項目が誤りかを明示しない（14_セキュリティ設計4.3節）
5.2 SCR-002 会員登録画面

表示項目

  • サイトロゴ、ログイン画面（SCR-001）への導線リンク

入力項目

  • 氏名（Required / Max 100）
  • メールアドレス（Required / Email形式 / Max 255 / Unique）
  • パスワード（Required / 8〜20文字）
  • パスワード確認（Required / パスワードと一致）

操作

  • 「登録する」→ API-001を呼び出し、成功時は自動ログイン状態でSCR-003（商品一覧）へ遷移（会員登録はCustomerのみが行う操作のため、Admin向けSCR-013には遷移しない）
  • 「ログインはこちら」→ SCR-001へ遷移

エラー

  • メールアドレス重複 → E003（10_API設計 API-001のUnique制約違反、422）
  • 入力形式不正・パスワード不一致 → E003

状態バリエーション

状態表示
Loading「登録する」押下後はButton loading状態
Empty対象外
ErrorE003は該当Inputのerror書式で直下に表示。メール重複時は「このメールアドレスは既に登録されています」
5.3 SCR-003 商品一覧画面

表示項目

  • 商品カード（サムネイル画像、商品名、価格帯「¥2,000〜」、カテゴリ）
  • カテゴリ絞込メニュー、キーワード検索窓、ページネーション

検索条件

状態バリエーション

状態表示
Loading商品カードをグレーのスケルトン（画像枠＋2行のバー）で件数分プレースホルダー表示、API-006応答後に差し替える
Empty検索/絞込の結果が0件のとき「該当する商品が見つかりませんでした。条件を変更してください。」+ 絞込リセットボタン（初回アクセス時の全件0件はマスタ未登録を意味するため対象外）

5.4 SCR-004 商品詳細画面

表示項目

  • 商品画像（複数、product_images.display_order順）
  • 商品名、説明、価格（選択バリエーションに応じて動的に変わる）
  • バリエーション選択（サイズ・色のプルダウン、skuごとの在庫状況）
  • レビュー平均・件数、レビュー一覧

入力・バリデーション

  • バリエーション：Required（未選択でカート追加不可）
  • 数量：Required / Integer / 1以上 / 在庫数以下

操作

  • 「カートに追加」→ SCR-005へ遷移（またはトースト表示で一覧に留まる）
  • 未ログイン時にカート追加を押すとSCR-001へ遷移（Progressive Login）

状態バリエーション

状態表示
Loading画像枠・商品名・価格の3ブロックをスケルトン表示、API-007応答後に差し替える
Emptyレビュー0件時は一覧を非表示にせず「まだレビューがありません」をText-secondaryで表示（本体の商品情報は0件対象外）
Error存在しない/非公開の商品ID（E007）→ 商品情報を出さず「商品が見つかりませんでした」+ SCR-003へ戻る導線。API通信失敗（ネットワーク断等、業務エラーコードなし）→ 再試行ボタン
  • 在庫0のバリエーション選択時：「カートに追加」ボタンをdisabled表示にし、「在庫切れ」をWarning色でバリエーション横に表示（在庫はSKUごとに独立するため、他バリエーションが在庫ありなら選択・購入は継続可能）

5.6 SCR-006 注文確認画面

表示項目

  • 金額内訳（小計、消費税、送料、割引、合計）

■ 9. まとめ

Customer向け画面はProgressive Login（閲覧はGuest可、購入操作でログイン要求）を採用し、離脱率を下げる設計とした。Admin向け画面は商品→バリエーション→在庫という登録順序をSCR-014→016→017の遷移で誘導し、初期データ投入時の迷いを減らす。
