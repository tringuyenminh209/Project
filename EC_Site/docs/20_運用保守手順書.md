本書はEC Siteの日常運用・障害対応の手順を定義する。Project 01と共通する運用タスクに加え、本プロジェクト特有の決済・在庫関連の障害対応（6・7章）を重点的に定義する。

# 3. 日次運用タスク

```
□ /api/health 死活確認
□ payments.status=failed の当日件数を確認（異常増加がないか）
□ orders.status=pending で30分以上経過しているものが自動キャンセルされているか確認（バッチ稼働確認）
□ inventories.quantity_available < 10 のバリエーション一覧を確認（発注要否の判断材料）
```

# 6. 決済関連の障害対応（重点）

## 6.1 Webhookが届かない・遅延している

```
1. Stripeダッシュボードの「Webhookログ」で送信履歴・レスポンスコードを確認する
2. 自社側で400/500を返していないか確認する（署名検証失敗、内部エラー）
3. 400（署名検証失敗）の場合: STRIPE_WEBHOOK_SECRETの設定ミスを疑い、Stripeダッシュボードの値と照合する
4. 対象のorders/paymentsを特定し、Stripe側の実際の決済結果（成功/失敗）とDBの状態を突き合わせる
5. 不整合があれば、Stripeダッシュボードから該当イベントを手動で「再送信」する
6. 再送でも解決しない場合、payments.statusを手動でDB更新する（要: 変更履歴を運用ログに残す）
```

## 6.2 二重決済・二重発送の疑い

```
1. payments テーブルで同一 stripe_payment_intent_id が複数の captured を持たないか確認する
   （BR-PAY-003の冪等性チェックが機能していれば理論上発生しない。発生していればバグとして最優先で調査）
2. orders.status の変化履歴（更新日時）を確認する
3. 顧客への二重請求が確認された場合、Stripeダッシュボードから返金処理を行う
```

# 7. 在庫関連の障害対応（重点）

## 7.1 在庫がマイナスになっている

```
1. inventories.quantity_available < 0 のレコードを特定する（09_テーブル定義8章のCHECK制約が
   正しく機能していれば発生しないはずのため、発生時はDB制約の設定漏れを疑う）
2. inventory_logsで対象variant_idの直近の変動履歴を時系列で確認し、原因操作を特定する
3. 原因が同時アクセス（BR-INV-007のロック漏れ）であれば、該当Serviceコードを緊急修正しリリースする
4. 該当在庫を正しい値へAdmin機能（在庫調整）で補正し、inventory_logsにadjustmentとして記録する
```

## 7.2 在庫確保が解放されない（pending注文が滞留）

```
1. スケジューラコンテナ（13_インフラ設計7章）が正常稼働しているか確認する
2. php artisan schedule:list でCancelExpiredPendingOrdersが登録されているか確認する
3. 手動で該当コマンドを実行し、滞留しているpending注文を解消する
```

# 9. マスタ運用（商品・クーポン）

```

                 明示的な無効化操作は必須ではないが、一覧の見通しのためstatus=inactiveへ変更することを推奨
```

# 11. 定期メンテナンス（Maintenance Window）

| 項目 | 内容 |
| --- | --- |
| 対象作業 | DBマイグレーション（破壊的変更）、依存パッケージの大型アップデート、AWSインフラ構成変更（19_リリース手順書6章のマイグレーション手順を伴う場合） |
| 実施可否の判断基準 | 19_リリース手順書4章のリリース前チェックリストで「マイグレーションあり」に該当する場合は本節の手順に従う。追加のみ・後方互換のあるマイグレーションはメンテナンスウィンドウ不要（通常のデプロイ手順で対応可） |
| 実施時間帯 | 個人開発かつ商用トラフィックがまだ存在しないため固定の時間帯は定めない。将来一般公開後は深夜帯（アクセス最小時間帯）に設定する想定 |
| 事前通知 | 商用ローンチ後を想定した方針: トップページへのバナー表示 + 実施予定日時をリリースノートに記載。現状（学習用途・利用者なし）は本手順を省略してよい |
| 実施中の状態 | 19_リリース手順書5章のデプロイ手順に加え、決済関連の変更を伴う場合はStripe Webhookエンドポイントを一時的に無効化し、メンテナンス終了後に再有効化する（Webhook未達分はStripe側が自動リトライするため データ欠損は発生しない、9章参照） |
| ロールバック | メンテナンス作業中に問題が発生した場合は19_リリース手順書9章の手順に従う |
| 実施記録 | 実施日時・作業内容・所要時間を20_運用保守手順書本節末尾または別途運用ログに追記する（現状は未実施のため記録なし） |

# 14. まとめ

本プロジェクトの運用における最重要監視ポイントは「決済とDBの状態不整合」と「在庫のマイナス化」の2点である。この2点さえ日次で確認できていれば、他の運用タスクはProject 01と同等の一般的なWebシステム運用に帰着する。
