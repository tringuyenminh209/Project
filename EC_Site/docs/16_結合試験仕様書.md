# 1. 本書の目的

単体試験（15章）で確認したComponent単位の正しさを前提に、UI↔API、API↔DB、API↔外部サービス（Stripe/S3）の連携が正しく機能することを確認する。

---

# 2. 試験方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| IT-POL-001 | 実データフロー | Factoryではなく実際のAPI呼び出しの連鎖でデータを作成し、画面〜DBまでの一貫性を確認する。 |
| IT-POL-002 | 外部サービスはSandbox | Stripeは実際のテストモードAPIを呼び出す（モックに頼りすぎない）。 |
| IT-POL-003 | 障害復旧確認 | 決済失敗・Webhook遅延・タイムアウト時の復旧動作を確認する。 |

---

# 3. 試験範囲

| 対象外 | 理由 |
| --- | --- |
| 実際のクレジットカード情報を使った本番決済 | Stripe Sandboxのテストカード番号を使用する |
| 実配送業者への連携 | 対象外（`00_開発計画書.md`2.2節） |

# 6. API ↔ Stripe 結合試験

| Test ID | シナリオ | 確認内容 |
| --- | --- | --- |
| IT-STRIPE-001 | API-015 → Stripe PaymentIntent作成 | 返却された`client_secret`でStripe.js側の決済が実行可能である |
| IT-STRIPE-002 | Stripeテストカード（成功用）で決済 → Webhook受信 | `payment_intent.succeeded`受信後、5秒以内にorders.statusがpaidになる |
| IT-STRIPE-003 | Stripeテストカード（拒否用）で決済 → Webhook受信 | `payment_intent.payment_failed`受信後、payments.status=failed、orders.statusはpending維持 |
| IT-STRIPE-004 | Webhookエンドポイントへ署名なしでPOST | 400、E009、DB変更なし |
| IT-STRIPE-005 | 商品画像アップロード → S3 | S3バケットに実際にオブジェクトが作成され、`product_images.image_url`からアクセス可能 |

# 10. まとめ

結合試験の中心はIT-STRIPE系（Webhook連携）とIT-SCN-002（在庫確保期限切れの自動解放）である。これらは単体試験では検証しきれない「時間経過」と「外部サービスとの非同期連携」を含むため、本章で重点的に確認する。
