# 1. 本書の目的

本書はEC Siteの単体試験仕様を定義する。Controller/Service/FormRequest/Model/Exception単位で、機能要件・入力検証・業務ルール（BR-ID）・権限制御・例外処理を確認する。特に10〜12章（注文確定・決済・在庫）は本プロジェクトの学習目標を直接検証する最重要試験である。

---

# 2. 試験方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UT-POL-001 | Requirement Based | REQ / FUNC / APIに対応した試験を作成する。 |
| UT-POL-002 | Normal / Abnormal | 正常系、異常系、境界値を確認する。 |
| UT-POL-003 | Database Assertion | 在庫・金額・ステータスはDB保存結果を直接確認する。 |
| UT-POL-004 | Concurrency | 在庫確保の同時アクセス試験を必須項目とする（NFR-004）。 |
| UT-POL-005 | Idempotency | Webhook処理は同一イベントの重複実行でも結果が変わらないことを確認する（BR-PAY-003）。 |
| UT-POL-006 | Snapshot Integrity | 商品価格変更後も過去の`order_items`が変わらないことを確認する（BR-ORD-002）。 |

# 6. 判定基準

| 判定 | 条件 |
| --- | --- |
| Pass | 期待結果、HTTPステータス、DB状態、Error IDが仕様通り |
| Fail | 期待結果と異なる、在庫/金額の不整合、トランザクションのロールバック漏れ |
| Blocked | Stripeテスト環境等の依存が利用できず実行不可 |

# 17. まとめ

本書の試験ケースのうち、UT-ORD-005（スナップショット不変性）、UT-PAY-003（Webhook冪等性）、UT-INV-001（同時アクセス在庫整合性）の3つが、本プロジェクトの学習目標を直接検証する最重要試験である。実装が完了した段階で、この3件を最優先で自動テスト化すること。
