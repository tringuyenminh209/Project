# 1. 本書の目的

EC Siteの各画面について、表示項目、入力項目、操作、バリデーション、エラー、画面遷移を定義する。02_要件定義書 03_ユースケース 05_画面遷移図 と整合し、10_API設計 15_単体試験仕様書 の基準とする。

# 2. 画面設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| UI-001 | OOUI（オブジェクト指向UI） | 「商品」「注文」という対象物を選んでから操作する設計とする（EC・管理画面に適したパターン）。 |
| UI-002 | Progressive Login | 商品閲覧はGuest可、カート追加以降でログインを要求する（離脱コストを下げる）。 |
| UI-003 | Error Visibility | 在庫不足・決済失敗・クーポンエラーを画面上で明確に表示する。 |
| UI-004 | Traceability | 各画面はSCR・UC・REQ・エラーIDと対応させる。 |
| UI-005 | Mobile First | Tailwind CSSのユーティリティクラスでスマートフォン表示を優先し、PCはブレークポイントで拡張する（NFR-021）。 |

## 2.1 デザイントークン

全20画面で使い回す値をここで確定する。個別画面で新しい色・余白を発明しない（Tailwind CSSのデフォルトスケールをそのまま採用し、独自トークンの二重管理を避ける）。

| カラー | Background | `gray-50` | ページ背景 |

# 5. 画面詳細

## 5.0 状態バリエーション 共通ルール

v1.2でSCR-003/004に試験適用した書式を全20画面へ展開する。各画面の「状態バリエーション」表は以下の共通ルールを前提とし、画面固有の差分のみを記載する（2.1節のトークン名で表記を統一）。

| 共通ルール | 内容 |
| --- | --- |
| Loading（取得系） | 一覧・詳細の初期表示はスケルトン（`Border`色の枠 + `Background`色のバー）で件数分プレースホルダー表示し、API応答後に差し替える |
| Loading（送信系） | 送信ボタンは2.1節Buttonトークンのloading状態（スピナー + 非活性）にし、二重送信を防止する。応答までフォーム入力も非活性にする |
| Empty | 0件は正常応答として扱い、Error表示と混同しない。「なぜ0件か + 次に取るべき行動」を`Text-secondary`で案内する |
| Error（通信失敗） | 「取得に失敗しました。」+ 再試行ボタン（SCR-003のError書式と同一） |
| Error（業務エラー） | E001〜E011は7章のメッセージを`Error`色で表示する。入力項目起因（E003）は該当Inputのerror状態 + 直下赤字（2.1節Inputトークン） |

## 5.1 SCR-001 ログイン画面

### 表示項目
- サイトロゴ、会員登録画面（SCR-002）への導線リンク

### 入力項目
- メールアドレス（Required / Email形式 / Max 255）
- パスワード（Required / 8〜20文字）

### 操作
- 「ログインする」→ API-002を呼び出し、成功時はCustomerはSCR-003（商品一覧）、AdminはSCR-013（管理者ダッシュボード）へ遷移。Progressive Login経由の場合は元の操作画面（例: SCR-004）に戻る。
- 「新規会員登録はこちら」→ SCR-002へ遷移

### エラー
- メールアドレス/パスワード不一致 → E001（アカウントの有無を推測されないよう「どちらが誤りか」は明示しない、14_セキュリティ設計4.3節と一致）
- 入力形式不正（未入力・Email形式違反等）→ E003

### 画面遷移
- Progressive Login（UI-002）によりSCR-004（商品詳細）でカート追加を試みた未ログインGuestは本画面へ遷移し、ログイン成功後は元の操作を継続する。

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 「ログインする」押下後はButton loading状態（共通ルール）。API-002応答まで再送信不可 |
| Empty | 対象外（一覧を持たない画面） |
| Error | E001/E003をフォーム上部に表示。E001はどちらの項目が誤りかを明示しない（14_セキュリティ設計4.3節） |

---

## 5.2 SCR-002 会員登録画面

### 表示項目
- サイトロゴ、ログイン画面（SCR-001）への導線リンク

### 入力項目
- 氏名（Required / Max 100）
- メールアドレス（Required / Email形式 / Max 255 / Unique）
- パスワード（Required / 8〜20文字）
- パスワード確認（Required / パスワードと一致）

### 操作
- 「登録する」→ API-001を呼び出し、成功時は自動ログイン状態でSCR-003（商品一覧）へ遷移（会員登録はCustomerのみが行う操作のため、Admin向けSCR-013には遷移しない）
- 「ログインはこちら」→ SCR-001へ遷移

### エラー
- メールアドレス重複 → E003（10_API設計 API-001のUnique制約違反、422）
- 入力形式不正・パスワード不一致 → E003

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 「登録する」押下後はButton loading状態 |
| Empty | 対象外 |
| Error | E003は該当Inputのerror書式で直下に表示。メール重複時は「このメールアドレスは既に登録されています」 |

## 5.3 SCR-003 商品一覧画面

### 表示項目
- 商品カード（サムネイル画像、商品名、価格帯「¥2,000〜」、カテゴリ）
- カテゴリ絞込メニュー、キーワード検索窓、ページネーション

### 検索条件
- キーワード（部分一致）、カテゴリ、価格帯

### 操作
- 商品カードをクリック → SCR-004へ遷移

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 商品カードをグレーのスケルトン（画像枠＋2行のバー）で件数分プレースホルダー表示、API-006応答後に差し替える |
| Empty | 検索/絞込の結果が0件のとき「該当する商品が見つかりませんでした。条件を変更してください。」+ 絞込リセットボタン（初回アクセス時の全件0件はマスタ未登録を意味するため対象外） |
| Error | API-006失敗時「商品一覧の取得に失敗しました。」+ 再試行ボタン（`Border`色の枠、`Text-secondary`文言、UI-003準拠） |

## 5.4 SCR-004 商品詳細画面

### 表示項目
- 商品画像（複数、`product_images.display_order`順）
- 商品名、説明、価格（選択バリエーションに応じて動的に変わる）
- バリエーション選択（サイズ・色のプルダウン、`sku`ごとの在庫状況）
- レビュー平均・件数、レビュー一覧

### 入力・バリデーション
- バリエーション：Required（未選択でカート追加不可）
- 数量：Required / Integer / 1以上 / 在庫数以下

### 操作
- 「カートに追加」→ SCR-005へ遷移（またはトースト表示で一覧に留まる）
- 未ログイン時にカート追加を押すとSCR-001へ遷移（Progressive Login）

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 画像枠・商品名・価格の3ブロックをスケルトン表示、API-007応答後に差し替える |
| Empty | レビュー0件時は一覧を非表示にせず「まだレビューがありません」を`Text-secondary`で表示（本体の商品情報は0件対象外） |
| Error | 存在しない/非公開の商品ID（E007）→ 商品情報を出さず「商品が見つかりませんでした」+ SCR-003へ戻る導線。API通信失敗（ネットワーク断等、業務エラーコードなし）→ 再試行ボタン |
- 在庫0のバリエーション選択時：「カートに追加」ボタンを`disabled`表示にし、「在庫切れ」を`Warning`色でバリエーション横に表示（在庫はSKUごとに独立するため、他バリエーションが在庫ありなら選択・購入は継続可能）

| --- | --- |

## 5.6 SCR-006 注文確認画面

### 表示項目
- 注文明細（カートと同内容）
- 配送先住所選択（登録済み住所帳 or 新規入力）
- クーポンコード入力欄
- 金額内訳（小計、消費税、送料、割引、合計）

### 入力・バリデーション
- 配送先：Required
- クーポンコード：Optional / Max 30 / 適用条件はBR-CPN-001

### 操作
- 「注文を確定する」→ Systemが在庫確保・スナップショット処理（UC-008 手順5）を実行し、SCR-007へ遷移
- クーポン条件不成立時はE005を表示し、確認画面に留まる

### 状態バリエーション
| 状態 | 表示 |
| --- | --- |
| Loading | 住所一覧・明細をスケルトン表示。「注文を確定する」押下後はButton loading — 在庫確保を伴うため二重送信防止が全画面中で最も重要（12_詳細設計書5.2節 / BR-INV-003） |
| Empty | 登録住所0件 → 新規入力フォームを展開した状態で表示（REQ-010）。カートが空の状態で直接アクセス → SCR-005へ戻す |
| Error | E004 → 該当明細行をError色で強調し数量修正を促す。E005 → クーポン入力欄直下に表示し画面に留まる（BR-CPN-001） |

# 7. エラーメッセージ

| エラーコード | 画面表示メッセージ |
| --- | --- |
| E001 | メールアドレスまたはパスワードが正しくありません。 |
| E002 | この操作を実行する権限がありません。 |
| E003 | 入力内容に誤りがあります。 |
| E004 | 在庫が不足しています。 |
| E005 | クーポンを適用できません（条件をご確認ください）。 |
| E006 | この注文は現在の状態から変更できません。 |
| E007 | 対象のデータが見つかりません。 |
| E008 | 決済に失敗しました。カード情報をご確認のうえ再度お試しください。 |
| E010 | ログインが必要です。 |
| E011 | この操作は既に行われています（例: この注文商品には既にレビューを投稿済みです）。 |

**補足1**: E009（Webhook署名検証失敗）はStripe→サーバー間の連携専用コードであり、画面に表示されることはないため本表に含めない。

**補足2**: 画面固有の文脈に合わせ、上表の標準文言を言い換えて表示してよい（例: SCR-007の確保期限切れはE006を「注文の有効期限が切れました。」と表示する — 5.7節）。ただしエラーコード自体は変更しない（試験仕様書はコードで判定する）。

# 9. まとめ

Customer向け画面はProgressive Login（閲覧はGuest可、購入操作でログイン要求）を採用し、離脱率を下げる設計とした。Admin向け画面は商品→バリエーション→在庫という登録順序をSCR-014→016→017の遷移で誘導し、初期データ投入時の迷いを減らす。
