# 1. 本書の目的

`17_システム試験仕様書.md`が技術的な正しさを確認するのに対し、本書は実際の購買体験の言葉・操作感でシステムを通し、`02_要件定義書.md`の受入条件（6章）と`01_企画書.md`の成功条件（12章）を最終的に満たすことを確認する。

# 6. UATシナリオ（Customer編）

| シナリオID | 内容 | 確認の観点（業務言葉） |
| --- | --- | --- |
| UAT-CUS-001 | 初めて訪れて商品を探す | 欲しい商品がカテゴリ・キーワードですぐ見つかるか |
| UAT-CUS-002 | サイズ違いの商品をカートに入れる | どのサイズ・色を選んだか画面上で迷わず分かるか |
| UAT-CUS-003 | クーポンを使って注文する | 割引額が注文確認画面で分かりやすく表示されるか |
| UAT-CUS-004 | クレジットカードで支払う | 入力項目が最小限で、エラー時に何が悪いか分かるか |
| UAT-CUS-005 | 注文後にマイページで履歴を見る | 過去に何を・いくらで買ったかが明確に残っているか |
| UAT-CUS-006 | 商品到着後にレビューを書く | 購入していない商品にレビューボタンが出ないか |

# 7. UATシナリオ（Admin編）

| シナリオID | 内容 | 確認の観点（業務言葉） |
| --- | --- | --- |
| UAT-ADM-001 | 新商品をサイズ・色違いで登録する | 登録の手順が迷わず追える順序になっているか（商品→バリエーション→在庫→画像） |
| UAT-ADM-002 | 入荷した在庫数を反映する | 誤って現在庫を上書きせず、加算として扱われているか |
| UAT-ADM-003 | 注文を確認し発送する | 発送すべき注文（paid状態）が一目でわかるか |
| UAT-ADM-004 | 割引クーポンを企画する | 条件（期間・上限・最低購入額）が直感的に設定できるか |
| UAT-ADM-005 | 今月の売上を確認する | 数字が業務判断に使える粒度（月別・カテゴリ別）で出るか |

# 9. 不具合判定基準

| 重要度 | 基準 |
| --- | --- |
| Critical | 購入・決済・在庫の整合性が崩れる（二重販売、金額誤り、注文が消える等） |
| High | 業務は続行できるが誤解を招く表示・エラーメッセージ |
| Medium | UI改善要望レベル |
| Low | 表記ゆれ等 |

---

# 10. 受け入れ判定基準（サインオフ条件）

- UATシナリオ（6/7/8章）すべてがPassすること
- Critical不具合が0件であること
- `02_要件定義書.md`19章の受入条件をすべて満たすこと
- `01_企画書.md`12章の成功条件（テストカバレッジ、同時アクセス試験、スナップショット整合性試験）を満たすこと

# 12. まとめ

本UATは、技術試験では見落としがちな「業務として自然か」を最終確認する工程である。特にUAT-ADM-001（商品登録の手順の分かりやすさ）とUAT-CUS-004（決済のシンプルさ）は、実務のEC案件でも顧客から必ず指摘される観点であり、重点的に確認する。
