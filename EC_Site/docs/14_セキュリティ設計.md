# 1. 本書の目的

本書はEC Siteのセキュリティ設計を定義する。Project 01の認証・認可設計を踏襲しつつ、本プロジェクト特有の決済（PCI DSS）・Webhook・他会員データへのアクセス防止を重点的に定義する。

---

# 2. セキュリティ設計方針

| 方針ID | 方針 | 内容 |
| --- | --- | --- |
| SEC-POL-001 | カード情報非保持 | クレジットカード番号・CVC等は自社サーバーで一切保持・通過させない。 |
| SEC-POL-002 | 最小権限 | Admin機能は全て`role:admin`で保護し、Customerは自分のデータのみアクセス可能とする。 |
| SEC-POL-003 | Webhookは信頼しない | 署名検証に失敗したWebhookリクエストは即座に拒否する。 |
| SEC-POL-004 | 冪等性による二重処理防止 | 決済確定処理は`stripe_payment_intent_id`をキーに冪等化する。 |
| SEC-POL-005 | 監査可能性 | 在庫・注文・クーポンの変更はすべて履歴を残す。 |

---

# 3. 保護対象

| 対象 | 分類 | 保護方針 |
| --- | --- | --- |
| users.password_hash | 認証情報 | ハッシュ化（bcrypt）、平文保存禁止 |
| addresses / orders.shipping_* | 個人情報（住所・氏名・電話番号） | 本人および担当Adminのみ閲覧可、通信は全てHTTPS |
| クレジットカード情報 | 決済情報 | 自社では非保持（Stripeに委譲、SEC-POL-001） |
| coupons.code | ビジネス情報 | Admin以外への一覧公開はしない（適用APIのみ許可） |

# 4. 認証設計

## 4.1 認証方式

Laravel Sanctumトークン認証（Project 01と同一）。Token期限は24時間（02_要件定義書4.1節）。

## 4.2 認証フロー

10_API設計4章を参照。ログイン成功時にaccess_tokenを発行し、以後`Authorization: Bearer {token}`で認証する。

## 4.3 認証エラー

認証失敗（E001）・未認証アクセス（E010）は共に汎用的なメッセージとし、「メールアドレスが存在しない」「パスワードが違う」を区別して返さない（アカウント存在の推測を防ぐ）。

# 5. 認可設計

## 5.1 Role定義

`users.role`列に永続化される値は以下の2つのみ(09_テーブル定義 TBL-001)。

| Role | 説明 |
| --- | --- |
| customer | 一般会員 |
| admin | 事業者スタッフ |

10_API設計5章・6章の権限列に登場する「Guest」「System」はDBに保存されるroleではなく、API呼び出し元の区分を表すラベルである(03_ユースケースのアクター定義と一致)。

| ラベル | 実体 | 認可の仕組み |
| --- | --- | --- |
| Guest | 未認証アクセス | `auth:sanctum`を通さないPublicルート。roleチェック自体が発生しない。 |
| System | Stripe等の外部サービス | roleチェックの代わりにWebhook署名検証(本書9章「Webhookセキュリティ設計」)で真正性を担保する。 |

## 5.2 API認可

| 保護方式 | 対象 |
| --- | --- |
| `auth:sanctum` middleware | 全Customer/Admin API |
| `role:admin` middleware | `/admin/*` 配下すべて |
| Service層でのWHERE句スコープ | 自分の注文/住所/レビューへのアクセス制御（11_基本設計書9.2節） |

## 5.3 データ範囲制御（重要）

| API | 制御内容 |
| --- | --- |
| GET /orders, /orders/{id} | `orders.user_id = 認証ユーザーID`を必須条件とする。他会員の注文IDを直接指定してもE002/E007を返す（注文の存在有無を推測されないよう、所有者以外にはE007で統一することも検討可） |
| GET /addresses | 同上（`addresses.user_id`） |
| POST /reviews | `order_items`の親`orders.user_id`が本人であることを検証 |

# 7. 入力検証設計

## 7.1 Validation方針

FormRequestで形式チェック、Serviceで業務ルールチェック（12_詳細設計書2章DD-POL方針）。

## 7.2 主なValidation対象

| リスク | 対策 |
| --- | --- |
| SQLインジェクション | Eloquent ORM / パラメータバインディングを徹底し、生SQL連結を禁止する |
| XSS | Bladeの`{{ }}`エスケープ出力を徹底し、商品説明等のリッチテキストが必要な場合のみ`{!! !!}`をサニタイズ後に限定使用する |
| 数量・金額の改ざん | カート内容・金額はクライアントから受け取らず、常にサーバー側で`inventories`/`product_variants`から再計算する（10_API設計6.5節） |

# 8. 決済セキュリティ設計（PCI DSS）

| 項目 | 内容 |
| --- | --- |
| スコープ | カード情報を自社サーバーで一切保持・処理・通過させないため、PCI DSS自己問診票`SAQ-A`相当の最も簡易な区分に収まる。 |
| 実装方式 | Stripe Elements（クライアントサイドでカード情報をトークン化し、Stripeサーバーへ直接送信）を使用する。自社サーバーはトークン/PaymentIntent IDのみを扱う。 |
| ログ出力禁止 | カード番号・CVC・生のトークンはアプリケーションログに一切出力しない。 |
| 通信 | 全ページHTTPS必須（NFR-015）。 |

---

# 9. Webhookセキュリティ設計

| 項目 | 内容 |
| --- | --- |
| 署名検証 | `Stripe-Signature`ヘッダとStripe SDKの`Webhook::constructEvent()`で検証し、検証失敗はE009（400）で即座に拒否する。 |
| リプレイ攻撃対策 | Stripe SDKの検証はタイムスタンプの許容誤差（デフォルト5分）を含み、古いイベントの再送を自動的に拒否する。 |
| エンドポイント保護 | `/api/webhooks/stripe`はCSRF検証対象外とする一方、署名検証で代替の真正性確認を行う（13_インフラ設計8章）。 |
| シークレット管理 | `STRIPE_WEBHOOK_SECRET`は環境変数で管理し、コードにハードコードしない。 |

# 10. Web Security設計

## 10.1 OWASP対策

OWASP Top 10 (2021)の全10カテゴリと対策の対応関係。

| リスク | 対策 |
| --- | --- |
| A01 Broken Access Control | 5章のデータ範囲制御を徹底。`role:admin`ミドルウェア + Service層WHERE句スコープの二重チェック（5.2/5.3節） |
| A02 Cryptographic Failures | パスワードbcrypt、通信HTTPS必須（NFR-015）、カード情報はStripe Elementsでトークン化しサーバーを一切経由させない（8章） |
| A03 Injection | Eloquent ORM使用（生SQL・文字列結合クエリを禁止）、7章の入力検証 |
| A04 Insecure Design | 在庫2段階制御（確保→確定）・注文スナップショット設計により、設計レベルで二重販売・過去データ改変を防止（09_テーブル定義11章の設計判断メモ、16章Threat Model） |
| A05 Security Misconfiguration | `.env`本番値はAWS上のみで管理しGitに含めない、DBはPrivate Subnetで直接インターネット非公開（13_インフラ設計5章）、不要なLaravel debugモードは本番で無効化 |
| A06 Vulnerable and Outdated Components | `composer.lock`をリポジトリにコミットしバージョン固定、GitHub Dependabotで依存パッケージの脆弱性アラートを有効化する（実装フェーズで`.github/dependabot.yml`を追加） |
| A07 Identification and Authentication Failures | Laravel Sanctumトークン認証（4章）、パスワードポリシー8〜20文字（6章）、ログイン試行のRate Limit（11.1節）でブルートフォースを抑制 |
| A08 Software and Data Integrity Failures | Webhook署名検証（9章） |
| A09 Security Logging and Monitoring Failures | 14章のAudit/Logging設計で操作ログは記録するが、リアルタイム監視・アラートは13_インフラ設計15章の記載どおり現状未実装（CloudWatch/Slack通知は将来対応）。本スコープでは事後調査用ログのみで、侵害のリアルタイム検知は範囲外と明記する。 |
| A10 Server-Side Request Forgery (SSRF) | ユーザー入力のURLをサーバー側で取得する機能が存在しない（商品画像はLaravelサーバーが自身のIAMロールでS3へ直接書き込む方式、13_インフラ設計11章）ため、本システムではリスクが低いと判断する。将来外部URL取込機能を追加する場合は許可リスト方式で再検討する。 |

## 10.2 Browser Security Header

Project 01と同一（`X-Frame-Options` `X-Content-Type-Options`等をNginx/Laravelミドルウェアで付与）。

# 11. API Security設計

## 11.1 Rate Limit

`POST /auth/login`、`POST /coupons/validate`、`POST /orders`にLaravelの標準Rate Limiter（例: 1分あたり10回）を適用し、ブルートフォース攻撃・クーポン総当たりを抑制する。

## 12.1 Database保護

## 12.2 Data Masking / Response制御

# 16. Threat Model

| 脅威 | シナリオ | 対策 |
| --- | --- | --- |
| 在庫の二重販売 | 同時に複数リクエストが同一バリエーションへ注文 | `SELECT ... FOR UPDATE`による悲観的ロック（BR-INV-007） |
| クーポンの不正大量利用 | 同一クーポンを高速で連打 | Rate Limit + `used_count`のトランザクション内更新 |
| Webhook偽装 | 攻撃者が独自に決済成功イベントを送信し無料で商品を得ようとする | 署名検証必須（9章） |
| 他会員注文の閲覧 | 注文IDを推測して直接アクセス | 5.3節のデータ範囲制御 |
| カード情報漏洩 | 自社サーバーの侵害 | SEC-POL-001によりそもそもカード情報を保持しない |

---

# 17. Security Checklist

```
□ カード情報を自社サーバーで一切保持していない（Stripe Elements使用）
□ Webhook署名検証を実装している
□ 決済確定処理が冪等である（重複イベントで二重発送・二重在庫消費が起きない）
□ 他会員の注文/住所/レビューにアクセスできない
□ カート・注文金額はサーバー側で必ず再計算している（クライアント値を信用しない）
□ 在庫更新に排他制御（FOR UPDATE）を適用している
□ ログにパスワード・カード情報・Secretを出力していない
```

# 19. まとめ

本プロジェクトのセキュリティ設計における最大の学習ポイントは、「カード情報を持たない」という設計判断（SEC-POL-001）と、それに伴うWebhook署名検証（9章）である。この2点はEC/決済系案件のBrSEとして必ず説明できる必要がある基礎知識である。
